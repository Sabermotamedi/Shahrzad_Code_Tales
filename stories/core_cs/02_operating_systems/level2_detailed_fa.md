# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی آشپزخانه 🔍🍳

> همان قصه‌ی سام و آشپزخانه، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعی *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — سرآشپز واقعاً کیست؟ (Operating System)

«سرآشپز» همان **سیستم‌عامل** (Operating System) است — مثل Linux، Windows یا macOS. خودش هیچ کارِ واقعی‌ای (مثل مرورگر یا بازی) انجام نمی‌دهد، اما **واسطه‌ی بین برنامه‌ها و سخت‌افزار** است و منابع محدود (پردازنده، حافظه، دیسک) را بین همه تقسیم می‌کند.

| در قصه | در دنیای واقعی |
|---|---|
| سرآشپز | Operating System (Kernel) |
| آشپزها | Processes (پروسه‌ها) |
| اجاق‌ها | CPU Cores (هسته‌های پردازنده) |
| میزِ کار | Memory / RAM (حافظه) |
| انباری پشتی | Swap / Disk (حافظه‌ی مجازی) |
| کشوهای دستور | File System (سیستم فایل) |
| کلیدِ کشوها | Permissions (دسترسی‌ها) |
| پرسیدن از سرآشپز | System Calls (فراخوانی سیستمی) |
| چک‌لیست جادویی | Shell Script (اسکریپت Bash) |
| تابلوی اعلانات | Environment Variables (متغیرهای محیطی) |
| ایستگاه‌های همیشه‌روشن | Services / systemd |

بخشِ مرکزیِ سیستم‌عامل که این کنترل را دارد، **هسته** (Kernel) نام دارد. هسته در یک حالتِ ویژه و قدرتمند اجرا می‌شود (Kernel Mode)، در حالی که برنامه‌های عادی در حالتِ محدودتری هستند (User Mode). این جداسازی همان دیواری است که نمی‌گذارد یک آشپزِ خرابکار کل آشپزخانه را به هم بریزد.

## صحنه‌ی ۲ — آشپزها همان پروسه‌ها هستند (Processes)

هر «آشپز» یک **پروسه** (Process) است: یک برنامه‌ی در حالِ اجرا. وقتی مرورگر را باز می‌کنی، یک پروسه ساخته می‌شود. هر پروسه برای خودش چیزهایی دارد:

- یک شماره‌ی یکتا: **PID** (Process ID) — مثل شماره‌ی پرسنلیِ آشپز
- فضای حافظه‌ی **جداگانه** — میزِ کارِ مخصوص خودش که آشپزِ بغلی نمی‌تواند به آن دست بزند (Memory Isolation)
- یک «والد» (Parent) — هر آشپز را یک آشپزِ دیگر استخدام کرده؛ به این می‌گویند `fork`

```bash
ps aux           # فهرست همه‌ی آشپزها (پروسه‌ها) و اینکه هر کدام چقدر منبع می‌خورند
ps -ef --forest  # نشان دادن رابطه‌ی والد-فرزند به شکل درختی
```

نکته‌ی مهم: چون میزِ هر آشپز جداست، اگر یک آشپز ظرفی را بشکند (پروسه crash کند)، بقیه سالم می‌مانند. این یکی از بزرگ‌ترین کارهای سرآشپز است: **جداسازیِ پروسه‌ها**.

## صحنه‌ی ۳ — چند آشپز روی یک ایستگاه (Threads)

گاهی یک غذای بزرگ آن‌قدر کار دارد که **چند آشپز روی یک ایستگاهِ مشترک** کار می‌کنند: یکی سبزی خرد می‌کند، یکی هم می‌زند، روی همان میز و همان مواد. این‌ها **Threadها** هستند.

فرق کلیدی:

| | پروسه (Process) | نخ (Thread) |
|---|---|---|
| میزِ کار (حافظه) | جداگانه | **مشترک** بین نخ‌های یک پروسه |
| ساختنش | سنگین | سبک |
| crash یکی | بقیه سالم | ممکن است کلِ پروسه بیفتد |
| ارتباط | سخت (باید از سرآشپز کمک بگیرند) | راحت (همان میز مشترک) |

چون نخ‌ها میز را مشترک دارند، خیلی سریع با هم کار رد و بدل می‌کنند — اما خطری هم هست: اگر دو نخ هم‌زمان به یک ظرف دست بزنند، فاجعه می‌شود! به این می‌گویند **Race Condition**. راه‌حل: یک **قفل** (Lock / Mutex) که می‌گوید «تا من کارم با این ظرف تمام نشده، تو دست نزن».

## صحنه‌ی ۴ — نوبت‌دهیِ سرآشپز (Scheduling)

اجاق‌ها (هسته‌های CPU) کم‌اند و آشپزها زیاد. سرآشپز هر چند **میلی‌ثانیه** یک آشپز را از روی اجاق برمی‌دارد و یکی دیگر را می‌گذارد. به این کار **Context Switch** می‌گویند، و به مغزِ تصمیم‌گیرنده **Scheduler**.

```
زمان →
هسته‌ی ۱:  [آشپز A][آشپز B][آشپز A][آشپز C][آشپز A] ...
              ↑ هر تکه چند میلی‌ثانیه (time slice / quantum)
```

چون این جابه‌جایی خیلی سریع است، **حس می‌کنیم همه با هم اجرا می‌شوند** (Concurrency)، در حالی که روی یک هسته در هر لحظه فقط یکی واقعاً می‌پزد. (اگر چند هسته داشته باشیم، چند تا **واقعاً** هم‌زمان می‌پزند → Parallelism.)

سرآشپز کورکورانه نوبت نمی‌دهد؛ **اولویت** (Priority) دارد:

- سفارشِ فوریِ مشتری → اولویت بالا
- تمیز کردن انباری → اولویت پایین (وقتی بی‌کار شد)

در Linux این اولویت را با **nice value** تنظیم می‌کنیم (از ۲۰- تا ۱۹؛ عددِ کمتر = مهم‌تر):

```bash
nice -n 10 ./heavy_job.sh   # این کار را با اولویت پایین اجرا کن
renice -n 5 -p 1234         # اولویت پروسه‌ی 1234 را عوض کن
```

یک آشپز همیشه در یکی از این **حالت‌ها** است:

| حالت | در قصه | معنی |
|---|---|---|
| Running | روی اجاق مشغول | الان روی CPU است |
| Ready | کنار اجاق منتظر نوبت | آماده، منتظر CPU |
| Blocked / Waiting | منتظرِ رسیدنِ مواد از انبار | منتظر I/O (دیسک، شبکه) |

## صحنه‌ی ۵ — تقسیمِ میزِ کار (Memory Management)

سرآشپز میزِ کار (RAM) را تکه‌تکه بین آشپزها تقسیم می‌کند. هر آشپز فکر می‌کند یک میزِ بزرگِ پیوسته از صفر دارد — اما این یک **توهمِ زیبا** است که سرآشپز ساخته. به آدرس‌هایی که آشپز می‌بیند **Virtual Address** و به آدرس واقعیِ میز **Physical Address** می‌گویند.

سرآشپز یک **دفترِ نقشه** دارد (Page Table) که می‌گوید: «گوشه‌ی خیالیِ آشپز A در واقع گوشه‌ی واقعیِ شماره‌ی فلان است.» حافظه به تکه‌های هم‌اندازه تقسیم می‌شود به نام **Page** (معمولاً ۴ کیلوبایت).

```
دیدِ آشپز A (مجازی)        دفترِ نقشه‌ی سرآشپز        میزِ واقعی (فیزیکی)
   صفحه‌ی ۰      ───────────►  page table  ──────────►  قابِ ۷
   صفحه‌ی ۱      ───────────►              ──────────►  قابِ ۲
   صفحه‌ی ۲      ───────────►              ──────────►  (در انباری! روی دیسک)
```

## صحنه‌ی ۶ — انباریِ پشتی (Virtual Memory و Swap)

وقتی میز پر می‌شود، سرآشپز چیزهایی که الان لازم نیستند را می‌برد توی **انباریِ پشتی** (دیسک) و جا باز می‌کند. به این انتقال می‌گویند **Swap**. وقتی آشپز دوباره آن چیز را خواست، سرآشپز سریع از انبار برمی‌گرداند.

این جادو اسمش **Virtual Memory** است: هر آشپز می‌تواند بیشتر از کلِ میزِ موجود ادعای فضا کند، چون انبار پشتیبان است.

اما یک خطر دارد: اگر آشپز چیزی بخواهد که روی میز نیست و در انبار است، یک لحظه **توقف** پیش می‌آید تا از انبار بیاید — به این می‌گویند **Page Fault**. اگر آن‌قدر میز کم باشد که سرآشپز مدام بین میز و انبار چیز جابه‌جا کند و دیگر وقتی برای پختن نماند، رستوران فلج می‌شود! این کابوس اسمش **Thrashing** است (همان لحظه‌ای که کامپیوترت با RAM پر، قفل می‌کند).

```bash
free -h          # چقدر RAM و swap آزاد است
vmstat 1         # ستون‌های si/so = swap in/out؛ اگر بالا بود یعنی فشار حافظه
```

## صحنه‌ی ۷ — کشوهای دستور پخت (File System Hierarchy)

کشوها همان **سیستم فایلِ** Linux هستند. در Linux همه‌چیز از یک ریشه‌ی واحد `/` شروع می‌شود (برخلاف Windows که چند درایو `C:` و `D:` دارد). هر کشو یک کارِ مشخص دارد:

| کشو (مسیر) | چه چیزی داخلش است | معادلِ قصه |
|---|---|---|
| `/etc` | فایل‌های **تنظیمات** سیستم | دستورهای رسمی رستوران |
| `/home` | پوشه‌ی شخصیِ هر کاربر (`/home/sam`) | کمدِ شخصیِ هر آشپز |
| `/var` | داده‌های **متغیر**: لاگ‌ها، صف‌ها، کش | یادداشت‌های روزانه |
| `/usr` | برنامه‌ها و کتابخانه‌های نصب‌شده | کتابِ آشپزیِ بزرگ |
| `/bin`, `/usr/bin` | دستورهای اجرایی (`ls`, `cat`) | ابزارهای پایه |
| `/tmp` | فایل‌های موقت (با ریستارت پاک می‌شوند) | میزِ کارِ یک‌بارمصرف |
| `/dev` | دستگاه‌ها (دیسک، صفحه‌کلید) به شکل فایل | درهای ورودیِ مواد |
| `/proc` | پنجره‌ای زنده به وضعیتِ هسته و پروسه‌ها | دفترِ لحظه‌ایِ سرآشپز |
| `/root` | خانه‌ی کاربرِ مدیر (root) | دفترِ خودِ سرآشپز |

```bash
ls -la /etc      # داخل کشوی تنظیمات را ببین
cat /proc/cpuinfo  # /proc یک فایلِ واقعی نیست؛ هسته زنده تولیدش می‌کند!
```

## صحنه‌ی ۸ — کلیدِ کشوها (Permissions)

روی هر کشو نوشته شده چه کسی چه کاری می‌تواند بکند. در Linux هر فایل سه دسته دسترسی دارد، برای سه گروه:

```
-rwxr-xr--   1  sam   chefs   ...
 │└┬┘└┬┘└┬┘     └─┬─┘ └──┬─┘
 │ │  │  │       مالک   گروه
 │ │  │  └── دیگران (others):  r--  → فقط خواندن
 │ │  └───── گروه   (group) :  r-x  → خواندن + اجرا
 │ └──────── مالک   (owner) :  rwx  → خواندن + نوشتن + اجرا
 └────────── نوعِ فایل (- فایل، d پوشه، l لینک)
```

سه نوع اجازه:
- **r** (read) = خواندن
- **w** (write) = نوشتن/تغییر
- **x** (execute) = اجرا کردن (برای پوشه: یعنی اجازه‌ی ورود به آن)

و سه گروهِ مخاطب: **مالک** (user)، **گروه** (group)، **دیگران** (others). هر فایل یک مالک و یک گروه دارد — درست مثل اینکه هر کشو متعلق به یک آشپز و یک تیم است.

```bash
chmod 750 script.sh    # عددی: مالک=7(rwx) گروه=5(r-x) دیگران=0(---)
chmod u+x script.sh    # نمادین: به مالک اجازه‌ی اجرا بده
chown sam:chefs file   # مالک را sam و گروه را chefs کن
```

عددها از جمعِ این‌ها می‌آیند: **r=4، w=2، x=1**. پس `7 = 4+2+1 = rwx` و `5 = 4+1 = r-x`.

> یک آشپزِ خاص به نام **root** (یا superuser) کلیدِ همه‌ی کشوها را دارد و هیچ قفلی جلویش را نمی‌گیرد. به همین دلیل با root کار کردن خطرناک است — یک اشتباه و کلِ آشپزخانه به هم می‌ریزد. به جایش از `sudo` استفاده می‌کنیم: «فقط همین یک کار را با اجازه‌ی سرآشپز انجام بده».

## صحنه‌ی ۹ — پرسیدن قبل از دست زدن (System Calls)

قانونِ طلایی: آشپز (پروسه در User Mode) **حق ندارد** سرِخود به منابعِ مشترک دست بزند. هر بار باید از سرآشپز (Kernel) **اجازه بگیرد**. این درخواست همان **System Call** (فراخوانی سیستمی) است.

می‌خواهی یک فایل بخوانی؟ نمی‌توانی مستقیم به دیسک دست بزنی. باید بگویی:

```
آشپز (برنامه):   «سرآشپز، فایلِ recipe.txt را برایم باز کن»  → open()
سرآشپز (Kernel): فایل را باز می‌کند، یک «شماره» می‌دهد        → file descriptor
آشپز:            «حالا محتوایش را بخوان»                       → read()
آشپز:            «تمام شد، ببندش»                              → close()
```

چند system call پرکاربرد:

| فراخوانی | کار | معادلِ قصه |
|---|---|---|
| `open` / `read` / `write` | کار با فایل | باز کردن کشو |
| `fork` | ساختنِ پروسه‌ی فرزند | استخدامِ آشپزِ جدید |
| `exec` | اجرای یک برنامه‌ی جدید در پروسه | دادنِ دستورِ پختِ تازه |
| `mmap` / `brk` | گرفتنِ حافظه | درخواستِ میزِ بیشتر |
| `socket` | باز کردنِ ارتباطِ شبکه | برداشتنِ تلفن |

چرا این همه سخت‌گیری؟ چون فقط سرآشپز می‌تواند مطمئن شود **دو آشپز هم‌زمان یک کار خطرناک نکنند** و کسی به مالِ دیگری دست‌درازی نکند. هر بار که از User Mode به Kernel Mode می‌رویم، یک تعویضِ حالت رخ می‌دهد که کمی هزینه دارد — به همین دلیل برنامه‌های پرسرعت سعی می‌کنند system callهای کمتری بزنند.

## صحنه‌ی ۱۰ — چک‌لیستِ جادویی (Shell Script و Bash)

آن کاغذِ چک‌لیست همان **Shell Script** است. **Shell** برنامه‌ای است که دستورهای تو را می‌گیرد و به سرآشپز می‌رساند؛ رایج‌ترینش **Bash** است. وقتی چند دستور را پشت سر هم در یک فایل می‌نویسی، یک اسکریپت ساخته‌ای که با یک فرمان همه را اجرا می‌کند (= **Automation / خودکارسازی**).

```bash
#!/bin/bash
# این خطِ اول (shebang) می‌گوید با کدام مفسر اجرا شود

echo "صبح بخیر! آماده‌سازیِ آشپزخانه..."

for oven in 1 2 3; do          # حلقه: روی هر فر
  echo "روشن کردن فرِ $oven"
done

if [ -f /etc/menu.txt ]; then  # شرط: اگر فایلِ منو وجود داشت
  echo "منوی امروز آماده است"
else
  echo "هشدار: منو پیدا نشد!"
fi
```

برای اجرا، اول باید کاغذ را «اجراشدنی» کنیم (همان اجازه‌ی `x`)، بعد صدایش بزنیم:

```bash
chmod +x morning.sh
./morning.sh
```

همین چند خط، کاری را که سام نیم‌ساعت دستی می‌کرد، در یک ثانیه و **بدون اشتباه** انجام می‌دهد. این قلبِ DevOps و اتوماسیون است.

## صحنه‌ی ۱۱ — تابلوی اعلانات (Environment Variables)

تابلوی اعلاناتِ روی دیوار همان **متغیرهای محیطی** (Environment Variables) هستند: یک‌سری جفتِ «اسم=مقدار» که در محیطِ اجرا موجودند و **هر پروسه** می‌تواند بخواندشان — بدون اینکه از کسی بپرسد.

```bash
echo $HOME        # مسیرِ خانه‌ی من: /home/sam
echo $PATH        # سرآشپز برنامه‌ها را در این پوشه‌ها می‌گردد
export API_KEY="secret123"   # یک اعلانِ جدید روی تابلو بزن
env               # کلِ تابلو را ببین
```

مهم‌ترین اعلان، **`PATH`** است: فهرستی از کشوهایی که سرآشپز وقتی دستوری مثل `ls` می‌شنود، در آن‌ها دنبالِ برنامه می‌گردد. وقتی فرزندی (پروسه‌ی جدید) با `fork` ساخته می‌شود، **رونوشتی از تابلوی والدش** را به ارث می‌برد — برای همین تنظیماتِ محیط به برنامه‌ها منتقل می‌شوند.

> به همین دلیل رمزها و کلیدهای API را معمولاً در متغیرهای محیطی می‌گذارند، نه داخلِ کد — تابلو از کد جداست و راحت‌تر عوض می‌شود.

## صحنه‌ی ۱۲ — ایستگاه‌های همیشه‌روشن (Services و systemd)

کارهایی که نباید هیچ‌وقت تعطیل شوند، **Service** (یا daemon) نام دارند: وب‌سرور، دیتابیس، SSH. در Linuxِ مدرن، مدیرِ این ایستگاه‌ها **systemd** است — دستیارِ ارشدِ سرآشپز که اولین پروسه‌ای است که هنگامِ روشن شدن سیستم اجرا می‌شود (`PID 1`) و بقیه‌ی سرویس‌ها را راه می‌اندازد.

هر سرویس یک فایلِ تعریف (`unit file`) دارد:

```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=برنامه‌ی رستورانِ سام
After=network.target

[Service]
ExecStart=/usr/bin/python3 /opt/myapp/run.py
Restart=always          # اگر افتاد، فوری دوباره روشنش کن!
User=sam

[Install]
WantedBy=multi-user.target
```

و فرمان‌های روزمره برای مدیریتشان:

```bash
systemctl start  myapp     # روشن کن
systemctl stop   myapp     # خاموش کن
systemctl status myapp     # حالش چطور است؟
systemctl enable myapp     # هنگامِ بوت خودکار روشن شود
journalctl -u myapp -f     # لاگ‌های زنده‌ی این سرویس را ببین
```

ویژگیِ `Restart=always` همان قانونِ سرآشپز است: «اگر خاموش شد، فوری برش گردان» — به همین خاطر سرورها می‌توانند ماه‌ها بدون قطعی کار کنند.

## 🗺️ کل سفر در یک نگاه

```
تو برنامه‌ای را اجرا می‌کنی
   │
   ▼
١. سرآشپز (OS) یک آشپز (Process / PID) می‌سازد  ← fork + exec
   │
   ▼
٢. به او گوشه‌ای از میز (Memory) می‌دهد         ← virtual address space
   │   (میز کم آمد؟ → انباری/Swap)
   ▼
٣. روی اجاق (CPU) نوبتی می‌گذاردش              ← scheduler + context switch
   │
   ▼
٤. آشپز برای هر منبعِ مشترک اجازه می‌گیرد        ← system calls (open/read/write)
   │   (کشوها = File System، با کلید = Permissions)
   ▼
٥. کارهای تکراری با چک‌لیست خودکار می‌شوند        ← Bash script + Env vars
   │
   ▼
٦. ایستگاه‌های حیاتی همیشه‌روشن می‌مانند           ← systemd services
```

همه‌ی این‌ها هر لحظه، برای صدها پروسه، هم‌زمان دارد اتفاق می‌افتد — و تو حتی متوجهش نمی‌شوی!
