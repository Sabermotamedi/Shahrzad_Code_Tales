# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی کتابخانه‌ی بزرگ 🔍📚

> همان قصه‌ی سام و کتابخانه، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعی *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — کتابخانه واقعاً چیست؟ (Database، Table، Row)

«کتابخانه» همان **دیتابیس** (Database) است. هر **قفسه** یک **جدول** (Table) است، هر **کتاب** یک **ردیف** (Row) و **برگه‌ی مشخصاتِ** هم‌شکلِ هر کتاب، همان **ستون‌ها** (Columns) هستند.

| در قصه | در دنیای واقعی |
|---|---|
| کتابخانه | دیتابیس (Database) |
| قفسه | جدول (Table) |
| کتاب | ردیف (Row / Record) |
| برگه‌ی مشخصات (اسم، شماره، رنگ) | ستون‌ها (Columns) و نوعشان (Schema) |
| شماره‌ی یکتای هر کتاب | کلید اصلی (Primary Key) |

```sql
CREATE TABLE students (
    id    SERIAL PRIMARY KEY,      -- شماره‌ی یکتای هر دانش‌آموز
    name  TEXT NOT NULL,
    city  TEXT
);
```

اینکه همه‌ی کتاب‌های یک قفسه «برگه‌ی هم‌شکل» دارند، یعنی دیتابیس‌های رابطه‌ای (Relational / SQL) یک **Schema** ثابت دارند: ستون‌ها و نوعشان از پیش مشخص است.

## صحنه‌ی ۲ — جعبه‌ی کارت‌ها (Index)

وقتی سام دنبال «گربه‌ی نارنجی» بود، دو راه داشت:

```
بدون جعبه‌ی کارت:  کتاب اول؟ نه. دوم؟ نه. ... هزارمین؟ بله!   (Full Table Scan — کُند)
با جعبه‌ی کارت:     برو حرف «گ» → مستقیم به قفسه‌ی ۳         (Index Lookup — سریع)
```

**جعبه‌ی کارتِ مرتب‌شده** همان **ایندکس** (Index) است. ایندکس یک نسخه‌ی **مرتب‌شده** از یک ستون است که فقط می‌گوید «این مقدار، در کدام ردیف است». چون مرتب است، دیتابیس می‌تواند مثل ورق زدنِ دیکشنری، نصفِ نصفِ نصف کند و فوری برسد (به این می‌گویند ساختار **B-tree**).

```sql
CREATE INDEX idx_students_name ON students(name);
-- حالا WHERE name = 'نرگس' به‌جای دیدن همه‌ی کتاب‌ها، مستقیم می‌پرد
```

گاهی یک کارت کافی نیست. خانم کتابدار یک جعبه‌ی کارت داشت که اول بر اساسِ **نامِ خانوادگی** و بعد بر اساسِ **نام** مرتب بود — این **ایندکسِ ترکیبی** (Composite Index) است. با آن می‌توانی «احمدیِ سام» را فوری پیدا کنی، اما اگر فقط *نامِ* «سام» را بدانی (بدونِ فامیل)، این جعبه کمکی نمی‌کند — چون از سمتِ چپ (فامیل) مرتب شده. (به این می‌گویند قانونِ leftmost-prefix)

و گاهی کارت آن‌قدر کامل است که اصلاً لازم نیست بروی سرِ قفسه! اگر روی خودِ کارت همان چیزی که می‌خواستی (مثلاً عنوانِ کتاب) نوشته شده باشد، جوابت را همان‌جا می‌گیری و سفرِ تا قفسه حذف می‌شود — این **ایندکسِ پوششی** (Covering Index) است.

> نکته: ایندکس مجانی نیست! هر بار کتابِ تازه‌ای اضافه شود، باید **کارتش هم** در جعبه جا بیفتد. پس ایندکسِ زیاد، نوشتن را کُند می‌کند. (جزئیاتِ B-tree، composite و covering در سطح ۳)

## صحنه‌ی ۳ — چسباندنِ قفسه‌ها (JOIN)

یادت هست خانم کتابدار قفسه‌ی **امانت‌ها** و قفسه‌ی **دانش‌آموزها** را به هم چسباند؟ این عملِ «چسباندن» در SQL اسمش **JOIN** است.

```
قفسه‌ی امانت‌ها (loans)        قفسه‌ی دانش‌آموزها (students)
┌────┬───────────┐            ┌────┬─────────┐
│book│student_id │            │ id │ name    │
├────┼───────────┤            ├────┼─────────┤
│ 7  │    12     │──چسباندن──▶│ 12 │ نرگس    │
│ 9  │    12     │            │ 30 │ سام     │
└────┴───────────┘            └────┴─────────┘
            student_id ها با هم تطبیق داده می‌شوند
```

```sql
SELECT students.name, loans.book
FROM loans
JOIN students ON loans.student_id = students.id;
-- خروجی: نرگس | 7  و  نرگس | 9
```

اما گاهی سؤال فرق می‌کند، و نوعِ چسباندن هم فرق می‌کند:

| نوع JOIN | در قصه یعنی | چه برمی‌گرداند |
|---|---|---|
| `INNER JOIN` | فقط کتاب‌هایی که **هم** امانت‌گیرنده دارند **هم** اسمشان در قفسه هست | فقط ردیف‌های جفت‌شده |
| `LEFT JOIN` | **همه‌ی امانت‌ها**، حتی اگر دانش‌آموزش پیدا نشد | همه‌ی چپ + جفتِ راست (یا خالی) |
| `RIGHT JOIN` | **همه‌ی دانش‌آموزها**، حتی آن‌ها که چیزی امانت نگرفته‌اند | همه‌ی راست + جفتِ چپ (یا خالی) |
| `FULL OUTER JOIN` | **هم** امانت‌های بی‌صاحب **هم** دانش‌آموزهای بی‌کتاب | همه‌ی هر دو طرف |

```
INNER ∩          LEFT ⟕         RIGHT ⟖        FULL ⟗
 فقط مشترک      همه‌ی چپ       همه‌ی راست     همه‌ی هر دو
```

## صحنه‌ی ۴ — شمردن و خلاصه کردن (GROUP BY، HAVING)

سام پرسید: «هر دانش‌آموز **چند تا** کتاب امانت گرفته؟» این یعنی کتاب‌ها را **دسته‌بندی** کن (هر دانش‌آموز یک دسته) و در هر دسته **بشمار**. این کارِ `GROUP BY` و توابعِ **خلاصه‌کننده** (Aggregation: `COUNT`, `SUM`, `AVG`, `MAX`) است.

```sql
SELECT student_id, COUNT(*) AS book_count
FROM loans
GROUP BY student_id;            -- هر دانش‌آموز یک ردیف، با تعداد کتاب‌هایش
```

حالا سام گفت: «فقط آن‌هایی را بگو که **بیشتر از ۳ تا** گرفته‌اند.» اینجا یک نکته‌ی ظریف است:

```sql
SELECT student_id, COUNT(*) AS book_count
FROM loans
WHERE book IS NOT NULL          -- فیلتر روی تک‌تک کتاب‌ها (قبل از دسته‌بندی)
GROUP BY student_id
HAVING COUNT(*) > 3;            -- فیلتر روی دسته‌ها (بعد از دسته‌بندی)
```

| در قصه | کلمه‌ی فنی |
|---|---|
| «فقط کتاب‌های سالم را بشمار» (شرط روی تک‌کتاب) | `WHERE` — قبل از گروه‌بندی |
| «فقط دانش‌آموزهای پُرکتاب را نشانم بده» (شرط روی دسته) | `HAVING` — بعد از گروه‌بندی |

## صحنه‌ی ۵ — سؤال داخل سؤال (Subquery)

سام پرسید: «اسمِ آن دانش‌آموزی که **بیشترین** کتاب را گرفته چیست؟» این دو سؤال است که تو در توست:

```sql
SELECT name FROM students
WHERE id = (
    SELECT student_id FROM loans          -- سؤالِ داخلی: پُرکتاب‌ترین کیست؟
    GROUP BY student_id
    ORDER BY COUNT(*) DESC
    LIMIT 1
);
```

اول سؤالِ **داخلی** جواب می‌دهد (مثلاً «شماره ۱۲»)، بعد سؤالِ **بیرونی** از آن استفاده می‌کند. به این می‌گویند **Subquery** — یک پرس‌وجو که جوابش، ورودیِ پرس‌وجوی دیگری است.

## صحنه‌ی ۶ — هر چیز فقط یک‌جا (Normalization)

یادت هست خانم کتابدار گفت آدرس نرگس را **فقط یک‌جا** می‌نویسد؟ این طلایی‌ترین قانونِ مدل‌سازیِ داده است و اسمش **نرمال‌سازی** (Normalization) است: حذفِ تکرار، تا هر واقعیت فقط در یک نقطه باشد.

سه قدمِ معروف:

| فرم | قانون | در قصه |
|---|---|---|
| **1NF** | هر خانه فقط **یک** مقدار (نه لیستِ تو در تو) | روی برگه‌ی هر کتاب ننویس «امانت‌گیرنده: نرگس، سام، علی» — هر امانت یک ردیفِ جدا |
| **2NF** | 1NF + هر ستون به **کلِ** کلید وابسته باشد | اگر کلید «کتاب+دانش‌آموز» است، نامِ کتاب به دانش‌آموز ربطی ندارد → ببرش جای خودش |
| **3NF** | 2NF + هیچ ستونی به ستونِ **غیرکلید** دیگری وابسته نباشد | «کدِ شهر → نامِ شهر» را در جدولِ دانش‌آموز نگه ندار؛ جدولِ جداگانه‌ی شهرها بساز |

```
قبل (تکراری، خطرناک):                    بعد (نرمال‌شده):
loans(book_name, student_name,           students(id, name, city)
      student_address, student_city)     loans(book_id, student_id)
   ▲ آدرس نرگس صد بار تکرار شده!             ▲ آدرس فقط یک‌جا
```

اگر آدرس صد جا تکرار شود و نرگس اسباب‌کشی کند، یا باید صد ردیف را عوض کنی (و یکی را جا می‌اندازی → داده‌ی متناقض)، یا فاجعه. نرمال‌سازی جلوی این **Anomaly** را می‌گیرد.

## صحنه‌ی ۷ — انواعِ رابطه بین قفسه‌ها (Relationships)

وقتی قفسه‌ها به هم وصل می‌شوند، سه جور رابطه ممکن است:

| رابطه | یعنی | مثالِ کتابخانه | پیاده‌سازی |
|---|---|---|---|
| **1:1** | هر طرف فقط یک جفت دارد | هر دانش‌آموز یک کارتِ عضویت | کلید خارجیِ یکتا (UNIQUE FK) |
| **1:N** | یکی به چندتا | یک نویسنده، چندین کتاب | کلید خارجی در سمتِ «چند» |
| **N:M** | چندتا به چندتا | هر دانش‌آموز چند کتاب، هر کتاب چند خواننده | **جدولِ واسط** (Junction Table) |

```
N:M را نمی‌شود مستقیم وصل کرد. یک قفسه‌ی وسطی می‌خواهد:

students ──1:N──▶ loans ◀──N:1── books
                 (جدول واسط: student_id + book_id)
```

```sql
-- جدول واسط برای رابطه‌ی N:M
CREATE TABLE loans (
    student_id INT REFERENCES students(id),
    book_id    INT REFERENCES books(id),
    PRIMARY KEY (student_id, book_id)
);
```

«کلید خارجی» (**Foreign Key**) همان «شماره‌ی ۱۲» روی برگه‌ی امانت بود — اشاره به ردیفی در قفسه‌ی دیگر.

## صحنه‌ی ۸ — خانم کتابدار و کتابِ آخر (Transaction و ACID)

یادت هست خانم کتابدار یا **همه‌ی** کار را می‌کرد یا **هیچ‌کدام**، و نمی‌گذاشت دو نفر سرِ یک کتاب دعوا کنند؟ این مجموعه از کارها که باید **یکپارچه** انجام شوند، یک **تراکنش** (Transaction) است.

مثالِ کلاسیک: انتقال یک کتاب از انبار به دستِ نرگس = دو کار که باید **با هم** انجام شوند:

```sql
BEGIN;                                          -- شروع تراکنش
  UPDATE books SET available = available - 1 WHERE id = 7;
  INSERT INTO loans(student_id, book_id) VALUES (12, 7);
COMMIT;                                          -- یا همه با هم ثبت...
-- اگر وسطِ کار برق رفت:  ROLLBACK → انگار هیچ‌کدام اتفاق نیفتاد
```

تضمین‌های خانم کتابدار، چهار حرفِ **ACID** است:

| حرف | اسم | قولِ خانم کتابدار |
|---|---|---|
| **A** | Atomicity (تجزیه‌ناپذیری) | «یا همه‌ی کارها، یا هیچ‌کدام — نصفه ول نمی‌کنم» |
| **C** | Consistency (سازگاری) | «قانون‌ها همیشه برقرارند — کتاب منفی نمی‌شود» |
| **I** | Isolation (انزوا) | «تا کارم با این کتاب تمام نشده، کسِ دیگری دستش نزند» |
| **D** | Durability (ماندگاری) | «وقتی در دفتر ثبت کردم، حتی اگر برق برود، گم نمی‌شود» |

## صحنه‌ی ۹ — چقدر سختگیر باشیم؟ (Isolation Levels و سه آشوب)

آن «کسِ دیگری دستش نزند» چقدر سختگیرانه باشد؟ هرچه سختگیرتر، **امن‌تر ولی کُندتر**. سه جور بدبختی ممکن است پیش بیاید:

| آشوب (Anomaly) | در قصه |
|---|---|
| **Dirty Read** | خواندنِ نوشته‌ای که هنوز قطعی نشده — «نرگس کتاب را برداشت ولی بعد پشیمان شد و گذاشت سرجایش؛ تو دیدی کتاب رفته!» |
| **Non-Repeatable Read** | یک ردیف را دو بار خواندی، وسطش یکی **عوضش** کرد — «قیمت کتاب اول ۵ بود، دوباره نگاه کردی شد ۷!» |
| **Phantom Read** | همان فیلتر را دو بار اجرا کردی، وسطش یکی ردیفِ **تازه** اضافه کرد — «اول ۳ کتابِ گربه بود، دوباره شمردی شد ۴!» |

سطوح انزوا، از شُل تا سفت:

```
READ UNCOMMITTED  → همه چیز آزاد        (dirty read هم ممکن)
READ COMMITTED    → فقط نوشته‌ی قطعی    (dirty read جلوگیری)
REPEATABLE READ   → عکسِ ثابت از داده   (non-repeatable هم جلوگیری)
SERIALIZABLE      → انگار تک‌تک، پشت سر هم  (همه‌ی آشوب‌ها جلوگیری — کُندترین)
```

> نکته‌ی مصاحبه: سطحِ پیش‌فرضِ اکثر دیتابیس‌ها `READ COMMITTED` است — تعادلِ خوبِ بینِ امنیت و سرعت. (رفتارِ دقیقِ PostgreSQL در سطح ۳)

## صحنه‌ی ۱۰ — دیدنِ نقشه‌ی راهِ پرس‌وجو (EXPLAIN)

سام پرسید: «خانم کتابدار، از کجا می‌فهمید کدام راهِ پیدا کردنِ کتاب سریع‌تر است؟» خانم کتابدار گفت: «قبل از اینکه راه بیفتم، **نقشه‌ام را می‌کشم**: از جعبه‌ی کارت بروم یا قفسه به قفسه بگردم؟»

این نقشه همان `EXPLAIN` است — دیتابیس **پلنِ اجرایش** را نشانت می‌دهد، بدون اینکه واقعاً اجرا کند:

```sql
EXPLAIN SELECT * FROM students WHERE name = 'نرگس';
-- اگر ببینی "Seq Scan" یعنی دارد قفسه‌به‌قفسه می‌گردد (کُند، احتمالاً ایندکس کم است)
-- اگر ببینی "Index Scan" یعنی از جعبه‌ی کارت رفته (سریع)
```

## صحنه‌ی ۱۱ — شعبه‌ها (Replication، Sharding، Partitioning)

وقتی کتابخانه شلوغ شد، خانم کتابدار سه ترفند زد:

**۱) شعبه‌ی کمکی = Replication (تکثیر):**
کپیِ کاملِ کتابخانه. یک شعبه‌ی **اصلی** (Primary) که فقط آنجا کتابِ تازه اضافه می‌شود، و چند شعبه‌ی **کپی** (Replica) که فقط برای **خواندن** هستند.

```
نوشتن (write) ─────▶ Primary ──کپی می‌کند──▶ Replica 1 (فقط خواندن)
                       │                  └─▶ Replica 2 (فقط خواندن)
خواندن (read) ─────────────────────────────▶ از replicaها پخش می‌شود
```

| روشِ کپی کردن | یعنی | بده‌بستان |
|---|---|---|
| **Sync** (هم‌زمان) | تا کپی نرسد، نوشتن تمام‌شده حساب نمی‌شود | امن ولی کُند |
| **Async** (ناهم‌زمان) | اول می‌نویسد، بعداً کپی می‌رساند | سریع ولی replica ممکن است کمی **عقب** باشد (Replication Lag) |

**۲) تقسیمِ کتاب‌ها بین ساختمان‌ها = Sharding:**
«حرف الف تا میم در ساختمانِ شمالی، ن تا ی در جنوبی.» هر ساختمان (**Shard**) فقط **بخشی** از داده را دارد، روی سرورِ جدا. وقتی داده آن‌قدر زیاد است که در یک سرور جا نمی‌شود.

**۳) تقسیمِ یک قفسه‌ی بزرگ = Partitioning:**
وقتی **یک** قفسه خودش غول‌پیکر است (مثلاً امانت‌های ۱۰ سال)، آن را به تکه‌های منطقی می‌بُری (مثلاً هر سال یک تکه) — ولی **در همان کتابخانه**.

> فرقِ ظریف: **Partitioning** = شکستنِ یک جدول به تکه‌ها (معمولاً در یک سرور). **Sharding** = پخشِ آن تکه‌ها روی **چند سرورِ جدا**. شاردینگ تقریباً همان پارتیشن‌بندیِ افقی است که روی ماشین‌های مختلف پخش شده.

## صحنه‌ی ۱۲ — شبِ طوفانی (CAP Theorem)

یادت هست تلفنِ بین شعبه‌ها قطع شد و خانم کتابدار باید بینِ «در باز» و «جوابِ دقیق» یکی را انتخاب می‌کرد؟ این همان **قضیه‌ی CAP** است. وقتی سیستم **توزیع‌شده** است (چند سرور)، سه چیز می‌خواهی ولی فقط **دو تا** را هم‌زمان می‌توانی داشته باشی:

| حرف | اسم | یعنی |
|---|---|---|
| **C** | Consistency | همه‌ی شعبه‌ها همیشه **یک** جواب می‌دهند |
| **A** | Availability | هر شعبه همیشه **جواب** می‌دهد (هرچند شاید قدیمی) |
| **P** | Partition Tolerance | اگر تلفن قطع شد، سیستم **نمی‌میرد** |

```
شبکه که قطع شد (P اجباری است، چون شبکه واقعاً قطع می‌شود)، باید انتخاب کنی:
   CP → در را ببند تا جوابِ غلط ندهی  (Consistency را نگه دار، Availability را فدا کن)
   AP → باز بمان، حتی اگر جوابت کمی قدیمی است  (Availability را نگه دار، Consistency را فدا کن)
```

> نکته‌ی مصاحبه: در عمل `P` همیشه اجباری است (شبکه قطع می‌شود)، پس انتخابِ واقعی همیشه بینِ **C و A** است.
