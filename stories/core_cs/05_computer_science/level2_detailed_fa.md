# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی اتاقِ اسباب‌بازی‌ها 🔍🧸

> همان قصه‌ی سام و اتاقِ اسباب‌بازی‌اش، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعیِ کامپیوتر *دقیقاً* به چه چیزی اشاره دارد. اسم‌های فنی را کم‌کم وارد می‌کنیم (واژه‌ی انگلیسی را داخلِ متن می‌آوریم).

---

## صحنه‌ی ۱ — ردیفِ شماره‌دار همان آرایه است (Array)

«ردیفِ صافِ ماشین‌ها با شماره‌ی زیرشان» در دنیای واقعی یک **آرایه** (Array) است: یک تکه‌ی **پیوسته** از حافظه که خانه‌هایش پشتِ‌سرِ‌هم‌اند و هرکدام یک **اندیس** (Index) دارند که از `0` شروع می‌شود.

```
اندیس:   0     1     2     3     4
        ┌────┬────┬────┬────┬────┐
آرایه:  │ 🚗 │ 🚙 │ 🏎️ │ 🚓 │ 🚕 │
        └────┴────┴────┴────┴────┘
```

چرا «ماشینِ شماره‌ی ۵» فوری پیدا می‌شود؟ چون کامپیوتر آدرسِ خانه را با یک ضرب‌وجمعِ ساده حساب می‌کند: `آدرسِ شروع + ۵ × اندازه‌ی هر خانه`. به این می‌گویند **دسترسی تصادفی** (Random Access) و هزینه‌اش **O(1)** است — یعنی فرقی نمی‌کند آرایه ۱۰ خانه باشد یا یک میلیون.

چرا «جا دادنِ وسط» سخت بود؟ چون باید همه‌ی عناصرِ بعدی را یک خانه به جلو **شیفت** بدهی → **O(n)**.

```python
toys = ["🚗", "🚙", "🏎️"]
print(toys[2])        # دسترسی مستقیم — O(1)
toys.insert(1, "🚓")  # جا دادنِ وسط — O(n) چون بقیه شیفت می‌خورند
toys.append("🚕")     # افزودن به ته — معمولاً O(1)
```

## صحنه‌ی ۲ — عروسک‌های دست‌به‌دست همان لیست پیوندی‌اند (Linked List)

«عروسک‌هایی که دستِ عروسکِ بعدی را گرفته‌اند» یک **لیست پیوندی** (Linked List) است. اینجا خبری از تکه‌ی پیوسته‌ی حافظه نیست؛ هر عروسک یک **گره** (Node) است که دو چیز دارد: **داده‌اش** و یک **اشاره‌گر** (Pointer/Reference) به گره بعدی.

```
┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐
│ 🧸  │───▶│ 🤖  │───▶│ 🦖  │───▶│ ⚽  │───▶ None
│ next│    │ next│    │ next│    │ next│
└─────┘    └─────┘    └─────┘    └─────┘
 head
```

| در قصه | در واقعیت |
|---|---|
| باز کردنِ دو دست و چپاندنِ عروسک تازه وسط | تغییرِ دو اشاره‌گر → درج O(1) (اگر گره را در دست داشته باشی) |
| از اولِ زنجیره دست‌به‌دست جلو رفتن | پیمایش از head → دسترسیِ عنصرِ k اُم O(n) |

پس **توازنِ معامله** (Trade-off) روشن است: آرایه در دسترسیِ مستقیم برنده است، لیست پیوندی در درج/حذفِ وسط (وقتی گره را داری). لیست پیوندی **اندیس** ندارد، پس برای «عنصرِ پنجم» باید بشماری.

```python
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

# 🧸 -> 🤖 -> 🦖
head = Node("🧸")
head.next = Node("🤖")
head.next.next = Node("🦖")
```

## صحنه‌ی ۳ — بشقاب‌های روی هم همان پشته‌اند (Stack)

«بشقاب‌های تلنبارشده» یک **پشته** (Stack) با قانونِ **LIFO** هستند: *Last In, First Out* — آخرین چیزی که گذاشتی، اولین چیزی است که برمی‌داری. فقط دو کارِ اصلی دارد:

- **push**: گذاشتنِ یک چیز روی پشته
- **pop**: برداشتنِ چیزِ بالایی

```
   push ▼      ▲ pop
        ┌──────┐
   top →│  🍽️3 │   ← آخرین که آمد، اولین که می‌رود
        ├──────┤
        │  🍽️2 │
        ├──────┤
        │  🍽️1 │
        └──────┘
```

هر دو کار **O(1)**اند. پشته همه‌جا هست: دکمه‌ی **Undo**، دکمه‌ی **Back** مرورگر، و مهم‌تر از همه **Call Stack** (که در صحنه‌ی بازگشت می‌بینیم).

```python
stack = []
stack.append("🍽️1")   # push
stack.append("🍽️2")   # push
print(stack.pop())    # 🍽️2  ← آخرین که آمد
```

## صحنه‌ی ۴ — صفِ سُرسُره همان صف است (Queue)

«صفِ سُرسُره» یک **صف** (Queue) با قانونِ **FIFO** است: *First In, First Out* — هرکه زودتر آمد، زودتر می‌رود. دو کارِ اصلی:

- **enqueue**: افزودن به **ته** صف
- **dequeue**: برداشتن از **سرِ** صف

```
 dequeue ◀── [ 🧒1 ][ 🧒2 ][ 🧒3 ] ◀── enqueue
  (زودتر آمده)              (تازه آمده)
```

صف قلبِ زمان‌بندیِ کارها (Task Scheduling)، صفِ چاپگر، و — مهم — پیمایشِ **BFS** در گراف است. نکته‌ی عملی: از `list` پایتون برای صف استفاده نکن (حذف از سر O(n) است)؛ از `collections.deque` استفاده کن که هر دو سرش O(1) است.

```python
from collections import deque
q = deque()
q.append("🧒1")        # enqueue
q.append("🧒2")
print(q.popleft())    # 🧒1  ← اولین که آمد
```

## صحنه‌ی ۵ — کمدِ جادویی همان جدول هش است (Hash Table)

«کمدی که اسمِ اسباب‌بازی را بگویی، کشویش باز می‌شود» یک **جدول هش** (Hash Table) است — همان `dict` در پایتون. ترفندِ جادو این است: یک **تابعِ هش** (Hash Function) اسم را می‌گیرد و یک **عدد** بیرون می‌دهد، و آن عدد می‌گوید کشو (Bucket) کجاست. پس لازم نیست بگردیم؛ مستقیم می‌رویم سرِ کشو → **O(1)** به‌طور میانگین.

```
"خرسی" ──hash──▶ 3 ──▶ bucket[3] = 🧸
"ربات" ──hash──▶ 0 ──▶ bucket[0] = 🤖

         bucket
        ┌───┬──────┐
      0 │ 🤖│      │
      1 │   │      │
      2 │   │      │
      3 │ 🧸│      │
      ...
```

### وقتی دو اسم یک کشو می‌خواهند (Collision)

«دو اسباب‌بازی که اسمشان شبیه است و یک کشو می‌خواهند» همان **برخورد** (Collision) است: دو کلیدِ مختلف، یک شماره‌ی هش. دو راهِ حل:

- **زنجیره‌کردن (Chaining):** در آن کشو یک لیستِ کوچک بگذار و هر دو را کنار هم نگه دار. موقع خواندن، لیست را بگرد.
- **آدرس‌دهیِ باز (Open Addressing):** اگر کشو پر بود، برو کشوی بعدیِ خالی.

نکته‌ی مهم: اگر **خیلی** برخورد داشته باشی (تابعِ هشِ بد یا جدولِ پُر)، همه در یک کشو جمع می‌شوند و جستجو از O(1) به **O(n)** سقوط می‌کند. برای همین جدول هش با پُرشدن، **خودش را بزرگ‌تر و دوباره می‌چیند** (Resize / Rehash).

| در قصه | در واقعیت |
|---|---|
| فریاد زدنِ اسم | اعمالِ تابعِ هش روی کلید |
| کشوی مخصوص | bucket / slot |
| دو اسمِ شبیه، یک کشو | collision |
| کشوی بزرگ‌تر با هر دو کنار هم | chaining |

```python
cabinet = {}
cabinet["خرسی"] = "🧸"   # درج — O(1) میانگین
print(cabinet["خرسی"])  # خواندن — O(1) میانگین
print("ربات" in cabinet)  # عضویت — O(1) میانگین
```

## صحنه‌ی ۶ — درختِ خانوادگی همان درخت است (Tree)

«درختِ خانوادگیِ خرس‌ها» یک **درخت** (Tree) است: یک **ریشه** (Root) در بالا، و هر گره **فرزندانی** (Children) دارد. وقتی هر گره **حداکثر دو فرزند** داشته باشد، اسمش **درخت دودویی** (Binary Tree) است.

```
            بابابزرگ (root)
           /            \
       خرس‌چپ          خرس‌راست
       /    \           /     \
     🧸     🐻        🐼      🐨   ← برگ‌ها (leaves)
```

### درختِ جستجوی دودویی (BST)

اگر درخت را با یک **قانون** بچینی — «هر چیزِ **کوچک‌تر** از یک گره می‌رود سمتِ **چپ**، هر چیزِ **بزرگ‌تر** سمتِ **راست**» — می‌شود **درخت جستجوی دودویی** (Binary Search Tree). آن‌وقت جستجو دقیقاً مثل «هر بار یک طرف را انتخاب کن»ِ قصه است: در هر گره نصفِ احتمالات را دور می‌ریزی → **O(log n)**.

```
        8
       / \
      3   10        دنبالِ ۶ می‌گردی:
     / \    \       ۶ < ۸ → چپ، ۶ > ۳ → راست، رسیدی به ۶ ✅
    1   6    14
```

### چرا «متوازن» مهم است (Balanced)

اگر اعداد را مرتب وارد کنی (۱، ۲، ۳، ۴...)، درخت **کج** می‌شود و عملاً به یک لیست پیوندی تبدیل می‌شود → جستجو دوباره **O(n)**! اینجا همان «شاخه‌های یک‌طرفه‌ی کج»ِ قصه است.

```
کج (بد):  1            متوازن (خوب):    2
            \                          / \
             2                        1   3
              \                            \
               3                            4
                \
                 4   ← مثل لیست پیوندی، O(n)
```

راه‌حل: درخت‌های **خودمتوازن** (Self-balancing) مثل **AVL** و **Red-Black Tree** که موقع درج/حذف خودشان را صاف می‌کنند تا ارتفاع همیشه نزدیکِ `log n` بماند.

## صحنه‌ی ۷ — نقشه‌ی دوستی‌ها همان گراف است (Graph)

«نقشه‌ی دوستیِ اسباب‌بازی‌ها» یک **گراف** (Graph) است: مجموعه‌ای از **گره/رأس** (Vertex/Node) که با **یال** (Edge) به هم وصل‌اند.

- **بدون‌جهت (Undirected):** دوستیِ دوطرفه — «من دوستِ تو، تو دوستِ من». یال دو طرفه است.
- **جهت‌دار (Directed):** «ربات، خرسی را دوست دارد ولی برعکس نه». یال یک پیکانِ یک‌طرفه است.

دو راهِ ذخیره‌ی گراف:

```
لیستِ مجاورت (Adjacency List)        ماتریسِ مجاورت (Adjacency Matrix)
🧸: [🤖, 🦖]                              🧸  🤖  🦖
🤖: [🧸]                            🧸 [ 0   1   1 ]
🦖: [🧸]                            🤖 [ 1   0   0 ]
                                    🦖 [ 1   0   0 ]
```

| | لیست مجاورت | ماتریس مجاورت |
|---|---|---|
| حافظه | O(V + E) — صرفه‌جو برای گرافِ تُنُک (Sparse) | O(V²) — همیشه، حتی بی‌یال |
| «آیا A و B یال دارند؟» | O(درجه‌ی A) | O(1) — یک خانه |
| پیمایشِ همسایه‌ها | عالی | باید کلِ ردیف را بخوانی |

قاعده‌ی سرانگشتی: گرافِ واقعی معمولاً **تُنُک** است، پس **لیست مجاورت** انتخابِ پیش‌فرض است.

## صحنه‌ی ۸ — راه‌های مرتب‌کردن همان الگوریتم‌های مرتب‌سازی‌اند (Sorting)

«راهِ تنبل» و «راهِ باهوش»ِ قصه، دو فلسفه‌ی متفاوتِ **مرتب‌سازی** (Sorting) هستند:

- **مرتب‌سازی حبابی (Bubble Sort) — راهِ تنبل:** هر دو عنصرِ کنارِ هم را مقایسه کن و اگر بد بودند جابه‌جا کن؛ آن‌قدر دور بزن تا چیزی جابه‌جا نشود. **O(n²)** — برای هزار عنصر یعنی یک میلیون مقایسه!
- **مرتب‌سازی ادغامی (Merge Sort) — نصف‌کن:** آرایه را **مدام نصف** کن تا به تک‌عنصرها برسی، بعد دو نیمه‌ی مرتب را مثل زیپ **به هم بدوز** (Merge). همیشه **O(n log n)**، پایدار (Stable)، اما O(n) حافظه‌ی اضافه می‌خواهد.
- **مرتب‌سازی سریع (Quick Sort):** یک عنصر را **محور** (Pivot) بگیر، کوچک‌ترها را چپ و بزرگ‌ترها را راستش بریز (Partition)، بعد همین کار را با هر طرف تکرار کن. به‌طور میانگین **O(n log n)** و **در جا** (In-place، حافظه‌ی کم)، اما در بدترین حالت (محورِ بد) **O(n²)**.

```
Merge Sort روی [5,2,8,1]:
        [5, 2, 8, 1]
        /          \
     [5, 2]      [8, 1]        ← نصف کن
     /    \      /    \
   [5]   [2]  [8]   [1]        ← تک‌عنصرها (مرتب‌اند!)
     \   /      \   /
    [2, 5]      [1, 8]         ← مثل زیپ به هم بدوز
        \          /
        [1, 2, 5, 8]           ← خروجیِ مرتب
```

> در عمل، خودِ پایتون از **Timsort** (ترکیبِ هوشمندِ merge و insertion sort) استفاده می‌کند؛ `sorted()` همین است.

## صحنه‌ی ۹ — راه‌های پیدا کردن همان جستجوست (Searching)

«یکی‌یکی نگاه کردن» در برابر «نصف‌کردنِ صفِ مرتب»:

- **جستجوی خطی (Linear Search):** از اول تا آخر یکی‌یکی نگاه کن. روی هر داده‌ای کار می‌کند، اما **O(n)**.
- **جستجوی دودویی (Binary Search):** فقط روی داده‌ی **از قبل مرتب‌شده**. وسط را نگاه کن؛ اگر هدف کوچک‌تر بود برو نیمه‌ی چپ، اگر بزرگ‌تر بود نیمه‌ی راست. هر بار نصفِ داده حذف می‌شود → **O(log n)**.

```
دنبالِ ۷ در [1,3,5,7,9,11]:
  وسط=5 → ۷>۵ → نیمه‌ی راست [7,9,11]
  وسط=9 → ۷<۹ → نیمه‌ی چپ [7]
  پیدا شد ✅  (فقط ۲–۳ نگاه به‌جای ۶ تا)
```

### پیمایشِ گراف/درخت: BFS و DFS

برای گشتن در گراف یا درخت دو راهِ کلاسیک هست:

- **BFS (جستجوی سطح‌اول، Breadth-First):** لایه‌به‌لایه — اول همه‌ی همسایه‌های نزدیک، بعد دورترها. از **صف** (Queue) استفاده می‌کند. برای **کوتاه‌ترین مسیر** در گرافِ بی‌وزن عالی است.
- **DFS (جستجوی عمق‌اول، Depth-First):** تا ته یک شاخه برو، به بن‌بست که رسیدی برگرد. از **پشته** (Stack) — یا بازگشت (Recursion) — استفاده می‌کند. برای کشفِ مسیرها، تشخیصِ دور، و مرتب‌سازی توپولوژیک.

```python
from collections import deque
def bfs(graph, start):
    seen, q = {start}, deque([start])
    while q:
        node = q.popleft()      # صف → FIFO
        for nb in graph[node]:
            if nb not in seen:
                seen.add(nb)
                q.append(nb)
```

## صحنه‌ی ۱۰ — عروسکِ تو در تو همان بازگشت است (Recursion)

«هر عروسک را باز کن، بعد همین کار را با عروسکِ توییش بکن، تا به ریزترین برسی» همان **بازگشت** (Recursion) است: تابعی که **خودش را صدا می‌زند**، روی نسخه‌ی **کوچک‌ترِ** مسئله.

دو جزءِ حیاتی:

- **حالتِ پایه (Base Case):** «ریزترین عروسک که تویش چیزی نیست» — جایی که تابع **می‌ایستد**. بدونِ آن، تا بی‌نهایت خودش را صدا می‌زند و به **Stack Overflow** می‌رسد.
- **گامِ بازگشتی (Recursive Step):** صدا زدنِ خود روی مسئله‌ی کوچک‌تر.

هر صدا زدن یک **قابِ تابع** (Stack Frame) روی همان **Call Stack** (بشقاب‌های صحنه‌ی ۳!) می‌گذارد. وقتی به base case رسیدی، قاب‌ها یکی‌یکی از بالا برداشته می‌شوند (pop).

```python
def factorial(n):
    if n <= 1:          # حالتِ پایه — اینجا می‌ایستیم
        return 1
    return n * factorial(n - 1)   # گامِ بازگشتی
```

## صحنه‌ی ۱۱ — دفترچه‌ی یادداشت همان برنامه‌نویسی پویاست (Dynamic Programming)

«جوابِ سؤالِ سخت را در دفترچه نوشتن تا دوباره فکر نکنی» همان **به‌خاطرسپاری** (Memoization) است، پایه‌ی **برنامه‌نویسی پویا** (Dynamic Programming). ایده: وقتی یک مسئله **زیرمسئله‌های تکراری** دارد، جوابِ هر زیرمسئله را **یک بار** حساب کن و ذخیره کن.

### مثالِ کلاسیک ۱ — فیبوناچی

بازگشتِ ساده‌ی فیبوناچی، `fib(n-1)` و `fib(n-2)` را حساب می‌کند و این دو، خودشان زیرمسئله‌های **مشترکِ** فراوان دارند — درختِ محاسبه‌ای انفجاری با **O(2ⁿ)**! با دفترچه (memo)، هر `fib(k)` فقط یک بار حساب می‌شود → **O(n)**.

```python
def fib(n, memo={}):
    if n <= 1:
        return n
    if n in memo:           # دفترچه را نگاه کن
        return memo[n]
    memo[n] = fib(n-1, memo) + fib(n-2, memo)
    return memo[n]
```

### مثالِ کلاسیک ۲ — خردکردنِ پول (Coin Change)

«با سکه‌های ۱، ۵، ۱۰ تومانی، کم‌ترین تعدادِ سکه برای ساختنِ ۱۲ تومان چند تاست؟» اینجا هم زیرمسئله‌ها تکراری‌اند (مبلغِ ۷ تومان از مسیرهای مختلف تکرار می‌شود)، پس جوابِ هر مبلغ را در دفترچه نگه می‌داریم.

```python
def coin_change(coins, amount):
    dp = [0] + [float('inf')] * amount      # dp[m] = کم‌ترین سکه برای مبلغِ m
    for m in range(1, amount + 1):
        for c in coins:
            if c <= m:
                dp[m] = min(dp[m], dp[m - c] + 1)
    return dp[amount] if dp[amount] != float('inf') else -1
```

## صحنه‌ی ۱۲ — «اگر هزار تا بود چی؟» همان Big O است

سؤالِ آخرِ شبِ سام — «اگر تعداد ۱۰۰۰ برابر شد، کارم چقدر کندتر می‌شود؟» — دقیقاً همان چیزی است که **پیچیدگیِ زمانی** (Time Complexity) و نمادِ **Big O** اندازه می‌گیرند: نه زمانِ دقیق به ثانیه، بلکه **نرخِ رشدِ** کار وقتی ورودی (`n`) بزرگ می‌شود.

| Big O | اسم | تشبیهِ روزمره | اگر n=۱۰ → n=۱۰۰۰ |
|---|---|---|---|
| **O(1)** | ثابت | کشوی جادویی — مهم نیست چند تا اسباب‌بازی | هیچ تغییری |
| **O(log n)** | لگاریتمی | جستجوی دودویی — هر بار نصف | فقط کمی بیشتر (~۷ برابرِ گام) |
| **O(n)** | خطی | یکی‌یکی نگاه کردنِ همه | ۱۰۰ برابر |
| **O(n log n)** | خطی-لگاریتمی | مرتب‌سازیِ باهوش (merge/quick) | ~۷۰۰ برابر |
| **O(n²)** | درجه‌دو | مرتب‌سازیِ تنبل (هرکس با همه) | ۱۰٬۰۰۰ برابر! |
| **O(2ⁿ)** | نمایی | فیبوناچیِ بدونِ دفترچه | عملاً تا ابد 💀 |

و یک خواهرِ کوچک‌تر هم هست: **پیچیدگیِ فضایی** (Space Complexity) — «چقدر **جای اضافه** لازم دارم؟». مثلاً merge sort علاوه بر زمانِ O(n log n)، فضای اضافه‌ی O(n) می‌خواهد؛ ولی quick sort تقریباً در جا کار می‌کند. همیشه یک **توازنِ زمان در برابر فضا** (Time-Space Trade-off) هست: دفترچه‌ی فیبوناچی زمان را از O(2ⁿ) به O(n) آورد، اما به قیمتِ O(n) حافظه‌ی اضافه.

> **نکته‌ی مصاحبه:** Big O فقط **بزرگ‌ترین جمله** را نگه می‌دارد و ضریب‌ها را دور می‌ریزد. `O(3n² + 5n + 100)` می‌شود `O(n²)`، چون وقتی n خیلی بزرگ شود، فقط `n²` اهمیت دارد.
