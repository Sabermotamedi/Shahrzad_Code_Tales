# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی کارخانه‌ی کلیدها 🔍🏭

> همان کارخانه‌ی کلیدهای جادویی که سام دید، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخشِ قصه در دنیای واقعی *دقیقاً* چطور کار می‌کند. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — کلیدها واقعاً چه هستند؟ (Binary و عددها)

هر «کلیدِ روشن/خاموش» در دنیای واقعی یک **بیت** (bit) است: کوچک‌ترین واحدِ اطلاعات، با دو حالتِ `1` (روشن) و `0` (خاموش). داخلِ تراشه، این همان «ولتاژ هست / ولتاژ نیست» است.

| در قصه | در دنیای واقعی |
|---|---|
| یک کلید | یک **بیت** (bit) |
| هشت کلیدِ کنار هم | یک **بایت** (byte) — ۸ بیت |
| ترتیبِ کلیدها برای گفتنِ یک عدد | نمایشِ **باینری** (مبنای ۲) |

چرا مبنای ۲؟ چون ساختنِ سخت‌افزاری که فقط دو حالت را تشخیص دهد (هست/نیست) خیلی ساده‌تر و قابل‌اعتمادتر از ده حالت است.

**تبدیل باینری ↔ ده‌دهی.** هر رقمِ باینری از راست به چپ، ارزشِ ۲ به توانِ جای خودش را دارد:

```
باینری:  1 1 0 1
ارزش:    8 4 2 1
عدد:     8 + 4 + 0 + 1 = 13
```

**هگزادسیمال (Hex، مبنای ۱۶).** نوشتنِ بیت‌ها خسته‌کننده است (`11111111`)، پس برنامه‌نویس‌ها هر ۴ بیت را یک رقمِ هگز می‌نویسند (`0`–`9` و بعد `a`–`f`). مثلاً `11111111` می‌شود `ff`. به همین خاطر رنگ‌ها در وب `#ff0000` نوشته می‌شوند.

**عددهای منفی — متممِ دو (Two's Complement).** سؤال: با کلیدهایی که فقط `0` و `1` دارند، علامتِ منفی را کجا بگذاریم؟ ترفند: بالاترین بیت را **علامت** در نظر می‌گیریم و منفی‌ها را طوری می‌چینیم که جمعِ معمولی خودش درست کار کند. در یک بایتِ ۸ بیتی، `00000000` تا `01111111` می‌شود ۰ تا ۱۲۷، و `10000000` تا `11111111` می‌شود ۱۲۸- تا ۱-. زیبایی‌اش این است که `-1` می‌شود `11111111` و سخت‌افزار برای جمع و تفریق به مدارِ جداگانه نیاز ندارد.

**سرریز (Overflow).** کارگر فقط «تعدادِ معینی کلید» دارد. اگر در یک بایتِ علامت‌دارِ ۸ بیتی به بزرگ‌ترین عدد (`127`) یکی اضافه کنی، کلیدها سرریز می‌شوند و عدد **می‌پیچد** به `-128`! این همان باگِ معروفی است که هواپیمای جنگی را وسطِ خط استوا وارونه می‌کرد و بازی‌های قدیمی را در مرحله‌ای خاص خراب می‌کرد.

```python
import numpy as np
x = np.int8(127)
print(x + np.int8(1))   # ⚠️ -128  (پیچید!)
u = np.uint8(255)
print(u + np.uint8(1))  # ⚠️ 0      (بدونِ علامت، از اول شروع شد)
```

> نکته: زبانِ Python خودش اعدادِ صحیح را **بی‌نهایت‌بزرگ** نگه می‌دارد و سرریز نمی‌کند — `2**100` بدونِ مشکل کار می‌کند. ولی زیرِ پوست، C و سخت‌افزار همان کلیدهای محدود را دارند؛ به همین خاطر سرریز در زبان‌هایی مثل C/Java/Go واقعی است.

## صحنه‌ی ۲ — دفترِ الگوها (Character Encoding)

«دفترِ الگو» همان **encoding** است: قراردادی که می‌گوید کدام عدد، کدام حرف است.

- **ASCII** قدیمی‌ترین دفتر است: ۱۲۸ نشانه (حرفِ انگلیسی، عدد، علامت)، هرکدام در ۷ بیت. حرفِ `A` می‌شود عددِ ۶۵. مشکل: جای فارسی، چینی و شکلک نبود!
- **Unicode** یک **فهرستِ غول‌پیکر** است که به هر نشانه‌ی هر زبانِ دنیا یک شماره می‌دهد (به آن شماره می‌گویند **code point**، مثل `U+0633` برای «س»). اما Unicode فقط می‌گوید «این شماره مالِ این حرف است» — نمی‌گوید چطور آن را در بایت‌ها بنویسی.
- **UTF-8** روشِ نوشتنِ آن شماره‌ها در بایت است. حرف‌های انگلیسی ۱ بایت، فارسی و عربی ۲ بایت، و شکلک‌ها ۴ بایت می‌گیرند.

| اصطلاح | یعنی چه | مثال |
|---|---|---|
| Code point | شماره‌ی منحصربه‌فردِ یک حرف در Unicode | `س` = `U+0633` |
| Encoding (UTF-8) | چطور آن شماره را در بایت بنویسیم | `س` → `d8 b3` (۲ بایت) |

**چرا حرف‌ها به‌هم‌ریخته می‌شوند؟ (Mojibake)** دقیقاً همان صحنه‌ی کارخانه‌ی همسایه! یک طرف متن را با **UTF-8** می‌نویسد، طرفِ دیگر آن را با یک دفترِ قدیمیِ تک‌بایتی (مثل Latin-1) می‌خواند. نتیجه؟ هر بایت اشتباه ترجمه می‌شود:

```python
'ال'.encode('utf-8').decode('latin-1')   # → 'Ø§Ù„'  (به‌هم‌ریخته!)
```

اگر در یک API دیدی به جای «سلام» نوشته `Ø³Ù„Ø§Ù…`، یعنی همین اتفاق افتاده. درمان: همه‌جا — دیتابیس، فایل، هدرِ HTTP — را روی **UTF-8** ست کن.

**تله‌ی طولِ رشته با شکلک.** یک شکلکِ ساده مثل 👍 از نظرِ آدم «یک حرف» است، ولی ۴ بایت جا می‌گیرد. بدتر، شکلکِ خانواده 👨‍👩‍👧 در واقع **چند code point** است که با چسبِ نامرئی به هم وصل شده‌اند:

```python
len('👍')                      # 1  (یک code point)
len('👍'.encode('utf-8'))      # 4  (چهار بایت)
len('👨‍👩‍👧')                   # 5  (پنج code point، نه یک!)
len('👨‍👩‍👧'.encode('utf-8'))   # 18 بایت
```

به همین خاطر «شمارشِ حروفِ» یک متن سؤالِ ساده‌ای نیست: بایت بشماری؟ code point؟ یا چیزی که چشم می‌بیند؟

## صحنه‌ی ۳ — کلیدهای کسری (Floating Point)

«بعضی کسرها هیچ‌وقت دقیق جا نمی‌شوند» — این یعنی **اعداد اعشاری شناور (floating point)** طبقِ استانداردِ **IEEE 754**.

کامپیوتر اعشار را شبیه «نوشتنِ علمی» نگه می‌دارد: یک عددِ پایه × ۲ به یک توان (مثل `۱٫۵ × ۲³`). اما این همه در **مبنای ۲** انجام می‌شود، و خیلی از کسرهای ساده‌ی مبنای ۱۰ در مبنای ۲ **تمام‌نشدنی** می‌شوند — درست مثل `۱/۳ = ۰٫۳۳۳...` در مبنای ۱۰.

مقدارِ `0.1` در مبنای ۲ یک کسرِ تمام‌نشدنی است؛ کامپیوتر مجبور است یک‌جایی **گردش کند**. وقتی دو تا گرد‌شده را جمع می‌کنی، خطاها روی هم می‌آیند:

```python
0.1 + 0.2            # → 0.30000000000000004  (نه دقیقاً 0.3!)
0.1 + 0.2 == 0.3     # → False
```

این **باگ نیست** — طبیعتِ ذخیره‌ی محدود است.

**کِی به این کلیدها اعتماد نکنیم؟ پول!** اگر قیمت‌ها را با float جمع بزنی، یک فروشگاه ممکن است به جای ۲۰ تومان، عددِ `19.999999` نشان دهد. راه‌حل‌ها:

```python
from decimal import Decimal
Decimal('0.1') + Decimal('0.2')   # → 0.3  (دقیق!)
```

- یا همه‌چیز را **با کوچک‌ترین واحد به‌صورتِ عددِ صحیح** نگه دار: به جای «۲۰٫۵۰ تومان»، عددِ صحیحِ `2050` ریال. هیچ‌وقت سرریز و گردش ندارد.
- float برای علم، گرافیک و چیزهایی که خطای ذره‌ای مهم نیست عالی است؛ برای پول و شمارش، **هرگز**.

## صحنه‌ی ۴ — دست، میز، قفسه، انبار (Memory Hierarchy)

چهار جای نگه‌داریِ کارگر، چهار لایه‌ی حافظه در دنیای واقعی‌اند. قانونِ طلایی: **هرچه بزرگ‌تر، کندتر و ارزان‌تر.**

| در قصه | در دنیای واقعی | اندازه (تقریبی) | سرعت |
|---|---|---|---|
| توی دست ✋ | **رجیستر** (Register) | چند ده بایت | فوری‌ترین |
| میزِ کوچک 🪑 | **کش** L1/L2/L3 (Cache) | چند KB تا چند MB | خیلی سریع |
| قفسه‌های اتاق 🗄️ | **RAM** (حافظه‌ی اصلی) | چند GB | متوسط |
| انبارِ تهِ حیاط 🏚️ | **SSD / دیسک** | صدها GB تا چند TB | کند (ولی دائمی) |

**اعداد تأخیری که هر برنامه‌نویس باید حس‌شان کند** (تقریبی):

```
دسترسی به کش L1     ~۱ نانوثانیه      (تویِ دست)
دسترسی به RAM       ~۱۰۰ نانوثانیه    (۱۰۰ برابر کندتر!)
خواندن از SSD       ~۱۵٬۰۰۰ نانوثانیه (۱۵۰ برابرِ RAM)
خواندن از دیسکِ چرخان ~۱۰٬۰۰۰٬۰۰۰ نانوثانیه
```

اگر RAM را «یک ثانیه» فرض کنی، رفتن به دیسکِ چرخان مثلِ **چند ماه** انتظار است!

**محلیتِ کش (Cache Locality) — چرا حلقه‌ی سطری از ستونی تندتر است.** وقتی کارگر سراغِ یک چیز در قفسه می‌رود، همسایه‌هایش را هم با خودش به میز می‌آورد (چون احتمالاً بعدی‌اش لازم می‌شود). اگر داده‌ها را **پشتِ سرِ هم** بخوانی، هر بار از میز برمی‌داری (سریع). اگر **پراکنده** بخوانی، هر بار باید برگردی به قفسه (کند):

```python
# سطری: هر سطر پشت سر هم در حافظه است → کش‌پسند → سریع‌تر
for r in range(N):
    for c in range(N):
        s += M[r][c]

# ستونی: هر بار می‌پری به سطرِ دیگر → کش را هدر می‌دهی → کندتر
for c in range(N):
    for r in range(N):
        s += M[r][c]
```

## صحنه‌ی ۵ — سه مترجم (Compiler، Interpreter، JIT)

دستورِکارِ ما به زبانِ آدمی نوشته شده، ولی کارگر فقط زبانِ کلیدها (کدِ ماشین) را می‌فهمد. سه راهِ ترجمه:

| در قصه | در دنیای واقعی | مثال | معامله |
|---|---|---|---|
| مترجمِ اول: کلِ دفتر را *قبل* ترجمه می‌کند | **Compiler** | C، C++، Rust، Go | شروعِ کند، اجرای فوقِ سریع |
| مترجمِ دوم: *خط‌به‌خط* همان لحظه | **Interpreter** | Python، Ruby (نسخه‌ی ساده) | شروعِ فوری، اجرای کندتر |
| مترجمِ سوم: زمزمه می‌کند، ولی پرتکرارها را یک‌بار حسابی ترجمه می‌کند | **JIT** (Just-In-Time) | Java (JVM)، JavaScript (V8) | بهترین هر دو دنیا |

**بایت‌کد (Bytecode).** Python واقعاً «خالص» خط‌به‌خط نیست؛ اول کدت را به یک زبانِ میانیِ ساده‌تر به نامِ **bytecode** ترجمه می‌کند، بعد یک ماشینِ مجازی آن را اجرا می‌کند. Java هم همین‌طور (به `.class` کامپایل می‌شود و JVM اجرایش می‌کند). با `dis.dis` می‌توانی این بایت‌کد را با چشمِ خودت ببینی.

**رباتِ نظافتچی (Garbage Collection).** آن رباتِ قصه، **GC** است: خودکار حافظه‌ای را که دیگر هیچ متغیری به آن اشاره نمی‌کند آزاد می‌کند. دو سبک:

- **شمارشِ ارجاع (Reference Counting)** — هر شیء می‌شمارد چند نفر «در دستش دارند»؛ به صفر که رسید، فوری دور انداخته می‌شود. (Python از این استفاده می‌کند.) مشکل: حلقه‌ها (دو شیء که فقط به هم اشاره می‌کنند).
- **ردیابی (Tracing GC)** — هرازگاهی از ریشه‌ها راه می‌افتد، هر چیزِ قابل‌دسترس را علامت می‌زند، و بقیه را جارو می‌کند. (Java، Go.) مشکل: گاهی برای جمع‌وجور **همه‌چیز را یک لحظه نگه می‌دارد** (GC pause).

و راهِ سوم که اصلاً ربات ندارد: **مالکیتِ Rust (Ownership).** در Rust هر مقدار دقیقاً **یک مالک** دارد؛ به محضِ اینکه مالک از صحنه خارج شود، حافظه‌اش آزاد می‌شود — همه‌ی این تصمیم‌ها **زمانِ کامپایل** گرفته می‌شوند، پس نه رباتِ زمانِ اجرا لازم است، نه خطرِ فراموش کردنِ آزادسازیِ دستی (مثلِ C). نتیجه: سرعتِ C با امنیتِ حافظه.

## صحنه‌ی ۶ — چرخه‌ی بی‌پایانِ کارگر (CPU و چرخه‌ی دستور)

رازِ کارگر همان **چرخه‌ی Fetch–Decode–Execute** است:

```
۱. Fetch   (بردار)   → دستورِ بعدی را از حافظه بیاور
۲. Decode  (بفهم)    → یعنی چه؟ جمع است؟ مقایسه؟
۳. Execute (انجام بده) → کار را بکن
۴. برو سراغِ دستورِ بعدی  ↺
```

- **سرعتِ کلاک (Clock Speed)** یعنی کارگر هر ثانیه چند ضربه می‌زند. CPUی ۳ گیگاهرتزی ۳ میلیارد ضربه در ثانیه دارد!
- **هسته‌ها (Cores)** یعنی چند کارگرِ مستقل که هم‌زمان کار می‌کنند. ۸ هسته ≈ ۸ کارگر.
- **چرا CPU منتظرِ حافظه می‌ماند؟** همان صحنه‌ی انبار: اگر داده در کش نباشد و CPU مجبور شود سراغِ RAM برود، صدها ضربه‌ی کلاک **بیکار** می‌ماند تا داده برسد. به این می‌گویند **stall**. برای همین معماری‌های مدرن کلی ترفند دارند (pipeline، اجرای خارج از ترتیب، پیش‌بینیِ شاخه) تا کارگر کم‌تر بیکار بماند.

```
   │
   ▼

   │
   ▼

   ▼

   │   ↳ داده‌ها از RAM → کش → رجیستر بالا می‌آیند
   ▼
٥. نتیجه: 0.30000000000000004  (نه دقیقاً 0.3!)
   │
   ▼
٦. ربات GC: حافظه‌ی موقتی که دیگر لازم نیست را آزاد می‌کند 🧹
```
