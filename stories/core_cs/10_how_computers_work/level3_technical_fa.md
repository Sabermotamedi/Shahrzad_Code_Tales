## ۱. Binary و نمایشِ عدد

> در قصه: کلیدهای روشن/خاموش.

### بیت، بایت و واحدها

- **bit** = کوچک‌ترین واحد، `0` یا `1`. **byte** = ۸ بیت. **nibble** = ۴ بیت = یک رقمِ hex.
- `n` بیت می‌تواند `2ⁿ` حالتِ مختلف را نشان دهد. ۸ بیت → ۲۵۶ حالت؛ ۳۲ بیت → ~۴.۳ میلیارد؛ ۶۴ بیت → ~۱.۸×۱۰¹⁹.
- KB در ذخیره‌سازی معمولاً ۱۰۰۰ (SI)، در RAM معمولاً ۱۰۲۴ (KiB) است — منشأ اختلافِ «۵۰۰ گیگ خریدم، ۴۶۵ گیگ نشان می‌دهد».

### تبدیلِ مبناها

| ده‌دهی | باینری | هگز |
|---|---|---|
| 13 | `1101` | `d` |
| 255 | `1111 1111` | `ff` |
| 256 | `1 0000 0000` | `100` |

هر ۴ بیت = یک رقمِ hex. به همین خاطر رنگِ `#ff0000` = (R=255, G=0, B=0)، آدرسِ حافظه `0x7ffe...` و ماسکِ شبکه با hex راحت خوانده می‌شوند.

### اعدادِ منفی: متممِ دو (Two's Complement)

روشِ استانداردِ نمایشِ عددِ علامت‌دار. برای منفی کردنِ یک عدد: همه‌ی بیت‌ها را معکوس کن و یکی اضافه کن.

| باینری ۸ بیتی | unsigned | signed (two's complement) |
|---|---|---|
| `0000 0000` | 0 | 0 |
| `0111 1111` | 127 | 127 |
| `1000 0000` | 128 | **−128** |
| `1111 1111` | 255 | **−1** |

مزیت: یک مدارِ جمع‌کننده برای هم جمع و هم تفریق کافی است؛ فقط یک نمایشِ صفر داریم (برخلافِ sign-magnitude).

### سرریز (Integer Overflow)

عددی که از ظرفیتِ بیت‌ها بزرگ‌تر شود **می‌پیچد (wraps around)**:

```python
import numpy as np
np.int8(127)  + np.int8(1)   # → -128   (سرریزِ علامت‌دار)
np.uint8(255) + np.uint8(1)  # → 0       (سرریزِ بدونِ علامت)
```

> نکته‌ی مصاحبه: در C/Java/Go سرریزِ صحیح واقعی است و منبعِ باگ‌های امنیتی (مثلِ تخصیصِ بافرِ منفی). در Python اعدادِ صحیح **arbitrary-precision** هستند و سرریز نمی‌کنند — `2**1000` کار می‌کند — اما به قیمتِ کندی و حافظه‌ی بیشتر. این تفاوت خودش یک سؤالِ کلاسیک است.

## ۲. Character Encoding

> در قصه: دفترِ الگوها، و فاجعه‌ی دو کارخانه با دفترِ متفاوت.

### ASCII، Unicode، UTF-8

| مفهوم | چیست | نکته |
|---|---|---|
| **ASCII** | ۱۲۸ نشانه در ۷ بیت | فقط انگلیسی؛ `A`=65، `a`=97، `0`=48 |
| **Unicode** | فهرستِ **code point**ها (~۱.۱ میلیون جا) | فقط شماره‌گذاری، نه ذخیره؛ `U+0633` = «س» |
| **UTF-8** | encodingِ متغیرطول (۱ تا ۴ بایت) | سازگار با ASCII؛ پیش‌فرضِ وب |
| **UTF-16** | ۲ یا ۴ بایت | داخلِ Java/JavaScript/Windows؛ surrogate pairها |
| **Latin-1** | تک‌بایتیِ قدیمی | منشأ خیلی از mojibakeها |

نکته‌ی کلیدی: **code point ≠ byte ≠ گرافِم (آنچه چشم می‌بیند).** UTF-8 یک code point را به ۱–۴ بایت تبدیل می‌کند؛ یک گرافِمِ دیداری (مثلِ شکلکِ خانواده) می‌تواند چند code point باشد.

### UTF-8: طولِ متغیر

| بازه | بایت | مثال |
|---|---|---|
| `U+0000`–`U+007F` | ۱ بایت | ASCII (`A`) |
| `U+0080`–`U+07FF` | ۲ بایت | فارسی/عربی (`س` = `d8 b3`) |
| `U+0800`–`U+FFFF` | ۳ بایت | اکثرِ CJK |
| `U+10000`–`U+10FFFF` | ۴ بایت | شکلک‌ها (`👍`) |

### Mojibake: چرا و چطور

mojibake وقتی رخ می‌دهد که متن با یک encoding **نوشته** و با encoding دیگری **خوانده** شود. کلاسیک‌ترین حالت: متنِ UTF-8 که به‌عنوانِ Latin-1/CP1252 تفسیر شود.

```python
'ال'.encode('utf-8').decode('latin-1')   # → 'Ø§Ù„'
# دیدنِ "Ø§Ù„" یا "Ø³Ù„Ø§Ù…" = همین باگ
'Ø§Ù„'.encode('latin-1').decode('utf-8')   # → 'ال'  (بازیابی)
```

> نکته‌ی مصاحبه: نشانه‌ی `Ã©`, `Ø§`, `â€™` در متن = «UTF-8 خوانده‌شده به‌عنوانِ تک‌بایتی». درمانِ ریشه‌ای: زنجیره را سرتاسر UTF-8 کن — `Content-Type: ...; charset=utf-8`، کانکشن دیتابیس (`utf8mb4` در MySQL، نه `utf8`!)، خواندن/نوشتنِ فایل با `encoding='utf-8'`.

### تله‌های طولِ رشته

```python
len('👍'),              len('👍'.encode('utf-8'))         # (1, 4)
len('👨‍👩‍👧'),           len('👨‍👩‍👧'.encode('utf-8'))      # (5, 18)
```

> نکته‌ی مصاحبه: «طولِ رشته» مبهم است. در Python `len` تعدادِ code point است؛ در JavaScript `"👍".length` برابرِ ۲ است (UTF-16 code unit!)؛ محدودیتِ ۲۸۰ کاراکترِ توییتر بر اساسِ گرافِم/وزنِ خاص است. برای محدودیتِ پایگاه‌داده معمولاً بایت مهم است (`utf8mb4` در MySQL تا ۴ بایت per char).

## ۳. Floating Point (IEEE 754)

> در قصه: کلیدهای کسری که بعضی عددها هیچ‌وقت دقیق در آن‌ها جا نمی‌شوند.

### ساختار

یک عددِ `double` (۶۴ بیتی) سه بخش دارد: **sign** (۱ بیت) + **exponent** (۱۱ بیت) + **mantissa/significand** (۵۲ بیت). مقدار = `(-1)^sign × 1.mantissa × 2^(exponent-bias)`.

| نوع | بیت | دقتِ تقریبی |
|---|---|---|
| `float32` (single) | ۳۲ | ~۷ رقمِ ده‌دهی |
| `float64` (double) | ۶۴ | ~۱۵–۱۷ رقمِ ده‌دهی |

### چرا `0.1 + 0.2 != 0.3`

`0.1` در مبنای ۲ کسرِ متناوبِ تمام‌نشدنی است (مثلِ `1/3` در مبنای ۱۰)، پس نزدیک‌ترین `double` ذخیره می‌شود. جمعِ دو تقریب → تقریبِ سوم:

```python
0.1 + 0.2           # 0.30000000000000004
0.1 + 0.2 == 0.3    # False
format(0.1, '.17f') # '0.10000000000000001'
```

مقادیرِ خاص: `inf`, `-inf`, `nan`. نکته: `nan != nan` همیشه `True` است (هر مقایسه با NaN غلط است).

> نکته‌ی مصاحبه: هیچ‌وقت float را با `==` مقایسه نکن؛ از `math.isclose(a, b)` یا `abs(a-b) < eps` استفاده کن. و جمعِ float ترتیب‌حساس است: `(a+b)+c` می‌تواند با `a+(b+c)` فرق کند.

### کِی Decimal/Integer؟ (پول!)

```python
0.1 + 0.2                              # 0.30000000000000004  (float — غلط برای پول)
Decimal('0.1') + Decimal('0.2')        # Decimal('0.3')        (دقیق)
```

- **پول و حسابداری**: `Decimal` (مبنای ۱۰، دقیق) یا **عددِ صحیح بر حسبِ کوچک‌ترین واحد** (سِنت/ریال). الگوی رایج: همه‌چیز را در `cents` نگه دار، فقط موقعِ نمایش تقسیم بر ۱۰۰.
- **علم/گرافیک/ML**: float64 عالی است؛ خطای ذره‌ای مهم نیست و سرعت حیاتی است.
- **شمارش/شناسه**: همیشه integer.

## ۴. Memory Hierarchy

> در قصه: دست (رجیستر)، میز (کش)، قفسه (RAM)، انبار (دیسک).

### لایه‌ها و اعدادِ تأخیر

| لایه | اندازه‌ی تقریبی | تأخیر | معادلِ انسانی (اگر L1 = ۱ ثانیه) |
|---|---|---|---|
| Register | ده‌ها بایت | < ۱ns | لحظه |
| L1 cache | ~۳۲–۶۴ KB | ~۱ns | ۱ ثانیه |
| L2 cache | ~۲۵۶KB–۱MB | ~۴ns | ۴ ثانیه |
| L3 cache | ~۸–۳۲ MB | ~۱۰–۲۰ns | ~۱۵ ثانیه |
| RAM | ۸–۶۴ GB | ~۱۰۰ns | ~۱.۵ دقیقه |
| SSD (NVMe) | ۲۵۶GB–چند TB | ~۱۵–۱۰۰µs | چند روز |
| HDD (چرخان) | چند TB | ~۵–۱۰ms | چند ماه |
| شبکه (همان دیتاسنتر) | — | ~۰.۵ms | چند هفته |

اعدادِ معروفِ «Latency Numbers Every Programmer Should Know» (Jeff Dean) — ترتیب و مرتبه‌ی بزرگیْ مهم است، نه عددِ دقیق.

### Cache Line و Locality

- CPU حافظه را در بلوک‌های **cache line** (معمولاً ۶۴ بایت) می‌خواند، نه بایت‌به‌بایت.
- **Spatial locality**: داده‌های مجاور احتمالاً با هم استفاده می‌شوند → پیمایشِ پیوسته سریع است.
- **Temporal locality**: داده‌ای که تازه استفاده شده، احتمالاً دوباره استفاده می‌شود → کش نگهش می‌دارد.
- به همین خاطر در آرایه‌ی دوبعدیِ row-major (C, NumPy)، حلقه‌ی **سطری** (`for r: for c:`) بسیار سریع‌تر از **ستونی** است: پیمایشِ ستونی هر بار به cache line جدید می‌پرد (cache miss).

> نکته‌ی مصاحبه: «چرا `array[i][j]` با `j` در حلقه‌ی داخلی سریع‌تر است؟» چون حافظه پیوسته است و p.refetch نمی‌شود. ساختارِ داده‌ی cache-friendly (آرایه‌ی پیوسته) اغلب از ساختارِ «بهتر از نظرِ Big-O» (لیستِ پیوندی) در عمل سریع‌تر است.

## ۵. چطور کد اجرا می‌شود

> در قصه: سه مترجم و رباتِ نظافتچی.

### Compiler vs Interpreter vs JIT

| مدل | چه می‌کند | زبان‌ها | معامله |
|---|---|---|---|
| **AOT Compiler** | کلِ کد → machine code، قبل از اجرا | C, C++, Rust, Go | اجرای فوقِ سریع؛ build لازم؛ وابسته به پلتفرم |
| **Interpreter** | خط‌به‌خط در زمانِ اجرا | Python, Ruby, PHP | شروعِ فوری، انعطاف؛ کندتر |
| **Bytecode VM** | کد → bytecode → VM اجرا می‌کند | Python (CPython), Java | پرتابل؛ سرعتِ متوسط |
| **JIT** | bytecode داغ → machine code در حینِ اجرا | JVM (HotSpot), V8, PyPy | شروعِ سریع + اجرای سریع پس از warm-up |

- **C**: مستقیم به machine code کامپایل می‌شود؛ سریع‌ترین، ولی بدونِ شبکه‌ی ایمنیِ زمانِ اجرا.
- **Python (CPython)**: کد → `.pyc` (bytecode) → اجرا توسطِ interpreterِ VM. کند ولی منعطف.
- **Java/JS**: bytecode → JIT بخش‌های «داغ» (پرتکرار) را به machine code بهینه ترجمه می‌کند و تا حدِ نزدیک به C می‌رساند.

### Bytecode (Python)

```python
import dis
def add(a, b):
    return a + b
dis.dis(add)
# LOAD_FAST  a
# LOAD_FAST  b
# BINARY_OP  + (یا BINARY_ADD در نسخه‌های قدیمی‌تر)
# RETURN_VALUE
```

(opcodeها بین نسخه‌های Python فرق می‌کنند؛ مفهوم ثابت است: کد به دستورهای ساده‌ی stack-based تبدیل می‌شود.)

### مدیریتِ حافظه

| روش | چطور | مثال | معامله |
|---|---|---|---|
| **Manual** | برنامه‌نویس `malloc`/`free` | C, C++ | کنترلِ کامل؛ خطرِ نشتی، use-after-free، double-free |
| **Reference Counting** | شمارشِ ارجاع، صفر شد → آزاد | CPython | فوری و قابل‌پیش‌بینی؛ مشکلِ حلقه‌های ارجاعی |
| **Tracing GC** | mark & sweep از ریشه‌ها | Java, Go, JS | راحت؛ **GC pause** و سربارِ حافظه |
| **Ownership** | زمانِ کامپایل، هر مقدار یک مالک | Rust | بدونِ GC و بدونِ نشتی؛ منحنیِ یادگیریِ تند |

- **Reference counting** (Python): هر شیء شمارنده دارد؛ به ۰ که رسید آزاد می‌شود. حلقه‌ها (`a.b = b; b.a = a`) با یک GC ردیابِ کمکی جمع می‌شوند.
- **Tracing GC**: گاهی برنامه را برای جمع‌آوری متوقف می‌کند (**stop-the-world pause**) — دردِ سیستم‌های لحظه‌ای؛ GCهای مدرن (G1, ZGC) pause را به زیرِ میلی‌ثانیه می‌رسانند.
- **Rust ownership** (یک پاراگراف): هر مقدار دقیقاً یک مالک دارد؛ با خروجِ مالک از scope، حافظه آزاد می‌شود (`Drop`). قواعدِ **borrow checker** در زمانِ کامپایل تضمین می‌کنند که هیچ‌وقت دو نوشتنِ هم‌زمان یا ارجاعِ معلق وجود ندارد — پس نه GC لازم است، نه آزادسازیِ دستی؛ امنیتِ حافظه **بدونِ** هزینه‌ی زمانِ اجرا.

> نکته‌ی مصاحبه: «refcounting در برابر tracing؟» refcounting فوری و قابل‌پیش‌بینی است ولی حلقه‌ها را نمی‌گیرد و هر انتساب سربار دارد؛ tracing حلقه‌ها را می‌گیرد ولی pause و throughput-tradeoff دارد. CPython از **هر دو** استفاده می‌کند.

## ۶. مبانیِ CPU

> در قصه: چرخه‌ی بی‌پایانِ کارگر.

### چرخه‌ی Fetch–Decode–Execute

```
Fetch   → دستورِ بعدی از حافظه (به آدرسِ Program Counter)
Decode  → رمزگشاییِ opcode و عملوندها
Execute → ALU/حافظه/پرش
Write-back → نتیجه در رجیستر یا حافظه
↺ (PC به بعدی)
```

### مفاهیمِ کلیدی

- **Clock speed**: تعدادِ سیکل در ثانیه (۳GHz = ۳ میلیارد). ولی «دستور در سیکل» (IPC) هم مهم است؛ کلاکِ بالاتر همیشه = سریع‌تر نیست.
- **Cores**: واحدهای پردازشِ مستقل برای موازی‌کاریِ واقعی. **Hyperthreading/SMT**: هر هسته دو رشته را با اشتراکِ منابع اجرا می‌کند.
- **Pipeline**: مراحلِ چرخه روی دستورهای مختلف هم‌پوشانی می‌کنند (مثلِ خطِ مونتاژ) تا throughput بالا برود.
- **Branch prediction**: CPU حدس می‌زند `if` به کدام سمت می‌رود تا pipeline خالی نماند؛ حدسِ غلط = جریمه (و منشأ Spectre).
- **چرا CPU منتظرِ حافظه می‌ماند؟** اگر داده در کش نباشد (**cache miss**)، CPU باید صدها سیکل برای RAM **stall** کند. شکافِ «memory wall» (سرعتِ CPU >> سرعتِ RAM) دلیلِ وجودِ سلسله‌مراتبِ کش و این‌همه ترفندِ مخفی‌کردنِ تأخیر است.

> نکته‌ی مصاحبه: یک cache miss به RAM ~۱۰۰ns ≈ ~۳۰۰ سیکلِ CPU است. به همین خاطر «الگوریتمِ کندتر ولی cache-friendly» اغلب در عمل از «الگوریتمِ سریع‌تر ولی پراکنده در حافظه» برنده می‌شود.

```python
# ۱) باینری و هگز
print(bin(13), hex(255))          # 0b1101 0xff
print(int('1101', 2), int('ff', 16))  # 13 255

# ۲) Floating point و راه‌حلِ پول
print(0.1 + 0.2)                  # 0.30000000000000004
print(0.1 + 0.2 == 0.3)           # False
from decimal import Decimal

# 4 b'\xd8\xb3\xd9\x84\xd8\xa7\xd9\x85' 8
print('👍 →', len('👍'), 'code point,', len('👍'.encode('utf-8')), 'bytes')
# mojibake را بساز و بازیابی کن
bad = 'ال'.encode('utf-8').decode('latin-1')
print('mojibake:', bad, '| fixed:', bad.encode('latin-1').decode('utf-8'))

# ۴) سرریزِ صحیح (Python سرریز نمی‌کند، NumPy می‌کند)
import numpy as np
print(np.int8(127) + np.int8(1))   # -128
print(2**100)                       # Python: بدونِ سرریز

# ۵) sys.getsizeof — اندازه‌ی واقعیِ اشیاء در حافظه
import sys
print(sys.getsizeof(0), sys.getsizeof(2**100), sys.getsizeof('a'), sys.getsizeof([]))
# 28 40 42 56  (هر شیءِ Python سربارِ هدر دارد!)

# ۶) Cache locality: مجموعِ سطری در برابر ستونی
import time
N = 2000
M = [[1]*N for _ in range(N)]
t = time.perf_counter()
s = 0
for r in range(N):
    row = M[r]
    for c in range(N): s += row[c]
print('row-wise :', round(time.perf_counter()-t, 3), 's')
t = time.perf_counter()
s = 0
for c in range(N):
    for r in range(N): s += M[r][c]
print('col-wise :', round(time.perf_counter()-t, 3), 's')
# در لیستِ-لیست خالصِ Python اختلاف کم است (~۱.۲ برابر) چون pointer indirection دارد.
# با NumPy روی آرایه‌ی پیوسته، اختلاف چشمگیر می‌شود:
A = np.ones((N, N), dtype=np.int64)
t = time.perf_counter(); A.sum(axis=1); print('numpy rows:', round((time.perf_counter()-t)*1000,2),'ms')
t = time.perf_counter(); A.sum(axis=0); print('numpy cols:', round((time.perf_counter()-t)*1000,2),'ms')
# axis=1 (سطری، پیوسته) چند برابر سریع‌تر از axis=0 (ستونی)

# ۷) Bytecode را ببین
import dis
def add(a, b): return a + b
dis.dis(add)   # LOAD_FAST ... BINARY_OP + ... RETURN_VALUE
```

## 🎯 سوالات مصاحبه

> روی هر سوال کلیک کن تا جواب را ببینی. اول خودت جواب بده، بعد چک کن!

### Binary و عددها

<details><summary>**۱. چرا کامپیوترها مبنای ۲ را به‌جای مبنای ۱۰ استفاده می‌کنند؟**</summary>

سخت‌افزارِ دو-حالته (ولتاژ هست/نیست) ساده، ارزان و مقاوم در برابر نویز است؛ تشخیصِ ده سطحِ ولتاژِ متمایز بسیار خطاپذیرتر و گران‌تر بود. منطقِ بولی هم مستقیماً روی دو حالت کار می‌کند.

</details>

<details><summary>**۲. متممِ دو (two's complement) چیست و چه مزیتی بر sign-magnitude دارد؟**</summary>

روشِ نمایشِ عددِ علامت‌دار که در آن منفیِ یک عدد = معکوسِ بیت‌ها + ۱. مزیت‌ها: یک نمایشِ صفر (نه `+0` و `-0`)، و جمع/تفریق با همان مدارِ جمع‌کننده بدونِ منطقِ ویژه. `-1` می‌شود تمام‌بیت‌ها یک.

</details>

<details><summary>**۳. در یک بازی، امتیازِ بازیکن ناگهان منفیِ بزرگ شد. چه اتفاقی افتاد؟**</summary>

سرریزِ صحیح (integer overflow): امتیاز از حداکثرِ نوع (مثلاً `2^31-1` برای int32) گذشت و **پیچید** به منفی. درمان: نوعِ بزرگ‌تر (int64)، saturating arithmetic، یا بررسیِ overflow. نمونه‌ی تاریخی: «kill screen»های Pac-Man/Donkey Kong.

</details>

<details><summary>**۴. چرا در C سرریز خطرناک است ولی در Python نه؟**</summary>

اعدادِ صحیحِ C ظرفیتِ ثابتِ بیت دارند و سرریزشان (به‌خصوص signed) رفتارِ تعریف‌نشده و منشأ باگ‌های امنیتی است. اعدادِ صحیحِ Python **arbitrary-precision** هستند و خودکار رشد می‌کنند؛ سرریز ندارند، اما کندتر و سنگین‌ترند. (برای سرریز در Python باید NumPy/ctypes با عرضِ ثابت استفاده کنی.)

</details>

### Character Encoding

<details><summary>**۵. فرقِ Unicode و UTF-8 چیست؟**</summary>

Unicode یک **استانداردِ شماره‌گذاری** است: به هر نشانه یک code point می‌دهد (`U+0633`). UTF-8 یک **encoding** است: می‌گوید آن code point چطور در بایت‌ها ذخیره شود (۱ تا ۴ بایت). Unicode «چه شماره‌ای»، UTF-8 «چطور در بایت».

</details>

<details><summary>**۶. سناریو: متنی که از یک API می‌گیری به‌جای فارسی `Ø§Ù„` نشان می‌دهد. چه شد و چطور حلش می‌کنی؟**</summary>

متن در UTF-8 کدگذاری شده ولی سمتِ تو به‌عنوانِ Latin-1/CP1252 (تک‌بایتی) decode شده — هر بایتِ UTF-8 جدا به یک کاراکترِ غلط نگاشت شد (mojibake). درمانِ موقت: `bytes.encode('latin-1').decode('utf-8')`. درمانِ ریشه‌ای: charset را در کلِ زنجیره UTF-8 کن — هدرِ `Content-Type`، تنظیماتِ کلاینت HTTP، و کلیدِ مهم: کانکشن و ستونِ دیتابیس (`utf8mb4`).

</details>

<details><summary>**۷. چرا `len('👨‍👩‍👧')` در Python برابرِ ۱ نیست؟**</summary>

این شکلک یک گرافِمِ دیداری است که از **چند code point** ساخته شده (سه چهره + کاراکترهای نامرئیِ ZWJ که آن‌ها را به هم می‌چسبانند). `len` تعدادِ code point را می‌شمارد (۵)، نه آنچه چشم می‌بیند (۱). برای شمارشِ گرافِم به کتابخانه‌ی grapheme-aware نیاز است.

</details>

<details><summary>**۸. در JavaScript `"👍".length` برابرِ ۲ است، چرا؟**</summary>

JavaScript رشته‌ها را در UTF-16 نگه می‌دارد و `.length` تعدادِ **UTF-16 code unit** را برمی‌گرداند. کاراکترهای بالای `U+FFFF` (مثلِ شکلک) با **surrogate pair** (دو code unit) نمایش داده می‌شوند، پس طولشان ۲ است. برای code pointها از `[...str].length` استفاده کن.

</details>

<details><summary>**۹. چرا در MySQL باید `utf8mb4` استفاده کرد نه `utf8`؟**</summary>

«`utf8`»ی MySQL فقط تا ۳ بایت per char را پشتیبانی می‌کند و شکلک‌ها و بعضی کاراکترهای ۴ بایتی را نمی‌تواند ذخیره کند (خطا یا برش). `utf8mb4` UTF-8 واقعی تا ۴ بایت است. همیشه `utf8mb4` استفاده کن.

</details>

### Floating Point

<details><summary>**۱۰. چرا `0.1 + 0.2 == 0.3` در اکثرِ زبان‌ها `False` است؟**</summary>

`0.1` و `0.2` در باینری کسرهای متناوبِ تمام‌نشدنی‌اند و به نزدیک‌ترین `double` گرد می‌شوند؛ جمعِ این دو تقریب یک عددِ کمی متفاوت از نزدیک‌ترین تقریبِ `0.3` می‌دهد (`0.30000000000000004`). این طبیعتِ IEEE 754 است، نه باگِ زبان.

</details>

<details><summary>**۱۱. سناریو: اپِ مالی به‌جای ۲۰ تومان `19.999999` نشان می‌دهد. چرا و چطور رفعش می‌کنی؟**</summary>

محاسباتِ پولی با float انجام شده و خطاهای گردِش انباشته شده‌اند. رفع: یا `Decimal` (مبنای ۱۰، دقیق) استفاده کن، یا — رایج‌تر — همه‌ی مبالغ را به‌صورتِ **عددِ صحیح بر حسبِ کوچک‌ترین واحد** (سِنت/ریال) نگه دار و فقط موقعِ نمایش تقسیم/فرمت کن. هیچ‌وقت پول را float نکن.

</details>

<details><summary>**۱۲. چرا نباید floatها را با `==` مقایسه کرد؟ جایگزین؟**</summary>

به‌خاطرِ خطای گردِش، دو محاسبه‌ی منطقاً مساوی ممکن است بیت‌به‌بیت یکی نباشند. جایگزین: `math.isclose(a, b, rel_tol=..., abs_tol=...)` یا `abs(a-b) < eps`. نکته: `float('nan') == float('nan')` همیشه `False` است.

</details>

<details><summary>**۱۳. کِی float خوب است و کِی نه؟**</summary>

خوب: علم، گرافیک، ML، سیگنال — جایی که محدوده‌ی دینامیکیِ وسیع و سرعت مهم است و خطای ذره‌ای بی‌اهمیت. بد: پول، شمارش، شناسه، و هر جایی که برابریِ دقیق یا انباشتِ بدونِ خطا لازم است.

</details>

### Memory Hierarchy

<details><summary>**۱۴. سلسله‌مراتبِ حافظه را از سریع به کند بگو و چرا این‌طور چیده شده.**</summary>

Register → L1 → L2 → L3 cache → RAM → SSD → HDD → شبکه. هرچه پایین‌تر: بزرگ‌تر، ارزان‌تر per byte، و کندتر. چینش هرمی این امکان را می‌دهد که با هزینه‌ی منطقی، **به‌طورِ متوسط** سرعتِ بالا داشته باشیم: داده‌ی داغ در کشِ کوچکِ سریع، داده‌ی سرد در ذخیره‌سازیِ بزرگِ کند.

</details>

<details><summary>**۱۵. مرتبه‌ی تأخیرِ L1 در برابر RAM در برابر SSD چقدر است؟**</summary>

L1 ~۱ns، RAM ~۱۰۰ns (≈۱۰۰ برابر)، SSD ~۱۰–۱۰۰µs (≈۱۰۰۰ برابرِ RAM)، HDD ~۱۰ms. عدد دقیق مهم نیست، **مرتبه‌ی بزرگی** مهم است: هر پله‌ی پایین‌تر تقریباً دو مرتبه کندتر است.

</details>

<details><summary>**۱۶. چرا پیمایشِ سطریِ یک ماتریس از ستونی سریع‌تر است؟**</summary>

در حافظه‌ی row-major (C/NumPy)، عناصرِ یک سطر **پیوسته** ذخیره می‌شوند. CPU هر cache line (~۶۴ بایت) را یک‌جا می‌آورد؛ پیمایشِ سطری از این پیش‌خوانده‌ها استفاده می‌کند (cache hit)، ولی پیمایشِ ستونی هر بار به آدرسِ دور می‌پرد و cache miss می‌دهد. این **spatial locality** است.

</details>

<details><summary>**۱۷. چرا گاهی یک الگوریتمِ O(n) از O(n) دیگری در عمل کندتر است؟**</summary>

Big-O ثابت‌ها و رفتارِ حافظه را نادیده می‌گیرد. الگوریتمی که حافظه را پراکنده می‌خواند (مثلِ پیمایشِ linked list) cache miss زیاد می‌دهد و در عمل می‌تواند از یک آرایه‌ی پیوسته‌ی cache-friendly کندتر باشد، حتی با Big-O یکسان. در سخت‌افزارِ واقعی، locality اغلب از پیچیدگیِ مجانبی مهم‌تر است.

</details>

### چطور کد اجرا می‌شود

<details><summary>**۱۸. فرقِ Compiler و Interpreter چیست؟ JIT کجای این طیف است؟**</summary>

Compiler کلِ کد را **قبل از اجرا** به machine code ترجمه می‌کند (سریع در اجرا، C/Rust). Interpreter آن را **در حینِ اجرا خط‌به‌خط** اجرا می‌کند (شروعِ فوری، انعطاف، کندتر — Python). JIT ترکیب است: اول bytecode را اجرا می‌کند، سپس بخش‌های «داغ/پرتکرار» را **در زمانِ اجرا** به machine code بهینه کامپایل می‌کند (JVM, V8, PyPy) — شروعِ سریع و اجرای سریع پس از warm-up.

</details>

<details><summary>**۱۹. bytecode چیست و چرا وجود دارد؟**</summary>

یک نمایشِ میانیِ سطحِ‌پایین‌تر از سورس و سطحِ‌بالاتر از machine code (مثلِ `.pyc` در Python یا `.class` در Java). مزیت: **پرتابل بودن** (یک bytecode روی هر پلتفرمی که VM دارد اجرا می‌شود) و سادگیِ اجرا برای VM. Java شعارش «write once, run anywhere» از همین است.

</details>

<details><summary>**۲۰. تفاوتِ مدیریتِ حافظه‌ی دستی، reference counting و tracing GC؟**</summary>

دستی (C): خودت `malloc`/`free`؛ کنترلِ کامل ولی خطرِ نشتی/use-after-free. Reference counting (CPython): هر شیء شمارنده دارد، صفر شد آزاد می‌شود — فوری و قابل‌پیش‌بینی ولی حلقه‌های ارجاعی را نمی‌گیرد. Tracing GC (Java/Go): دوره‌ای از ریشه‌ها mark & sweep می‌کند — حلقه‌ها را می‌گیرد ولی pause و سربار دارد.

</details>

<details><summary>**۲۱. GC pause چیست و چرا برای سیستم‌های لحظه‌ای دردسر است؟**</summary>

برخی tracing GCها برای جمع‌آوریِ امن، اجرای برنامه را لحظه‌ای **متوقف** می‌کنند (stop-the-world). در یک سرویسِ کم‌تأخیر (تریدینگ، بازی) همین مکثِ چندده‌میلی‌ثانیه‌ای فاجعه است. راه‌حل: GCهای concurrent/low-pause (G1، ZGC، Shenandoah)، کاهشِ allocation، یا زبانِ بدونِ GC مثلِ Rust.

</details>

<details><summary>**۲۲. Rust چطور بدونِ garbage collector از نشتیِ حافظه جلوگیری می‌کند؟**</summary>

با مدلِ **ownership** که در **زمانِ کامپایل** بررسی می‌شود: هر مقدار یک مالک دارد، با خروجِ مالک از scope حافظه‌اش آزاد می‌شود (`Drop`)، و قواعدِ **borrow checker** اجازه نمی‌دهند ارجاعِ معلق یا data race ساخته شود. پس نه GC زمانِ اجرا لازم است، نه `free` دستی — امنیتِ حافظه با هزینه‌ی صفرِ زمانِ اجرا.

</details>

<details><summary>**۲۳. چرا Python از زبانِ C کندتر است؟**</summary>

Python interpreted/bytecode-VM و **dynamically typed** است: هر عملیات باید نوع را در زمانِ اجرا چک کند، اشیاء boxed (با هدر و refcount) هستند، و GIL موازی‌سازیِ thread-based را محدود می‌کند. C مستقیم به machine code کامپایل می‌شود با typeهای استاتیک و دسترسیِ مستقیم به حافظه. (راه‌حل‌های Python: NumPy/C-extension، PyPy/JIT، یا انتقالِ hot path به C/Rust.)

</details>

### CPU

<details><summary>**۲۴. چرخه‌ی Fetch-Decode-Execute را شرح بده.**</summary>

CPU در حلقه: (۱) **Fetch** دستورِ بعدی را از حافظه به آدرسِ Program Counter می‌آورد؛ (۲) **Decode** opcode و عملوندها را رمزگشایی می‌کند؛ (۳) **Execute** عملیات را در ALU/حافظه انجام می‌دهد و نتیجه را write-back می‌کند؛ سپس PC به دستورِ بعدی می‌رود. CPUهای مدرن این مراحل را با pipeline روی دستورهای متعدد هم‌پوشانی می‌کنند.

</details>

<details><summary>**۲۵. آیا CPUی ۴GHz همیشه از CPUی ۳GHz سریع‌تر است؟**</summary>

نه لزوماً. کارایی = clock × IPC (دستور در سیکل) × تعدادِ هسته، به‌علاوه‌ی اندازه‌ی کش و معماری. یک CPU با کلاکِ پایین‌تر ولی IPC و کشِ بهتر می‌تواند سریع‌تر باشد. کلاکِ بالاتر گرما و مصرفِ توان هم بیشتر می‌برد.

</details>

<details><summary>**۲۶. چرا CPU «منتظرِ حافظه» می‌ماند و این یعنی چه برای کد؟**</summary>

شکافِ سرعتِ CPU و RAM («memory wall») یعنی یک cache miss به RAM ~۱۰۰ns ≈ صدها سیکلِ بیکار (**stall**) است. یعنی کدِ تو اغلب نه به‌خاطرِ محاسبات، بلکه به‌خاطرِ **انتظار برای داده** کند است. درس: ساختارهای داده‌ی cache-friendly و الگوهای دسترسیِ پیوسته اغلب بزرگ‌ترین بُردِ کارایی‌اند.

</details>

<details><summary>**۲۷. چند هسته یا hyperthreading چطور به موازی‌سازی کمک می‌کنند و محدودیتشان چیست؟**</summary>

چند هسته = اجرای واقعاً هم‌زمانِ چند رشته. Hyperthreading/SMT = هر هسته دو رشته را با اشتراکِ واحدهای اجرایی پیش می‌برد تا حفره‌های stall پر شوند (سودِ کم‌تر از هسته‌ی واقعی). محدودیت‌ها: قانونِ Amdahl (بخشِ سریال سقفِ شتاب را می‌گذارد)، هزینه‌ی همگام‌سازی/قفل، و در Python خودِ **GIL** که موازیِ CPU-bound را با threadها محدود می‌کند (راه‌حل: multiprocessing یا C-extension).

</details>
