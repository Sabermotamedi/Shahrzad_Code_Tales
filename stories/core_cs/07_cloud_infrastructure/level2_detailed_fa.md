# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی بندر 🔍🚢

> همان قصه‌ی سام در بندر، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعی *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — جعبه‌ی استاندارد غذا (Docker / Container)

«جعبه‌ی استاندارد» در دنیای واقعی همان **Container** است، و فناوری‌اش معروف‌ترین اسمش **Docker** است.

ایده‌ی بزرگ: برنامه‌ات روی لپ‌تاپ تو کار می‌کند، ولی روی سرور خراب می‌شود — چون نسخه‌ی Python فرق دارد، یا یک کتابخانه نصب نیست. شکایت همیشگی: «ولی روی دستگاه من که کار می‌کرد!» (works on my machine). Container این بهانه را برای همیشه می‌کشد: همه چیزِ لازم — کد، کتابخانه‌ها، تنظیمات — را در **یک جعبه** می‌گذاری که هر جا یک‌جور باز می‌شود.

چند کلمه‌ی کلیدی که نباید قاتی شوند:

| در قصه | اسم فنی | یعنی چه |
|---|---|---|
| **دستور پختِ نوشته‌شده** | `Dockerfile` | متنِ دستورالعملِ ساختِ جعبه |
| **جعبه‌ی بسته‌بندی‌شده روی قفسه** | `Image` (ایمیج) | قالبِ آماده و ثابت، هنوز روشن نشده |
| **همان جعبه که بازش کرده‌ای و غذا گرم است** | `Container` (کانتینر) | نمونه‌ی **در حالِ اجرا**ی یک ایمیج |
| **انبارِ مرکزیِ جعبه‌ها** | `Registry` | جایی که ایمیج‌ها را آپلود/دانلود می‌کنی (مثل Docker Hub) |

پس رابطه ساده است: از یک **Dockerfile** یک **Image** می‌سازی؛ هر بار که Image را اجرا می‌کنی یک **Container** زنده می‌شود — مثل کلاس و آبجکت. Image را در **Registry** می‌گذاری تا بقیه هم بردارند.

یک Dockerfile کوچک واقعی این شکلی است:

```dockerfile
FROM python:3.12-slim        # از این جعبه‌ی پایه شروع کن
WORKDIR /app                 # داخل جعبه، این پوشه را خانه کن
COPY requirements.txt .      # لیست خرید را بریز تو
RUN pip install -r requirements.txt   # خریدها را نصب کن
COPY . .                     # بقیه‌ی کد را بریز تو
CMD ["python", "app.py"]     # وقتی جعبه را باز کردند، این را اجرا کن
```

### چرا جعبه سریع و سبک است؟ (Layer و فرق با VM)

هر خط Dockerfile یک **لایه (Layer)** می‌سازد. لایه‌ها روی هم سوار می‌شوند مثل کیک، و Docker آن‌ها را **کش** می‌کند: اگر فقط آخرین خط را عوض کنی، لایه‌های قبلی دوباره ساخته نمی‌شوند — بازسازی برق‌آسا می‌شود. به همین خاطر `requirements` را **قبل** از کد کپی می‌کنیم: تا وقتی لیست خرید عوض نشده، لایه‌ی نصب از کش می‌آید.

حالا فرق Container با **ماشین مجازی (Virtual Machine / VM)**:

```
ماشین مجازی (سنگین):              کانتینر (سبک):
┌─────────┐ ┌─────────┐          ┌─────────┐ ┌─────────┐
│  App A  │ │  App B  │          │  App A  │ │  App B  │
├─────────┤ ├─────────┤          ├─────────┴─┴─────────┤
│ کل یک   │ │ کل یک   │          │   Docker Engine     │
│ سیستم‌  │ │ سیستم‌  │          ├─────────────────────┤
│ عامل    │ │ عامل    │          │  یک سیستم‌عامل      │ ← مشترک!
├─────────┤ ├─────────┤          ├─────────────────────┤
│  سخت‌افزار مجازی    │          │     سخت‌افزار        │
```

- **VM** کلِ یک سیستم‌عامل را شبیه‌سازی می‌کند → سنگین (گیگابایت‌ها)، کند برای روشن شدن (دقیقه).
- **Container** سیستم‌عاملِ میزبان را **به اشتراک می‌گذارد** و فقط برنامه‌ی خودش را جدا می‌کند → سبک (مگابایت‌ها)، روشن شدن در میلی‌ثانیه.

این جداسازیِ سبک با دو ابزارِ خودِ لینوکس انجام می‌شود (همان دیوارهای نامرئیِ بین جعبه‌ها):

- **Namespaces** — هر کانتینر فکر می‌کند **تنها برنامه‌ی روی دستگاه** است: لیست پروسه‌های خودش، شبکه‌ی خودش، فایل‌سیستم خودش را می‌بیند و بقیه را نه.
- **cgroups** (control groups) — سهمیه‌بندی: «این جعبه فقط حق دارد این‌قدر CPU و این‌قدر حافظه مصرف کند» تا یک جعبه‌ی پرخور بقیه را گرسنه نگذارد.

## صحنه‌ی ۲ — آقای بندر کیست؟ (Kubernetes / Orchestration)

«آقای بندر» در دنیای واقعی **Kubernetes** است (که کوتاه می‌نویسند **k8s** — چون بین K و s هشت حرف است). وقتی به جای ۱ کانتینر، **هزار** کانتینر روی **ده‌ها سرور** داری، یک نفر باید همه را بچیند، سالم نگه دارد و کم و زیاد کند. این کارِ **Orchestration** (ارکستراسیون) است، و k8s رهبرِ این ارکستر.

واژه‌نامه‌ی بندر به k8s:

| در قصه | اسم فنی | یعنی چه |
|---|---|---|
| **یک کشتیِ بزرگ** | `Node` | یک سرور (ماشین) که کانتینرها رویش اجرا می‌شوند |
| **کوچک‌ترین جعبه‌ی قابل‌جابه‌جایی** | `Pod` | کوچک‌ترین واحد k8s؛ یک (یا چند) کانتینرِ به‌هم‌چسبیده |
| **سفارشِ «همیشه ۵ تا از این جعبه باشد»** | `Deployment` | تعریف می‌کند چه ایمیجی و **چند نسخه** باید زنده باشد |
| **آدرسِ ثابتِ دمِ در برای رسیدن به جعبه‌ها** | `Service` | یک نقطه‌ی ورودیِ پایدار که ترافیک را بین Podها پخش می‌کند |

چرا Service لازم است؟ چون Podها **فانی** هستند: می‌میرند و با IP جدید دوباره متولد می‌شوند. اگر کسی بخواهد مستقیم با IP یک Pod حرف بزند، فردا آدرس عوض شده. **Service** یک نام و IP **ثابت** می‌دهد و خودش ترافیک را به Podهای زنده‌ی پشتش پخش می‌کند (شبیه Load Balancer از فصل شبکه).

### دو کارِ جادوییِ آقای بندر

این دو رفتار، علتِ وجودی k8s است:

1. **خوددرمانی (Self-healing)** — تو می‌گویی «همیشه ۵ Pod زنده می‌خواهم» (این حالتِ مطلوب = **desired state**). اگر یکی بمیرد، k8s مدام واقعیت را با خواسته‌ات مقایسه می‌کند (**reconciliation loop**) و فوراً یکی تازه می‌سازد تا دوباره ۵ تا شود. جعبه‌ی خراب → جعبه‌ی نو، خودکار.
2. **مقیاس‌پذیری (Scaling)** — اگر شلوغ شد، عدد ۵ را می‌کنی ۲۰، و k8s ۱۵ Pod تازه می‌سازد. می‌تواند **خودکار** هم باشد (با دیدن مصرف CPU، خودش کم و زیاد کند — همان «اگر پرطرفدار شد، ده جعبه بیشتر بچین»).

نکته‌ی مهمِ ذهنی: تو به k8s **دستورِ کار به کار نمی‌دهی** («این جعبه را بردار، آن‌جا بگذار»). تو فقط **خواسته‌ات را اعلام می‌کنی** و k8s راهش را پیدا می‌کند. این همان جمله‌ی آقای بندر بود: «من فقط می‌گویم چه می‌خواهم.»

سفارشِ «همیشه ۳ تا از این جعبه» این شکلی نوشته می‌شود:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cat-api
spec:
  replicas: 3                  # همیشه ۳ نسخه زنده باشد ← self-healing + scale
  selector:
    matchLabels: { app: cat-api }
  template:
    metadata:
      labels: { app: cat-api }
    spec:
      containers:
        - name: cat-api
          image: myregistry/cat-api:1.4   # همان ایمیج از Registry
          ports:
            - containerPort: 8080
```

و با ابزارِ خط فرمانِ k8s به اسم **kubectl** با این بندر حرف می‌زنی:

```bash
kubectl apply -f deployment.yaml   # این سفارش را اجرا کن
kubectl get pods                   # چند جعبه زنده است؟
kubectl scale deployment cat-api --replicas=10   # شلوغ شد، ۱۰ تا کن
kubectl logs <pod-name>            # یک جعبه چه می‌گوید؟
```

## صحنه‌ی ۳ — نوار نقاله‌ی جادویی (CI/CD)

«نوار نقاله» در دنیای واقعی **CI/CD** است:

- **CI** = Continuous Integration (یکپارچه‌سازیِ مداوم): هر بار کسی کدِ تازه‌ای می‌فرستد، خودکار **ساخته** و **آزمایش** می‌شود تا زود بفهمیم چیزی نشکسته.
- **CD** = Continuous Delivery/Deployment (تحویل/استقرارِ مداوم): اگر آزمایش‌ها سبز شد، خودکار **منتشر/مستقر** می‌شود.

به کلِ این مسیرِ خودکار می‌گویند **Pipeline** (خط لوله) که از چند **مرحله (stage)** تشکیل شده — دقیقاً همان سه قدمِ نوار نقاله:

```
کد تازه (git push)
      │
      ▼
┌──────────┐   ┌──────────┐   ┌──────────┐
│  Build   │ → │   Test   │ → │  Deploy  │
│ جعبه بساز│   │ بچش که   │   │ روی کشتی │
│          │   │ خراب نباشد│  │ بفرست    │
└──────────┘   └──────────┘   └──────────┘
```

اگر هر مرحله‌ای **قرمز** شد، نوار نقاله می‌ایستد و کدِ خراب هرگز به مردم نمی‌رسد. آشپز (برنامه‌نویس) فقط دستور را عوض می‌کند؛ بقیه خودکار است.

یکی از رایج‌ترین نوار نقاله‌ها **GitHub Actions** است که در یک فایل YAML نوشته می‌شود:

```yaml
# .github/workflows/ci.yml
name: CI
on: [push]                       # هر بار push شد، راه بیفت
jobs:
  build-test-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4          # کد را بردار
      - run: docker build -t cat-api .     # مرحله‌ی Build: جعبه را بساز
      - run: docker run cat-api pytest     # مرحله‌ی Test: بچش
      - run: docker push myregistry/cat-api  # مرحله‌ی Deploy: بفرست
```

## صحنه‌ی ۴ — نقشه‌ی بندر روی کاغذ (Infrastructure as Code)

«دفترِ نقشه‌ی بندر» در دنیای واقعی **Infrastructure as Code (IaC)** است: به جای اینکه با دست و کلیک در پنل ابری سرور و شبکه بسازی (که فردا یادت می‌رود چه کردی)، **کلِ زیرساخت را در فایل‌های متنی می‌نویسی**. آن فایل را اجرا کنی، همان بندر ساخته می‌شود؛ به یک محیطِ خالی بدهی، **مو به مو** همان دوباره ساخته می‌شود. مشهورترین ابزارش **Terraform** است.

اما اینجا یک نکته‌ی فکریِ مهم هست — فرق **declarative** و **imperative**:

| سبک | یعنی | مثال در قصه |
|---|---|---|
| **Imperative** (دستوری) | **قدم‌به‌قدم می‌گویی چه‌کار کن** | «اول این چاله را بکن، بعد این پایه را بگذار، بعد...» |
| **Declarative** (اعلامی) | فقط **نتیجه‌ی نهایی را اعلام می‌کنی** | «یک بندر با ۳ اسکله می‌خواهم» — راهش با ابزار |

Terraform و k8s هر دو **declarative** هستند: تو **حالتِ مطلوب** را می‌نویسی، و ابزار خودش می‌فهمد چه باید بسازد، چه باید پاک کند تا واقعیت به خواسته‌ات برسد. (دقیقاً همان منطقِ «من فقط می‌گویم چه می‌خواهم» آقای بندر.) یک اسکریپت bash که خط‌به‌خط دستور می‌دهد، imperative است.

یک تکه Terraform این‌طوری است — توجه کن که فقط **چه چیزی** را توصیف می‌کند، نه **چطور**:

```hcl
resource "aws_instance" "web" {     # «یک سرور می‌خواهم»
  ami           = "ami-0abc123"     # با این سیستم‌عامل
  instance_type = "t3.micro"        # این اندازه
  tags = { Name = "cat-api-server" }
}
```

## صحنه‌ی ۵ — دوربین، دفتر و حسگر (Observability)

«چشم و گوشِ بندر» در دنیای واقعی **Observability** (رصدپذیری) است، که روی **سه ستون (three pillars)** ایستاده — هر کدام معادلِ یکی از وسایلِ قصه:

| در قصه | ستون | یعنی چه | ابزار رایج |
|---|---|---|---|
| 📷 **دوربینِ عددده** | **Metrics** (سنجه‌ها) | اعدادِ لحظه‌ای در طول زمان: مصرف CPU، تعداد درخواست، تأخیر | Prometheus + Grafana |
| 📒 **دفترِ ثبتِ اتفاقات** | **Logs** (لاگ‌ها) | متنِ هر رویداد: «ساعت ۳:۰۰، خطای ۵۰۰ رخ داد» | جمع‌آوریِ متمرکز (aggregation) |
| 🔦 **حسگرِ ردگیرِ مسیر** | **Traces** (ردها) | سفرِ یک درخواست از سرویسی به سرویس دیگر | Jaeger / OpenTelemetry |

- **Metrics**: عدد است و ارزان؛ خوب برای نمودار و هشدار («اگر CPU بالای ۹۰٪ رفت، خبرم کن»). Prometheus عددها را جمع می‌کند و Grafana آن‌ها را نمودار می‌کشد.
- **Logs**: وقتی ده‌ها کانتینر هر کدام دفترِ خودشان را دارند، باید همه را در **یک جای متمرکز** جمع کنی (log aggregation)، و بهتر است لاگ‌ها **ساخت‌یافته (structured)** باشند — یعنی به جای متنِ آزاد، به شکل JSON تا ماشین بتواند جستجو و فیلتر کند.
- **Traces**: در یک درخواست که از سرویسِ A به B به C می‌رود، **trace** کلِ سفر است و هر قدمش یک **span**. این همان «ردگیریِ توزیع‌شده (distributed tracing)» است که نشان می‌دهد **کدام قدم** کُند بوده.

### فرق ظریف: Monitoring در برابر Observability

اینجا همان جمله‌ی پایانیِ آقای بندر مهم می‌شود:

- **Monitoring** = پاسخ به سؤال‌هایی که **از قبل می‌دانستی** خواهی پرسید. داشبوردها و هشدارهای آماده: «آیا CPU بالاست؟ آیا سرویس بالاست؟» — تماشای **مجهول‌های شناخته‌شده (known unknowns)**.
- **Observability** = توانایی پاسخ به سؤال‌هایی که **از قبل فکرش را هم نکرده بودی**، فقط با کاوش در داده‌ی موجود. وقتی یک باگِ عجیب پیش می‌آید که هرگز پیش‌بینی‌اش نکرده بودی، سیستمِ رصدپذیر آن‌قدر داده‌ی غنی دارد که بتوانی **مجهول‌های ناشناخته (unknown unknowns)** را هم کشف کنی — بدون اینکه از قبل برایشان داشبورد ساخته باشی.

به زبان ساده: monitoring می‌گوید **چیزی خراب است**؛ observability کمک می‌کند بفهمی **چرا** — حتی اگر دلیلش را پیش‌بینی نکرده بودی. به همین خاطر سام تعجب کرد که بندر «به هر سؤالی جواب دارد، حتی آن‌هایی که فکرش را هم نکرده بودیم».

## 🗺️ کل سفرِ بندر در یک نگاه

```
برنامه‌نویس کد را عوض می‌کند (git push)
   │
   ▼
🎢 CI/CD: Build → Test → جعبه‌ی تازه (Image) ساخته و به Registry فرستاده شد
   │
   ▼
📐 IaC (Terraform): بندر/سرورها از قبل دقیقاً طبق نقشه ساخته شده‌اند
   │
   ▼
🚢 Kubernetes: ایمیج تازه را در Podها روی Nodeها می‌چیند
        ├─ همیشه N نسخه زنده (self-healing)
        ├─ شلوغ شد؟ خودکار بیشتر می‌چیند (scaling)
        └─ Service یک درِ ثابت برای رسیدن به Podها می‌سازد
   │
   ▼
📷 Observability: Metrics + Logs + Traces هر لحظه حالِ بندر را می‌گویند
```

این چرخه مدام تکرار می‌شود — هر تغییرِ کوچک، خودکار از نوار نقاله می‌گذرد و امن به دستِ کاربر می‌رسد.
