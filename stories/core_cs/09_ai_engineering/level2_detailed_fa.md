# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی دانا 🔍🤖

> همان قصه‌ی سام و دانا، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعی *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — دانا واقعاً کیست؟ (LLM)

در دنیای واقعی، «دانا» یک **مدل زبانی بزرگ** (Large Language Model — به اختصار LLM) است؛ مثل GPT، Claude یا Llama. این یک برنامه‌ی غول‌پیکر است که روی **حجم عظیمی از متنِ دنیا** (کتاب، سایت، کد) تمرین داده شده.

اما رازِ بزرگ این است: دانا در عمق وجودش فقط **یک کارِ ساده** بلد است — حدس زدن **کلمه‌ی بعدی**. تو نصف جمله را می‌دهی، او محتمل‌ترین تکه‌ی بعدی را می‌گوید، بعد دوباره، و دوباره. از همین یک کارِ ساده، انگار که فکر می‌کند بیرون می‌آید. به این می‌گویند **next-token prediction**.

## صحنه‌ی ۲ — تکه‌تکه کردن جمله (Tokenization)

دانا کلمه‌ها را همان‌طور که ما می‌بینیم نمی‌بیند. اول جمله را به تکه‌هایی به اسم **توکن** (Token) می‌شکند. هر توکن می‌تواند یک کلمه، نصفِ یک کلمه، یا حتی یک علامت باشد.

روشِ شکستن **BPE** (Byte-Pair Encoding) نام دارد: کلمه‌های پرتکرار یک توکنِ کامل می‌شوند، اما کلمه‌های نادر به تکه‌های کوچک‌تر شکسته می‌شوند.

```
"unbelievable"  →  ["un", "believ", "able"]      (سه توکن)
"cat"           →  ["cat"]                        (یک توکن)
```

چرا برایمان مهم است؟ سه دلیل بزرگ:

| در قصه | معنی واقعی |
|---|---|
| «جمله را تکه‌تکه می‌کنم» | Tokenization |
| «تکه‌های پازل» | توکن‌ها (tokens) |
| «مغزم جا برای تعداد مشخصی تکه دارد» | پنجره‌ی متن (Context Window) |
| «اولِ قصه‌ی طولانی یادم می‌رود» | محدودیت context window — توکن‌های بیرون از پنجره دیده نمی‌شوند |

- **پول و سرعت**: شرکت‌ها بر اساس **تعداد توکن** پول می‌گیرند، نه تعداد کلمه. جمله‌ی طولانی = توکن بیشتر = گران‌تر و کندتر.
- **پنجره‌ی متن (Context Window)**: حداکثر توکنی که مدل یک‌جا می‌تواند ببیند (مثلاً ۸٬۰۰۰ یا ۲۰۰٬۰۰۰). هرچه بیرونِ این پنجره باشد، انگار وجود ندارد.

## صحنه‌ی ۳ — حسِ نزدیکی (Embeddings)

دانا هر تکه‌ی معنا را به یک **لیست از عددها** تبدیل می‌کند که به آن **embedding** (یا **vector** = بُردار) می‌گویند. این عددها مختصاتِ آن کلمه روی یک نقشه‌ی غول‌پیکرِ معنایی‌اند.

```
"سیب"     → [0.91, 0.10, 0.77, ...]   ┐ نزدیک هم
"گلابی"   → [0.89, 0.12, 0.74, ...]   ┘
"دوچرخه"  → [0.02, 0.95, 0.08, ...]   ← خیلی دور
```

برای اندازه‌گیریِ «نزدیکی»، زاویه‌ی بین دو بردار را حساب می‌کنیم — به این می‌گویند **cosine similarity** (شباهت کسینوسی): عدد نزدیک به ۱ یعنی خیلی شبیه، نزدیک به ۰ یعنی بی‌ربط.

| در قصه | معنی واقعی |
|---|---|
| «حسِ نزدیکی» | embedding / vector |
| «سیب و گلابی نزدیک‌اند» | cosine similarity بالا |
| «سیب و دوچرخه دورند» | cosine similarity پایین |

## صحنه‌ی ۴ — نگاه کردن به کلمه‌های مهم (Attention و Transformer)

اینکه دانا می‌فهمد «پرید» مالِ «گربه» است نه «سام»، جادویِ یک ساختار به اسم **Transformer** است و قلبِ آن **مکانیزم توجه** (Attention).

شهودِ Attention را با سه نقش بفهم — برای هر کلمه، دانا سه چیز می‌سازد:

- **Query (پرسش)**: «من دنبال چه‌جور اطلاعاتی می‌گردم؟»
- **Key (کلید/برچسب)**: «من چه‌جور اطلاعاتی دارم؟»
- **Value (محتوا)**: «اطلاعاتِ واقعیِ من این است.»

هر کلمه با Queryِ خودش، Keyِ همه‌ی کلمه‌های دیگر را وارسی می‌کند؛ هرجا Query و Key جور شدند، آن کلمه **توجه بیشتری** جلب می‌کند و Valueش بیشتر در فهم اثر می‌گذارد. چون هر کلمه به **خودِ بقیه‌ی کلمه‌های همین جمله** نگاه می‌کند، اسمش **self-attention** است.

> **چرا Transformer برنده شد؟** مدل‌های قدیمی (RNN) کلمه‌ها را **یکی‌یکی و پشت‌سرِ هم** می‌خواندند — کُند و فراموش‌کار. Transformer به **همه‌ی کلمه‌ها هم‌زمان** نگاه می‌کند؛ هم خیلی موازی‌تر (روی GPU سریع) و هم وابستگی‌های دور را بهتر می‌گیرد. همین مقیاس‌پذیری، عصرِ LLMها را ساخت.

| در قصه | معنی واقعی |
|---|---|
| «برمی‌گردم به کلمه‌های مهم نگاه می‌کنم» | attention |
| «حواسم را روی کلمه‌های درست می‌گذارم» | self-attention |
| «حدسِ کلمه‌ی بعدی» | next-token prediction |
| کلِ توانایی دانا در فهم جمله | معماری Transformer |

## صحنه‌ی ۵ — دفترچه‌ی کتابخانه (RAG)

دانا حافظه‌اش تا روزی که تمرینش تمام شده ثابت است؛ از «دیروزِ مدرسه» چیزی نمی‌داند. اگر بپرسی، یا نمی‌داند یا **از خودش درمی‌آورد** (به این توهم‌گویی می‌گویند: **hallucination**).

راه‌حل: **RAG** (Retrieval-Augmented Generation = تولیدِ تقویت‌شده با بازیابی). قبل از جواب دادن، اول می‌رویم در یک کتابخانه‌ی مدارکِ خودمان **می‌گردیم**، تکه‌های مرتبط را پیدا می‌کنیم، آن‌ها را **چسبیده به سوال** به دانا می‌دهیم، و بعد دانا با **استناد به آن متنِ واقعی** جواب می‌دهد.

خطِ تولیدِ کاملِ RAG شش قدم دارد:

```
                        ┌── مرحله‌ی آماده‌سازی (یک‌بار، آفلاین) ──┐
   مدارک خام                                                    
      │                                                          
      ▼                                                          
 ١. Chunk    تکه‌تکه کردن مدارک به پاراگراف‌های کوچک             
      │                                                          
      ▼                                                          
 ٢. Embed    هر تکه → یک بردار (embedding)                       
      │                                                          
      ▼                                                          
 ٣. Store    ذخیره‌ی بردارها در Vector Database                  
                        └──────────────────────────────────────┘

                        ┌── مرحله‌ی پرسش (هر بار، زنده) ─────────┐
   سوالِ کاربر                                                    
      │                                                          
      ▼                                                          
 ٤. Retrieve   سوال را embed کن، نزدیک‌ترین تکه‌ها را بیاور      
      │                                                          
      ▼                                                          
 ٥. Augment    تکه‌های پیداشده را به prompt بچسبان               
      │           «با توجه به این متن‌ها، به سوال جواب بده: ...»  
      ▼                                                          
 ٦. Generate   LLM با همان متن‌ها جوابِ مستند می‌دهد            
                        └──────────────────────────────────────┘
```

**چرا RAG بهتر از این است که کلِ مدارک را توی prompt بریزیم؟** (به آن می‌گویند *context stuffing*)
- **پنجره‌ی متن محدود است** — کلِ یک کتابخانه جا نمی‌شود.
- **توکنِ زیاد = پول و تأخیرِ زیاد** — هر بار همه‌چیز را فرستادن دیوانگی است.
- **حواس‌پرتی** — وقتی متنِ بی‌ربطِ زیاد بدهی، مدل دقتش پایین می‌آید («lost in the middle»). RAG فقط **مرتبط‌ترین** تکه‌ها را می‌دهد.

**استراتژی‌های Chunking** (مهم‌تر از چیزی است که فکر می‌کنی):
- **اندازه‌ی ثابت** (مثلاً ۵۰۰ توکن) با **هم‌پوشانی** (overlap) تا جمله وسطش قطع نشود.
- **بر اساس ساختار** — هر پاراگراف/بخش/سرتیتر یک chunk.
- تعادل: chunkِ خیلی بزرگ = نویز و توکنِ هدررفته؛ خیلی کوچک = معنا تکه‌پاره و بی‌context.

| در قصه | معنی واقعی |
|---|---|
| «حدس نمی‌زنم» | جلوگیری از hallucination |
| «اول توی دفترچه می‌گردم» | Retrieval |
| «نوشته‌ی درست را پیدا می‌کنم» | top-k relevant chunks |
| «بعد جواب می‌دهم» | Augmented Generation |

## صحنه‌ی ۶ — قفسه‌های مرتب با حسِ نزدیکی (Vector Database)

دانا دفترچه‌ها را با حسِ نزدیکی مرتب کرده، یعنی **بردارهای شبیه کنار هم‌اند**. این کارِ یک **Vector Database** (پایگاه‌داده‌ی برداری) است.

اما گشتنِ یکی‌یکیِ میلیون‌ها بردار کُند است. پس به جای جوابِ **دقیقِ کامل**، از جست‌وجوی **تقریبیِ سریع** استفاده می‌کنیم: **ANN** (Approximate Nearest Neighbor — نزدیک‌ترین همسایه‌ی تقریبی). کمی دقت را فدا می‌کند تا صدها برابر سریع‌تر شود.

نمونه‌ها:
- **pgvector** — افزونه‌ی PostgreSQL؛ وقتی همین حالا دیتابیس SQL داری و نمی‌خواهی ابزار جدا اضافه کنی.
- **Pinecone** — سرویسِ ابریِ مدیریت‌شده‌ی مخصوص بردار؛ مقیاس بالا، دردسر کم.
- **Chroma** — سبک و متن‌باز؛ عالی برای شروع و نمونه‌سازی روی لپ‌تاپ.

| در قصه | معنی واقعی |
|---|---|
| «قفسه‌ها به ترتیب الفبا نیستند» | ایندکس برداری، نه ایندکس متنی |
| «شبیه‌ها کنار هم‌اند» | فضای embedding |
| «یک‌راست می‌روم سراغ قفسه‌ی درست» | جست‌وجوی ANN |

## صحنه‌ی ۷ — کلاس مخصوص (Fine-Tuning)

وقتی دانا را به کلاس مخصوص می‌فرستیم، در حال **fine-tuning** (تنظیم دقیق) هستیم: یک مدلِ از پیش‌آموخته را با داده‌ی تخصصیِ خودمان **بیشتر آموزش** می‌دهیم تا در یک کارِ خاص بهتر شود یا یک **سبک/لحنِ** خاص بگیرد.

سوالِ کلیدیِ هر مهندس: **RAG یا Fine-Tuning؟**

| | RAG | Fine-Tuning |
|---|---|---|
| به مدل **دانشِ تازه** می‌دهد | ✅ عالی | ❌ ضعیف |
| **رفتار/سبک/فرمت** را عوض می‌کند | ❌ محدود | ✅ عالی |
| داده مدام عوض می‌شود | ✅ فقط دیتابیس را به‌روز کن | ❌ باید دوباره آموزش بدهی |
| هزینه‌ی راه‌اندازی | کم | زیاد (GPU + داده) |

جمله‌ی طلایی: **RAG برای «چه چیزی بداند»، Fine-Tuning برای «چطور رفتار کند».**

و یک ترفندِ ارزان: لازم نیست کلِ مدلِ غول‌پیکر را دوباره آموزش بدهی. با **LoRA** فقط چند صفحه‌ی کوچکِ افزودنی را آموزش می‌دهی و به مدل می‌چسبانی — ارزان، سریع، و قابل‌جابه‌جایی.

## صحنه‌ی ۸ — دانا چطور به ما خدمت می‌دهد؟ (Model Serving)

دانا یک مدلِ سنگین است که روی کارت‌های گرافیکیِ گران (GPU) اجرا می‌شود. برای اینکه برنامه‌ی ما بتواند از او استفاده کند، پشتِ یک **Inference API** قرار می‌گیرد — یک آدرس که سوال می‌فرستی و جواب می‌گیری.

چند چیز که در دنیای واقعی مهم می‌شوند:
- **تأخیر و هزینه (Latency & Cost)**: مدل بزرگ‌تر = باهوش‌تر اما کندتر و گران‌تر. همیشه یک معامله است.
- **Streaming**: به‌جای انتظار برای کلِ جواب، توکن‌ها را **همان‌طور که تولید می‌شوند** کلمه‌به‌کلمه می‌فرستد — همان افکتِ تایپ‌شدنِ زنده.
- **Batching**: چند درخواست را **با هم** روی GPU پردازش می‌کند تا کارایی بالا برود.
- **Quantization**: عددهای داخل مدل را با دقتِ کمتر (مثلاً ۴ بیت به‌جای ۱۶ بیت) ذخیره می‌کند؛ مدل کوچک‌تر و سریع‌تر می‌شود با کمی افتِ کیفیت — این‌طوری مدل حتی روی سخت‌افزارِ ضعیف‌تر هم جا می‌شود.

## صحنه‌ی ۹ — ماشین‌حساب کنار دست (Tool Calling)

دانا حساب را بلد نیست، اما بلد است بگوید «ماشین‌حساب لازم دارم». این **Tool Calling** (یا Function Calling) است.

نکته‌ی ظریف: **خودِ مدل ماشین‌حساب را اجرا نمی‌کند!** مدل فقط یک **درخواستِ ساخت‌یافته** برمی‌گرداند («تابعِ `multiply` را با ۸۳۷ و ۴۹۲ صدا بزن»)، **برنامه‌ی تو** آن را اجرا می‌کند، و **نتیجه را به مدل برمی‌گردانی** تا جوابِ نهایی را بسازد.

حلقه‌ی Tool Calling:

```
   کاربر: «۸۳۷ ضربدر ۴۹۲ چنده؟»
        │
        ▼
 ┌─────────────┐   مدل تصمیم می‌گیرد ابزار لازم است
 │     LLM     │   و یک «درخواست تابع» برمی‌گرداند:
 └─────────────┘   { tool: "multiply", args: {a:837, b:492} }
        │
        ▼
 ┌─────────────┐   ★ کدِ تو ★ تابع را واقعاً اجرا می‌کند
 │ کدِ برنامه  │   نتیجه = 411804
 └─────────────┘
        │  نتیجه را به مدل برمی‌گردانی
        ▼
 ┌─────────────┐   مدل با عددِ درست جوابِ روان می‌سازد:
 │     LLM     │   «۸۳۷ ضربدر ۴۹۲ می‌شود ۴۱۱٬۸۰۴.»
 └─────────────┘
```

| در قصه | معنی واقعی |
|---|---|
| «حدس نمی‌زنم، ماشین‌حساب برمی‌دارم» | مدل به‌جای توهم، tool call برمی‌گرداند |
| «ماشین‌حساب را برمی‌دارم» | برنامه‌ی تو تابع را اجرا می‌کند |
| «جواب را می‌گویم» | مدل با نتیجه‌ی تابع جوابِ نهایی می‌سازد |

## صحنه‌ی ۱۰ — برنامه‌ریزی برای کارِ بزرگ (Agentic Workflows)

پختنِ کیک یک قدم نبود، چند قدم بود. وقتی به دانا یک **هدف** می‌دهیم و او خودش قدم‌ها را می‌چیند و اجرا می‌کند، یک **Agent** (عامل) داریم.

عاملِ هوشمند در یک **حلقه** کار می‌کند:

```
        ┌──────────────────────────────┐
        ▼                              │
   ١. Plan (نقشه)   «قدم بعدی چیست؟»    │
        │                              │
        ▼                              │
   ٢. Act (عمل)    یک ابزار را صدا بزن  │
        │                              │
        ▼                              │
   ٣. Observe (مشاهده) نتیجه را ببین ───┘
        │
        ▼ (وقتی هدف کامل شد)
      جوابِ نهایی
```

به این الگوی «فکر کن → عمل کن → نتیجه را ببین → دوباره فکر کن» می‌گویند **ReAct** (Reasoning + Acting). تفاوتش با Tool Calling ساده این است که عامل **خودش تصمیم می‌گیرد چند قدم لازم است و کِی تمام شده** — نه فقط یک ابزار، بلکه یک زنجیره‌ی پویا.

## صحنه‌ی ۱۱ — پریزِ همه‌کاره (MCP)

هر ابزار (ماشین‌حساب، تقویم، دیتابیس شرکت) یک سیمِ مخصوصِ خودش می‌خواست و وصل کردنِ هرکدام به هر مدل دردسر جدا بود. اگر `N` ابزار و `M` مدل داری، باید `N×M` وصله بنویسی!

**MCP** (Model Context Protocol) همان **پریزِ همه‌کاره** است: یک **استانداردِ باز** برای وصل کردنِ ابزارها و داده‌ها به مدل‌ها. مثل USB-C برای هوش مصنوعی.

- **MCP Server**: ابزار/داده‌ات را پشت این استاندارد عرضه می‌کند (مثلاً یک سرورِ MCP برای دیتابیس شرکت).
- **MCP Client**: برنامه‌ی هوش مصنوعیِ تو که به هر سرورِ MCP وصل می‌شود — بدون اینکه جزئیاتِ داخلیِ آن ابزار را بداند.

**چرا استانداردسازی مهم است؟** یک‌بار ابزارت را به‌صورت MCP بنویس، آن‌وقت **هر** مدل و **هر** برنامه‌ای که MCP بلد است می‌تواند ازش استفاده کند. از `N×M` وصله به `N+M` رسیدیم.

| در قصه | معنی واقعی |
|---|---|
| «سیمِ جدا برای هر ابزار» | اتصالِ سفارشی N×M |
| «پریزِ همه‌کاره» | پروتکل استاندارد MCP |
| «هر ابزاری بهش وصل می‌شود» | هر MCP Server به هر MCP Client |

```
١. Tokenize: سوال به توکن‌ها شکسته می‌شود
   │
   ▼
٢. Embed: سوال به بردار تبدیل می‌شود (حسِ نزدیکی)
   │
   ▼
٣. RAG Retrieve: نزدیک‌ترین تکه‌ها از Vector DB با جست‌وجوی ANN
   │
   ▼
٤. Augment: تکه‌ها چسبیده به سوال در prompt
   │
   ▼
٥. LLM (Transformer + Attention): کلمه‌ی بعدی را پی‌درپی حدس می‌زند
   │
   ├─► لازم شد ابزار؟ → Tool Calling → نتیجه برگردد → ادامه
   │     (و اگر چند قدمی بود → حلقه‌ی Agent: plan→act→observe)
   │     (ابزارها از طریق MCP وصل شده‌اند)
   ▼
٦. Generate (با Streaming): جوابِ مستند، کلمه‌به‌کلمه
```

این همان مغزِ هر دستیارِ هوش مصنوعیِ مدرن است!
