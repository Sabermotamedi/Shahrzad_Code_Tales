# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی دفترچه‌ی جادویی 🔍📓

> همان قصه‌ی سام و نرگس و دفترچه‌ی جادویی، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعیِ Git *دقیقاً* با کدام دستور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — دفترچه واقعاً چیست؟ (Repository)

«دفترچه‌ی جادویی» در دنیای واقعی یک **مخزن** (Repository یا به‌اختصار repo) است: یک پوشه که Git تمام تاریخچه‌اش را در یک زیرپوشه‌ی مخفی به نام `.git` نگه می‌دارد.

برای ساختن یک دفترچه‌ی تازه:

```bash
git init my-story        # یک repo خالیِ تازه می‌سازد
cd my-story
```

از این لحظه، هر کاری در این پوشه بکنی، دفترچه حواسش هست.

## صحنه‌ی ۲ — سه قفسه‌ی دفترچه (سه درخت / The Three Trees)

قبل از اینکه یک «عکس» ذخیره شود، نوشته‌ی سام از **سه قفسه** عبور می‌کند. فهم این سه قفسه، کلیدِ فهم کل Git است:

| در قصه | اسم فنی | یعنی چه |
|---|---|---|
| میز کارِ سام که رویش می‌نویسد | **Working Directory** (پوشه‌ی کار) | فایل‌هایی که همین الان داری ویرایش می‌کنی |
| سینیِ «این‌ها را آماده‌ی عکس کن» | **Staging Area** (یا Index) | چیزهایی که انتخاب کرده‌ای در عکسِ بعدی باشند |
| آلبومِ عکس‌های ذخیره‌شده | **Commit** (تاریخچه / `.git`) | عکس‌های دائمیِ ثبت‌شده |

جریان کار همیشه همین است:

```
میز کار  ──git add──▶  سینی (Staging)  ──git commit──▶  آلبوم (تاریخچه)
working                   index                            history
```

```bash
git status               # ببین چه چیزی روی میز است و چه چیزی در سینی
git add story.txt        # فایل را از میز بگذار توی سینی
git add .                # همه‌ی تغییرات را بگذار توی سینی
```

چرا یک سینیِ وسط؟ تا بتوانی **انتخاب کنی** کدام تغییرات در عکسِ بعدی باشند — مثلاً ده فایل عوض کرده‌ای ولی فقط می‌خواهی سه‌تایش با هم ذخیره شوند.

## صحنه‌ی ۳ — گرفتن عکس با یادداشت (Commit و پیام آن)

«ذخیره با یادداشت» در دنیای واقعی همان **commit** است: یک عکسِ دائمی از سینی، به‌علاوه‌ی یک **پیام** (commit message).

```bash
git commit -m "شخصیت اژدها را اضافه کردم"
```

هر commit یک **شناسه‌ی یکتا** (hash، چیزی شبیه `a1b2c3d`) می‌گیرد و می‌داند **عکسِ قبلی‌اش کدام بوده** (به آن می‌گویند *parent*). پس commitها مثل دانه‌های یک زنجیر به هم وصل‌اند:

```
o─────o─────o
شروع  اژدها  شاهزاده
```

**پیام خوب** خیلی مهم است — مثل یادداشتِ روی عکس. قراردادی محبوب به اسم **Conventional Commits** کمک می‌کند پیام‌ها مرتب باشند:

```
feat: شخصیت اژدها را اضافه کن        ← feature (قابلیت تازه)
fix: غلط املایی صفحه‌ی ۳ را درست کن   ← bug fix
docs: راهنما را به‌روز کن
```

برای دیدن آلبومِ خاطرات و مقایسه‌ی نسخه‌ها:

```bash
git log --oneline        # فهرست کوتاهِ همه‌ی commitها
git diff                 # روی میز چه چیزی نسبت به آخرین عکس عوض شده؟
git diff --staged        # در سینی چه چیزی آماده‌ی عکس است؟
```

## صحنه‌ی ۴ — انگشتِ اشاره به «الان کجاییم» (HEAD)

دفترچه همیشه می‌داند **روی کدام عکس ایستاده‌ای**. این انگشتِ اشاره در Git اسمش **HEAD** است — یعنی «نسخه‌ای که همین الان رویش هستی». معمولاً HEAD به آخرین commit شاخه‌ی فعلی اشاره می‌کند.

```
o─────o─────o   ← HEAD (اینجا ایستاده‌ایم)
```

## صحنه‌ی ۵ — لیست نادیده‌ها (.gitignore)

«لیست نادیده‌ها» همان فایلِ **`.gitignore`** است: فهرستی از چیزهایی که Git باید کاملاً نادیده بگیرد و هرگز عکسشان را نگیرد.

```gitignore
# .gitignore
node_modules/        # کتابخانه‌های دانلودی — حجیم و قابل بازسازی
*.log                # فایل‌های لاگ
.env                 # رمزها و کلیدهای محرمانه! ← خیلی مهم
secret-notes.txt
```

> نکته: `.gitignore` فقط فایل‌هایی را که **هنوز ذخیره نشده‌اند** نادیده می‌گیرد. اگر فایلی قبلاً commit شده، اول باید با `git rm --cached` از دستِ Git خارجش کنی.

## صحنه‌ی ۶ — سفر در زمان و پس گرفتن اشتباه (restore / reset / revert)

«سفر در زمان» و «پاک کردن خطا» در Git سه ابزار جدا دارند که خیلی‌ها قاطی‌شان می‌کنند:

| در قصه | دستور | چه می‌کند |
|---|---|---|
| «این صفحه را به حالتِ آخرین عکس برگردان» | `git restore <file>` | تغییرات ذخیره‌نشده‌ی یک فایل را دور می‌ریزد |
| «این صفحه را از سینی بردار» | `git restore --staged <file>` | فایل را از سینی بیرون می‌آورد (بدون پاک کردن نوشته) |
| «انگشتِ اشاره را عقب ببر» | `git reset` | HEAD را به یک commit قبلی می‌برد (سه حالت دارد، پایین‌تر) |
| «یک عکسِ تازه بگیر که اثرِ عکس قبلی را خنثی کند» | `git revert <commit>` | یک commit *جدید* می‌سازد که تغییرِ یک commit قدیمی را برعکس می‌کند |

### سه حالتِ reset

فرض کن می‌خواهی عقب بروی. سؤال این است: با نوشته‌هایت چه کنم؟

```bash
git reset --soft  HEAD~1   # عکس را باز کن، نوشته‌ها در سینی بمانند
git reset --mixed HEAD~1   # (پیش‌فرض) عکس را باز کن، نوشته‌ها روی میز بمانند ولی از سینی خارج
git reset --hard  HEAD~1   # عکس را باز کن و نوشته‌ها را هم پاک کن ⚠️ خطرناک!
```

| حالت | HEAD عقب می‌رود؟ | سینی پاک می‌شود؟ | میز کار پاک می‌شود؟ |
|---|---|---|---|
| `--soft` | ✅ | ❌ | ❌ |
| `--mixed` | ✅ | ✅ | ❌ |
| `--hard` | ✅ | ✅ | ✅ ⚠️ |

### قانون طلاییِ امنیت: `reset` یا `revert`؟

> اگر آن commit را **هنوز به کتابخانه (remote) نفرستاده‌ای** و فقط مال خودت است → `reset` آزاد است.
> اگر **قبلاً به کتابخانه فرستاده‌ای** و دیگران آن را گرفته‌اند → هرگز `reset` نکن! تاریخچه‌ی مشترک را عوض می‌کنی و دفترچه‌ی همه به هم می‌ریزد. به‌جایش `revert` کن — یک عکسِ تازه که خطا را خنثی می‌کند، بدون دست‌کاری گذشته.

## صحنه‌ی ۷ — کپیِ رویایی (Branch) و عوض کردن دنیا (switch)

«کپیِ رویایی» همان **branch** (شاخه) است: یک خطِ موازی از کار که از یک نقطه جدا می‌شود. در واقع branch فقط یک **اشاره‌گرِ سبک** به یک commit است — نه یک کپیِ سنگین!

```bash
git branch dragon-idea       # یک شاخه‌ی تازه بساز
git switch dragon-idea       # برو روی آن شاخه (روش قدیمی: git checkout)
git switch -c dragon-idea    # هر دو با هم: بساز و برو رویش
```

حالا تاریخچه دو راه می‌شود:

```
            o───o   ← dragon-idea (کپیِ رویاییِ سام)
           /
o───o───o          ← main (اصلِ دفترچه)
```

`main` (یا `master` در repoهای قدیمی‌تر) همان «اصلِ دفترچه» است. سام روی `dragon-idea` هر کاری بکند، `main` دست‌نخورده می‌ماند. اگر ایده خراب شد: `git branch -D dragon-idea` و انگار هیچ نشده.

## صحنه‌ی ۸ — به هم بافتن دو نسخه (Merge)

«به هم بافتن» همان **merge** است. سام کارش روی شاخه تمام شده و می‌خواهد آن را به `main` برگرداند:

```bash
git switch main
git merge dragon-idea
```

دو حالت پیش می‌آید:

**حالت ۱ — Fast-forward:** اگر `main` از وقتی شاخه جدا شد هیچ تغییری نکرده باشد، Git فقط اشاره‌گرِ `main` را جلو می‌کشد. هیچ commit تازه‌ای ساخته نمی‌شود — یک خطِ صاف:

```
قبل:  o───o───o            ← main
                \
                 o───o      ← feature

بعد:  o───o───o───o───o     ← main (فقط جلو کشیده شد)
```

**حالت ۲ — Merge commit:** اگر در این مدت `main` هم جلو رفته باشد، Git یک **commit دو-والدی** می‌سازد که هر دو خط را به هم می‌بافد:

```
o───o───o─────────M     ← main (M همان merge commit است)
         \       /
          o─────o       ← feature
```

## صحنه‌ی ۹ — وقتی هر دو یک خط را عوض کردند (Conflict)

«دعوای آرش و کاوه» همان **merge conflict** است: وقتی دو شاخه **همان خط** از همان فایل را جور دیگری عوض کرده‌اند، Git نمی‌تواند خودش تصمیم بگیرد و از تو کمک می‌خواهد.

Git داخل فایل، هر دو نسخه را با علامت‌گذاری به تو نشان می‌دهد:

```
<<<<<<< HEAD
اسم قهرمان: آرش
=======
اسم قهرمان: کاوه
>>>>>>> dragon-idea
```

تو فایل را باز می‌کنی، علامت‌ها را پاک می‌کنی، نسخه‌ی درست را نگه می‌داری («کاوه»)، و بعد:

```bash
git add story.txt        # «حلش کردم، این نسخه درست است»
git commit               # merge را نهایی کن
```

## صحنه‌ی ۱۰ — راه دیگرِ بافتن: Rebase (و قانون طلایی)

دفترچه یک ترفند دیگر هم برای یکی‌کردن کارها داشت: به‌جای اینکه دو نخ را *ببافد*، می‌توانست **commitهای شاخه را بردارد و یکی‌یکی روی نوکِ `main` دوباره بنشاند** — انگار از اول روی نسخه‌ی جدید کار کرده بودی. این **rebase** است و یک تاریخچه‌ی **صاف و خطی** می‌سازد:

```
Merge (تاریخچه‌ی واقعی، شاخه‌دار):      Rebase (تاریخچه‌ی صاف، بازنویسی‌شده):

o───o───o───M    ← main               o───o───o───o'──o'   ← main
     \     /                                      (commitها جابه‌جا شدند)
      o───o      ← feature
```

```bash
git switch feature
git rebase main          # commitهای feature را روی نوک main دوباره بچین
```

> ⚖️ **قانون طلاییِ rebase:** هرگز شاخه‌ای را که **به کتابخانه فرستاده‌ای و دیگران گرفته‌اند** rebase نکن! چون rebase commitهای قدیمی را پاک می‌کند و **commitهای جدید با شناسه‌ی تازه** می‌سازد — تاریخچه‌ی همه به هم می‌ریزد. rebase فقط برای کارِ خصوصیِ خودت، **قبل از** اشتراک‌گذاری.

## صحنه‌ی ۱۱ — کتابخانه‌ی شهر (Remote و GitHub)

«کتابخانه‌ی شهر» همان **remote** است: یک کپیِ مشترک از repo که روی سروری مثل **GitHub** یا GitLab زندگی می‌کند. اسمِ پیش‌فرضِ این کتابخانه **`origin`** است.

```bash
git clone https://github.com/sam/story.git   # کل کتابخانه را برای اولین بار بگیر
```

`clone` کل تاریخچه را کپی می‌کند و خودکار remote را با اسم `origin` ثبت می‌کند.

### فرستادن و گرفتن

```bash
git push origin main         # کارِ من را به کتابخانه بفرست
git fetch origin             # ببین در کتابخانه چه چیز تازه‌ای هست (ولی به کارم دست نزن)
git pull origin main         # fetch + merge: تازه‌ها را بگیر و با کارم یکی کن
```

> **فرق `fetch` و `pull` خیلی مهم است:**
> `fetch` فقط **خبر می‌آورد** («در کتابخانه این چیزها تازه است») بدون اینکه به کارِ روی میزت دست بزند.
> `pull` همان `fetch` است **به‌علاوه‌ی** یک `merge` خودکار. یعنی `pull = fetch + merge`.

### شاخه‌های ردیاب (Tracking Branches)

وقتی `clone` یا `pull` می‌کنی، شاخه‌ی محلیِ `main` تو می‌داند که جفتش در کتابخانه، `origin/main` است. به این رابطه می‌گویند **tracking** — به همین خاطر `git push` و `git pull` بدون نوشتنِ اسم، می‌دانند کجا بروند.

## صحنه‌ی ۱۲ — «اول صفحه‌هایم را بخوان» (Pull Request)

«رفتن پیش کتابدار و گفتن لطفاً بخوان» همان **Pull Request** (در GitHub) یا **Merge Request** (در GitLab) است. این یک قابلیتِ پلتفرم است، نه خود Git.

جریانش این است:

```
۱. سام شاخه‌ی feature را به origin می‌فرستد:   git push origin feature/dragon
۲. در GitHub یک Pull Request باز می‌کند:        feature/dragon ──▶ main
۳. نرگس کد را review می‌کند، نظر می‌گذارد، تأیید می‌کند (Approve)
۴. وقتی همه راضی شدند، PR در main ادغام (merge) می‌شود
```

- **هدفِ PR:** هیچ تغییری مستقیم به `main` نرود؛ همه‌چیز اول **خوانده و تأیید** شود.
- **Draft PR (پیش‌نویس):** وقتی کارت هنوز تمام نشده ولی می‌خواهی زود بازخورد بگیری، PR را به‌صورت «پیش‌نویس» باز می‌کنی — یعنی «هنوز آماده‌ی merge نیست، فقط نگاهی بینداز».

## صحنه‌ی ۱۳ — جیب مخفی، برداشتن یک عکس، و برچسب

سه ترفندِ کوچکِ آخرِ دفترچه:

**جیب مخفی (Stash):** کارِ نیمه‌تمامت را موقتاً کنار بگذار، یک کار فوریِ دیگر بکن، بعد برگرد:

```bash
git stash            # کارِ روی میز را بگذار توی جیب (میز تمیز می‌شود)
git switch main      # برو یک کار فوری بکن...
git switch feature
git stash pop        # کارِ نیمه‌تمام را از جیب دربیاور و ادامه بده
```

**برداشتن فقط یک عکسِ خاص (Cherry-pick):** فقط *یک* commit از یک شاخه‌ی دیگر را بردار و روی شاخه‌ی فعلی بچسبان:

```bash
git cherry-pick a1b2c3d    # فقط همین یک commit را بیاور
```

**برچسب طلایی (Tag):** یک نقطه‌ی مهم (مثل انتشار نسخه) را با اسم نشان‌دار کن:

```bash
git tag -a v1.0 -m "نسخه‌ی ۱ — آماده‌ی انتشار"
git push origin v1.0
```

## صحنه‌ی ۱۴ — پیدا کردن صفحه‌ای که قصه را خراب کرد (git bisect)

یک روز فهمیدند یک‌جای قصه خراب شده، ولی نمی‌دانستند کِی! دفترچه گفت: «بیا با هم **جستجوی دوبخشی** کنیم — هر بار وسطِ تاریخچه را چک می‌کنیم.» این همان **`git bisect`** است که با تقسیمِ مدام، خرابیِ بین صدها commit را در چند قدم پیدا می‌کند (پایین‌تر در سطح ۳ کامل می‌بینیمش).

## صحنه‌ی ۱۵ — نقشه‌های کار کردن با هم (Workflowها)

تیم‌ها برای اینکه روی هم نریزند، یک **نقشه‌ی کار** (workflow) انتخاب می‌کنند:

| نقشه | ایده‌ی اصلی | برای چه کسی؟ |
|---|---|---|
| **GitHub Flow** | `main` همیشه سالم؛ برای هر کار یک شاخه‌ی کوتاه، PR، merge، deploy | تیم‌های وب و سرویس‌های ابری، deploy مداوم |
| **Git Flow** | شاخه‌های جدا برای develop/release/hotfix | محصولاتی با نسخه‌های رسمی و منظم (مثل اپ موبایل) |
| **Trunk-based** | همه نزدیک به `main` کار می‌کنند، شاخه‌ها خیلی کوتاه‌عمر | تیم‌های بزرگ با CI/CD قوی و سرعت بالا |

نخِ مشترکِ همه: کارِ هر کس روی یک **شاخه‌ی feature** جدا انجام می‌شود تا `main` همیشه قابل‌اعتماد بماند.

## 🗺️ کل سفر در یک نگاه

```
        ┌─────────────── میز کار (Working Directory) ───────────────┐
        │  ویرایش فایل‌ها                                            │
        └──────────────────────────┬───────────────────────────────┘
                                    │ git add
                                    ▼
        ┌─────────────── سینی (Staging Area / Index) ───────────────┐
        └──────────────────────────┬───────────────────────────────┘
                                    │ git commit
                                    ▼
        ┌─────────── تاریخچه‌ی محلی (Local Repo / .git) ─────────────┐
        │   o───o───o   ← main                                      │
        │            \                                              │
        │             o───o   ← feature/dragon                      │
        └──────────────────────────┬───────────────────────────────┘
                            git push │ ▲ git fetch / git pull
                                    ▼ │
        ┌──────────── کتابخانه‌ی شهر (Remote = origin / GitHub) ─────┐
        │   نسخه‌ی مشترک  +  Pull Requestها  +  review              │
        └───────────────────────────────────────────────────────────┘
```

این چرخه را روزی ده‌ها بار تکرار می‌کنی: ویرایش ← `add` ← `commit` ← `push` ← PR ← `merge`. 🔁
