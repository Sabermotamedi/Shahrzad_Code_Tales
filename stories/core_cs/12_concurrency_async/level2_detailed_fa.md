# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی آشپزخانه‌ی شلوغ 🔍🍲

> همان قصه‌ی سام و آشپزخانه، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعیِ برنامه‌نویسی *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

> یادآوری: در فصلِ **سیستم‌عامل** هر آشپز یک «پروسه» (Process) بود — با میز و وسایلِ جداگانه. آنجا قصه از دیدِ سیستم‌عامل بود. اینجا قصه از دیدِ **برنامه‌نویس** است: چند آشپز روی **یک قابلمه** کار می‌کنند، یعنی چند **نخ** (Thread) که **حافظه را مشترک** دارند. همان آشپزخانه، اما این بار تمرکز روی قابلمه‌ی مشترک است.

---

## صحنه‌ی ۱ — قابلمه‌ی مشترک یعنی چه؟ (Concurrency در برابر Parallelism)

«آشپزخانه‌ی شلوغ» یعنی چند کار **در یک بازه‌ی زمانی** در جریان است. اما دو معنیِ کاملاً متفاوت دارد:

| در قصه | اسم فنی | یعنی |
|---|---|---|
| یک آشپز که سریع بینِ چند کار می‌پرد | **Concurrency** (هم‌روندی) | چند کار «در جریان»‌اند، ولی در هر لحظه فقط یکی واقعاً اجرا می‌شود |
| چند آشپز که واقعاً همزمان دست‌به‌کارند | **Parallelism** (موازی‌سازی) | چند کار در یک لحظه‌ی واقعی، روی چند هسته‌ی CPU، باهم اجرا می‌شوند |

مثالِ ساده روی یک هسته‌ی CPU:

```
Parallelism (دو هسته):     Concurrency (یک هسته، تعویضِ سریع):
هسته۱: ▓▓▓▓▓▓▓▓ کارِ الف     هسته: ▓▓░░▓▓░░▓▓░░  (الف، ب، الف، ب...)
هسته۲: ▓▓▓▓▓▓▓▓ کارِ ب
       (واقعاً همزمان)             (به نظر همزمان، ولی نوبتی)
```

جمله‌ی کلیدی: **Concurrency یعنی مدیریتِ چند کار، Parallelism یعنی اجرای واقعاً همزمانِ چند کار.** Concurrency بدونِ Parallelism هم ممکن است (یک آشپز، چند کار).

## صحنه‌ی ۲ — سوپِ دوبار نمک‌خورده (Race Condition)

این هسته‌ی همه‌چیز است. در قصه، دو آشپز همزمان چشیدند و هر دو نمک زدند. در کد، این عددِ مشترک است:

```python
counter = 0          # قابلمه‌ی مشترک

def add_one():
    global counter
    counter += 1     # «بچش، نمک بزن، بگذار سرِ جا»
```

آن خطِ بی‌گناهِ `counter += 1` در واقع **سه** کار است، نه یک کار:

```
۱. READ:   مقدارِ counter را بخوان        (می‌چشد)
۲. MODIFY: یکی به آن اضافه کن             (تصمیم می‌گیرد نمک بزند)
۳. WRITE:  نتیجه را برگردان توی counter   (نمک می‌ریزد)
```

به این می‌گویند **Read-Modify-Write**. حالا ببین وقتی دو نخ همزمان این را اجرا کنند، چه فاجعه‌ای می‌شود:

```
زمان    نخِ الف                  نخِ ب                   counter
────────────────────────────────────────────────────────────────
t1     READ counter (=5)                                   5
t2                              READ counter (=5)           5     ← ب هم ۵ را خواند!
t3     5 + 1 = 6                                            5
t4                              5 + 1 = 6                   5
t5     WRITE 6                                              6
t6                              WRITE 6                     6     ← به جای ۷ شد ۶!
────────────────────────────────────────────────────────────────
نتیجه: دو بار اضافه کردیم، ولی counter فقط یکی بالا رفت. یک به‌روزرسانی «گم» شد.
```

این **Race Condition** است: نتیجه به **ترتیبِ دقیقِ اجرای نخ‌ها** وابسته است، و آن ترتیب هر بار فرق می‌کند. بخشی از کد که نباید دو نخ همزمان واردش شوند (همان لحظه‌ی دست‌زدن به قابلمه) را **Critical Section** (ناحیه‌ی بحرانی) می‌گویند.

## صحنه‌ی ۳ — قاشقِ سخنگو (Lock / Mutex)

قاشقِ سخنگوی قصه دقیقاً همان **قفل** (Lock) یا **Mutex** است: یک شیء که فقط **یک نخ** می‌تواند هم‌زمان «بگیردش».

```python
import threading

counter = 0
lock = threading.Lock()      # قاشقِ سخنگو

def add_one():
    global counter
    with lock:               # «قاشق را برمی‌دارم» (هر کس دیگر صبر کند)
        counter += 1         # حالا تنها منم که به قابلمه دست می‌زنم
    # پایانِ with: «قاشق را می‌گذارم سرِ جا» — خودکار آزاد می‌شود
```

نکته‌های مهم:

- **`with lock:`** بهترین راه است، چون قفل را **خودکار** آزاد می‌کند — حتی اگر وسطِ کار خطا بدهد. اگر دستی `lock.acquire()` کنی و یادت برود `lock.release()` بزنی، قاشق برای همیشه دستت می‌ماند و بقیه گیر می‌کنند.
- **دانه‌بندیِ قفل (Lock Granularity):** یک قاشق برای کلِ آشپزخانه؟ یا یک قاشق برای هر قابلمه؟ قفلِ درشت (یکی برای همه) ساده ولی کند است (همه پشتِ هم صف می‌کشند). قفلِ ریز (یکی برای هر چیز) سریع‌تر ولی خطرناک‌تر است (راحت‌تر deadlock می‌شود).
- **عملیاتِ اتمی (Atomic Operation):** بعضی کارها از پایه «نشکستنی»‌اند — مثل کلِ `READ-MODIFY-WRITE` که یک‌جا و بدونِ وقفه انجام شود. اگر `counter += 1` خودش اتمی بود، اصلاً نیازی به قاشق نداشتیم.

## صحنه‌ی ۴ — گیرِ ابدی (Deadlock، Livelock، Starvation)

به یاد بیاور: الف نمکدان را گرفته منتظرِ فلفل، ب فلفل را گرفته منتظرِ نمک. این **Deadlock** است. تصویرش یک **حلقه‌ی انتظار** است:

```
       منتظرِ فلفل
   ┌──────────────────┐
   ▼                  │
 الف                  ب
   │                  ▲
   └──────────────────┘
       منتظرِ نمک

الف قفلِ نمک را دارد، منتظرِ فلفل  ──┐
ب   قفلِ فلفل را دارد، منتظرِ نمک   ──┘  حلقه بسته شد → هیچ‌کس تکان نمی‌خورد
```

برای اینکه deadlock اصلاً ممکن شود، هر **چهار** شرطِ زیر (شرایطِ Coffman) باید هم‌زمان برقرار باشند — شکستنِ هر کدام، deadlock را ناممکن می‌کند:

| شرط | در قصه |
|---|---|
| **Mutual Exclusion** | نمک و فلفل هر کدام فقط دستِ یک نفر |
| **Hold and Wait** | الف نمک را نگه داشته و منتظرِ فلفل است |
| **No Preemption** | کسی نمی‌تواند به‌زور نمک را از دستِ الف بگیرد |
| **Circular Wait** | الف منتظرِ ب، ب منتظرِ الف (حلقه) |

ساده‌ترین درمان، شکستنِ **Circular Wait** است: همان قانونِ آشپزِ دانا — **همه به یک ترتیب قفل بگیرند** (همیشه اول نمک، بعد فلفل). به این می‌گویند **Lock Ordering** (ترتیبِ ثابتِ قفل‌ها). اگر همه به یک ترتیب پیش بروند، حلقه هرگز بسته نمی‌شود.

دو بیماریِ خویشاوند هم هست:

- **Livelock:** دو آشپز در راهروی باریک به هم می‌رسند. الف می‌رود راست، ب هم راست؛ باز روبه‌رو می‌شوند. الف می‌رود چپ، ب هم چپ. هیچ‌کس گیر نکرده (هر دو دارند تکان می‌خورند!) ولی هیچ کارِ مفیدی هم پیش نمی‌رود. یعنی **فعالیت هست، پیشرفت نیست**.
- **Starvation (گرسنگی):** آشپزِ خجالتیِ گوشه‌ی آشپزخانه هر بار که می‌خواهد قاشق را بردارد، یک آشپزِ پرروتر زودتر برش می‌دارد. او گیر نکرده، ولی **هیچ‌وقت نوبتش نمی‌شود**. درمان: نوبت‌دهیِ عادلانه (مثلاً قدیمی‌ترها اولویت بگیرند).

## صحنه‌ی ۵ — چند آشپز یا یک قابلمه‌ی جدا؟ (Threads در برابر Processes و معمای GIL)

دو راه برای «بیشتر کردنِ دست‌ها» داریم:

| | چند نخ (Threads) | چند پروسه (Processes) |
|---|---|---|
| در قصه | چند آشپز، **یک قابلمه‌ی مشترک** | چند آشپز، **هر کدام قابلمه‌ی جدا** |
| حافظه | مشترک (سریعِ ارتباط، خطرِ race) | جدا/ایزوله (امن، ولی ارتباط گران‌تر) |
| اگر یکی crash کند | کلِ برنامه می‌افتد | فقط همان پروسه می‌افتد |
| ساختنش | سبک و ارزان | سنگین‌تر |

و اینجا یک تله‌ی معروفِ پایتون است: **GIL (Global Interpreter Lock)**.

> در CPython (پایتونِ استاندارد) یک قفلِ بزرگ به نام GIL هست که می‌گوید: **در هر لحظه فقط یک نخ می‌تواند بایت‌کدِ پایتون اجرا کند.** انگار کلِ آشپزخانه فقط **یک اجاقِ پایتون** دارد، هرچند چند آشپز داشته باشی.

نتیجه‌ی مهم و ضدِشهود:

- اگر کارت **محاسباتیِ سنگین** باشد (مثلِ خرد کردنِ گونیِ سیب‌زمینی = شمارشِ سنگینِ CPU)، چند نخ در پایتون **تندترش نمی‌کند** — چون فقط یک اجاقِ پایتون هست و نخ‌ها نوبتی روی همان یک اجاق می‌روند. گاهی حتی **کندتر** می‌شود (هزینه‌ی جابه‌جاییِ نوبت اضافه می‌شود).
- جوابِ کارِ محاسباتیِ سنگین در پایتون: **چند پروسه** (multiprocessing). هر پروسه اجاقِ پایتونِ خودش (GIL خودش) را دارد و واقعاً موازی می‌شوند.

> ⚠️ هشدارِ مهم درباره‌ی race: ممکن است فکر کنی «حالا که GIL هست و فقط یک نخ کار می‌کند، پس دیگر `counter += 1` مشکل ندارد!» — **اشتباه است.** GIL فقط تضمین می‌کند هر **بایت‌کد** تنها اجرا شود، ولی `counter += 1` چند بایت‌کد است (READ، MODIFY، WRITE) و مفسر می‌تواند **وسطِ همین‌ها** نخ را عوض کند. پس race حتی **با وجودِ GIL** هم رخ می‌دهد. به همین خاطر باز هم به قفل نیاز داری.

> نکته: از CPython 3.13 یک نسخه‌ی آزمایشیِ «بدونِ GIL» (free-threaded) هم هست، ولی هنوز پیش‌فرض نیست؛ فعلاً همان مدلِ GIL را مبنا بگیر.

## صحنه‌ی ۶ — کدام کار، کدام ابزار؟ (I/O-bound در برابر CPU-bound)

این مهم‌ترین تصمیمِ آشپزخانه است. اول بفهم کارت چه نوعی است:

| نوعِ کار | در قصه | یعنی | ابزارِ درست |
|---|---|---|---|
| **I/O-bound** | بیشترِ وقت **منتظر** است (جوشِ آب، گرم شدنِ فر، صبر برای پاسخِ سرور/دیسک/شبکه) | CPU بی‌کار است، فقط منتظریم | **async** یا **threads** |
| **CPU-bound** | بیشترِ وقت **کارِ سنگین** است (خرد کردنِ سیب‌زمینی، رمزنگاری، پردازشِ تصویر) | CPU جان می‌کند | **چند پروسه** (multiprocessing) |

قانونِ سرانگشتی:

```
کارت بیشتر «منتظر ماندن» است؟  →  async/await (یک آشپزِ باهوش)  یا threads
کارت بیشتر «محاسبه» است؟       →  multiprocessing (چند آشپزِ واقعی روی اجاق‌های جدا)
```

## صحنه‌ی ۷ — آشپزی که هیچ‌وقت بی‌کار نمی‌ماند (Async / Await)

این همان آشپزِ باهوشِ تنهاست. در دنیای کد، اسمِ این مدل **async/await** است و قطعاتش این‌هاست:

| در قصه | اسم فنی | یعنی |
|---|---|---|
| ذهنِ آشپز که تصمیم می‌گیرد بعدی چه کند | **Event Loop** (حلقه‌ی رویداد) | یک حلقه که مدام می‌گردد دنبالِ کاری که آماده‌ی ادامه است |
| یک کار که می‌تواند وسطش «مکث» کند و بعد ادامه بدهد | **Coroutine** (هم‌روال) | تابعی با `async def` که با `await` می‌تواند کنترل را موقتاً پس بدهد |
| «کتری را گذاشتم، تا جوش بیاید می‌روم سراغِ سبزی» | **Non-blocking I/O** | منتظرِ I/O نمی‌مانیم؛ کنترل را پس می‌دهیم تا کارِ دیگری انجام شود |

```python
import asyncio

async def boil_water():
    print("کتری روی گاز")
    await asyncio.sleep(2)        # «تا جوش بیاید» — اینجا کنترل را پس می‌دهد
    print("آب جوش آمد")

async def chop_veggies():
    print("شروعِ خرد کردن")
    await asyncio.sleep(1)
    print("سبزی آماده")

async def main():
    # هر دو را همزمان پیش ببر؛ آشپز بینشان می‌پرد
    await asyncio.gather(boil_water(), chop_veggies())

asyncio.run(main())
```

دو نکته‌ی طلایی:

- **`gather` = چند کار را با هم پیش ببر.** به جای اینکه اول کاملاً منتظرِ جوشِ آب بمانی بعد بروی سراغِ سبزی (پشتِ سرِ هم، کند)، با `gather` آشپز بینِ هر دو می‌پرد. اگر سه درخواستِ شبکه داری، با `gather` هر سه «همزمان» منتظر می‌مانند نه پشتِ هم.

```
بدونِ gather (پشتِ سرِ هم):   منتظرِ آب (۲ث) ──► منتظرِ سبزی (۱ث)  = ۳ ثانیه
با gather (هم‌زمان):          منتظرِ آب  (۲ث)
                              منتظرِ سبزی (۱ث) همپوشان           = ۲ ثانیه
```

- **قانونِ «حلقه را بند نیار» (Don't block the loop):** اگر آشپزِ باهوش وسطِ کار بنشیند و **واقعاً** ۱۰ دقیقه به آب زل بزند (یک کارِ سنگینِ بدونِ `await`)، همه‌ی کارهای دیگر هم می‌خوابند — چون فقط همین یک آشپز هست. در async هرگز کارِ سنگین یا انتظارِ مسدودکننده (مثلِ `time.sleep` یا کوئریِ دیتابیسِ غیرِ async) را داخلِ coroutine نگذار؛ وگرنه کلِ حلقه می‌ایستد.

## صحنه‌ی ۸ — ریلِ سفارش (Queue: ارتباط به‌جای حافظه‌ی مشترک)

به جای اینکه چند آشپز سرِ یک قابلمه (حافظه‌ی مشترک) دعوا کنند، **ریلِ سفارش** پیام را امن رد و بدل می‌کند. در کد این **Queue** است و خودش thread-safe است (نیازی به قفلِ دستی نداری):

```python
import queue, threading

orders = queue.Queue()           # ریلِ سفارش

def waiter():                    # گارسون: سفارش می‌گذارد
    for i in range(5):
        orders.put(f"سفارش {i}")

def cook():                      # آشپز: سفارش برمی‌دارد
    while True:
        order = orders.get()     # امن؛ اگر خالی بود صبر می‌کند
        print("پختن:", order)
        orders.task_done()
```

فلسفه‌ی thread safety در عمل، سه ستون دارد:

1. **به جای حافظه‌ی مشترک، پیام رد و بدل کن** (Queue) — «قابلمه را مشترک نکن، سفارش رد کن».
2. **داده‌ی تغییرناپذیر (Immutable)** — چیزی که هیچ‌کس نمی‌تواند عوضش کند، اصلاً race ندارد.
3. **ساختارهای thread-safe** — مثلِ `queue.Queue` که قفل‌کاری را داخلِ خودش انجام داده.

## صحنه‌ی ۹ — اشتباه‌های رایجِ آشپزها (Common Bugs)

سه اشتباهی که تقریباً همه مرتکب می‌شوند:

- **`await` فراموش‌شده:** اگر بنویسی `boil_water()` به جای `await boil_water()`، آشپز کتری را می‌گذارد ولی **منتظرِ جوشش نمی‌ماند** و رد می‌شود — کار اصلاً انجام نمی‌شود (یک coroutine ساخته‌ای که هیچ‌وقت اجرا نشده). علامتش معمولاً هشدارِ `coroutine was never awaited` است.
- **مقدارِ پیش‌فرضِ تغییرپذیرِ مشترک (Shared Mutable Default):** نوشتنِ `def add(item, basket=[])` تله است — آن لیستِ `[]` **یک‌بار** ساخته می‌شود و بینِ همه‌ی فراخوانی‌ها **مشترک** می‌ماند (مثلِ قابلمه‌ای که فکر می‌کردی هر بار نو است ولی همان قبلی است). درست: `basket=None` و داخلِ تابع `if basket is None: basket = []`.
- **قفلی که آزاد نمی‌شود:** اگر `acquire()` کنی و قبلِ `release()` خطا رخ دهد، قاشق برای همیشه دستت می‌ماند و بقیه گیر می‌کنند. درمان: همیشه از **`with lock:`** استفاده کن تا آزادسازی خودکار و تضمین‌شده باشد.

```
   │
   └──「محاسبه‌ی سنگین» (CPU-bound) ──► multiprocessing
            (چون GIL، threadها در پایتون CPU را موازی نمی‌کنند)

و هر جا حافظه‌ی مشترک داری:
   counter += 1 از دو نخ  ──►  Race Condition (گم شدنِ به‌روزرسانی)
        │ درمان
        ▼
   Lock / Mutex (با `with lock:`)  ──►  مراقبِ Deadlock باش
        │ درمان deadlock
        ▼
   Lock Ordering (همه به یک ترتیب)
        │ امن‌ترین راه از اول
        ▼
   Queue (پیام به جای حافظه‌ی مشترک)
```
