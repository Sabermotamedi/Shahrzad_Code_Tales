# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی خانه‌ای که سام ساخت 🔍🏠

> همان قصه‌ی سام و عموجان، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعیِ مهندسی نرم‌افزار *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — نقشه‌ی تمیز یعنی کدِ تمیز (Clean Code)

در دنیای واقعی، «خانه» همان **برنامه‌ی** توست، «نقشه» همان **کد**، و «اتاق‌ها» همان **توابع** (functions). وقتی عمو گفت «اتاق‌ها را کوچک بساز و اسم درست بگذار»، دقیقاً داشت از **Clean Code** حرف می‌زد.

| در قصه | در دنیای واقعی |
|---|---|
| اسم درست روی هر در | نام‌گذاری گویا (Naming) |
| اتاق‌های کوچک و مشخص | توابع کوتاه که **یک کار** انجام می‌دهند |
| توضیح کنار نقشه فقط وقتی لازم | کامنت برای **چرا**، نه **چه** |

**نام‌گذاری (Naming):** اسم باید بگوید چیست، بدون اینکه مجبور شوی داخل کد را بخوانی.

```python
# ❌ بد — اتاق ۱ و اتاق ۲
def calc(d, t):
    return d * t * 0.05

# ✅ خوب — اسم خودش توضیح است
def calculate_late_fee(days_overdue, daily_rate):
    return days_overdue * daily_rate * PENALTY_MULTIPLIER
```

**توابع کوچک (Small Functions):** هر تابع فقط **یک کار**. اگر مجبور شدی برای توضیحِ کارِ تابع از کلمه‌ی «و» استفاده کنی، احتمالاً دو تابع است که در هم چپانده شده.

**کامنت (Comments):** کامنت خوب می‌گوید **چرا**، نه **چه**. اگر کد خودش گویا باشد، نیازی به کامنتِ «چه» نداری.

```python
# ❌ کامنت بی‌فایده — کد خودش این را می‌گوید
i += 1  # i را یکی زیاد کن

# ✅ کامنت ارزشمند — «چرا» را توضیح می‌دهد
# بانک مرکزی نرخ را فقط روزهای کاری اعمال می‌کند، پس آخر هفته را رد می‌کنیم
if not is_weekend(date):
    apply_interest()
```

## صحنه‌ی ۲ — نوسازی بدون خراب کردن (Refactoring)

وقتی عمو یک اتاق را مرتب کرد بدون اینکه **رفتار** خانه عوض شود، اسم کارش **Refactoring** بود: تغییر *ساختار داخلی* کد بدون تغییر *کاری که از بیرون انجام می‌دهد*.

آن «گوشه‌های بدقواره» اسم فنی دارند: **Code Smell** (بوی بد کد). چندتای معروف:

| بوی بد (Code Smell) | در قصه |
|---|---|
| تابع طولانی (Long Method) | اتاقی که هزار کار توش انجام می‌شود |
| کد تکراری (Duplicated Code) | همان کمد، عیناً در سه اتاق |
| نام‌های مبهم (Mysterious Name) | درِ بی‌اسم |
| لیست پارامتر طولانی (Long Parameter List) | کلیدی با ۹ تا دندانه |
| کلاس خدا (God Class) | یک اتاق که آشپزخانه و حمام و انباری با هم است |

**مرحله‌های امنِ refactoring** (دقیقاً مثل عمو که اتاق‌به‌اتاق پیش رفت):

```
١. اول یک تست بنویس که رفتار فعلی را قفل کند (تور ایمنی)
٢. یک تغییر کوچک بده (مثلاً یک تکه را به تابع جدا ببر — Extract Method)
٣. تست‌ها را اجرا کن → سبز؟ ادامه بده. قرمز؟ برگرد عقب.
٤. تکرار کن
```

> نکته‌ی مصاحبه: refactoring یعنی **رفتار بیرونی ثابت می‌ماند**. اگر رفتار عوض شد، دیگر refactoring نیست — feature یا bugfix است.

## صحنه‌ی ۳ — طرح‌های آماده‌ی استادکارها (Design Patterns)

دفترچه‌ی عمو همان **Design Patterns** است: راه‌حل‌های اثبات‌شده برای مشکلات تکرارشونده. این الگوها در سه **دسته** جای می‌گیرند:

| دسته | یعنی | الگوهای ما |
|---|---|---|
| **Creational** (ساختنی) | چطور یک شیء را **بسازیم** | Singleton، Factory |
| **Structural** (ساختاری) | چطور اشیا را **به هم وصل کنیم** | Decorator، Adapter |
| **Behavioral** (رفتاری) | چطور اشیا با هم **حرف بزنند** | Strategy، Observer |

**Singleton — فقط یک کلید برق اصلی.** کل خانه فقط یک کلید برق اصلی دارد؛ هر جای خانه که بپرسی «کلید اصلی کجاست؟»، همان یکی را تحویل می‌گیری.

```python
class Database:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
# هر بار Database() همان یک شیء را برمی‌گرداند
```

**Factory — کارگاهِ ساختِ در.** به‌جای اینکه خودت در را با چکش و میخ بسازی، به کارگاه می‌گویی «یک درِ چوبی می‌خواهم» و کارگاه برایت می‌سازد. تو نمی‌دانی *چطور* ساخته شد.

```python
def door_factory(kind: str) -> Door:
    if kind == "wooden": return WoodenDoor()
    if kind == "steel":  return SteelDoor()
    raise ValueError(kind)
```

**Strategy — روشِ گرم کردن خانه، قابل‌تعویض.** خانه یک «جای شوفاژ» دارد؛ امروز گاز، فردا برق، پس‌فردا خورشیدی — بدون دست‌زدن به خود خانه، فقط *روش* را عوض می‌کنی.

```python
def checkout(cart, payment_strategy):
    return payment_strategy.pay(cart.total)   # کارت، کیف‌پول، رمزارز...
```

**Observer — تابلوی اعلانات.** (همان نانواییِ صحنه‌ی ۶!) وقتی چیزی تغییر کرد، همه‌ی علاقه‌مندانِ ثبت‌نام‌شده **خبردار می‌شوند**.

**Decorator — کاغذدیواری روی دیوار.** بدون عوض‌کردن خود دیوار، رویش یک لایه‌ی تزئینی می‌چسبانی؛ می‌توانی چند لایه روی هم بزنی. در کد یعنی **اضافه‌کردن قابلیت به یک شیء، بدون تغییر کلاسش**.

```python
def with_logging(func):
    def wrapper(*a, **k):
        print(f"calling {func.__name__}")
        return func(*a, **k)
    return wrapper
```

**Adapter — مبدلِ دوشاخه.** پریز خانه سه‌خانه است ولی دوشاخه‌ی مهمان دو‌خانه؛ یک **مبدل** بینشان می‌گذاری تا دو رابطِ ناسازگار با هم کار کنند.

```python
class LegacyPrinterAdapter:
    def __init__(self, legacy): self.legacy = legacy
    def print(self, text):              # رابطِ جدیدی که کد ما انتظار دارد
        self.legacy.do_old_print(text)  # ترجمه به رابط قدیمی
```

## صحنه‌ی ۴ — قانون‌های بنّاییِ سالم (SOLID)

وقتی عمو می‌گفت «این‌طوری نساز چون فردا اذیت می‌شوی»، داشت **پنج قانونِ طلاییِ SOLID** را یاد سام می‌داد. این‌ها قانون‌هایی هستند که کد را **قابل‌تغییر و قابل‌نگهداری** می‌کنند:

| حرف | نام کامل | در قصه |
|---|---|---|
| **S** | Single Responsibility | هر اتاق فقط **یک کاربری** دارد |
| **O** | Open/Closed | برای افزودن اتاق نو، دیوارهای قدیمی را **نشکن** |
| **L** | Liskov Substitution | هر «درِ» جدید باید واقعاً مثل یک در رفتار کند |
| **I** | Interface Segregation | به مهمان فقط کلیدِ درِ ورودی را بده، نه دسته‌کلیدِ کل خانه |
| **D** | Dependency Inversion | خانه به «منبع گرما» وابسته باشد، نه مستقیم به «شوفاژ گازیِ مدل ۱۲۳۴» |

جزئیات و مثال کد هر پنج‌تا را در سطح ۳ می‌بینیم. اینجا کافی است حس کنی: SOLID یعنی **تغییر فردا را امروز ارزان کن**.

## صحنه‌ی ۵ — یک خانه‌ی بزرگ یا یک دهکده؟ (Monolith vs Microservices)

«خانه‌ی بزرگ» همان **Monolith** است: همه‌ی برنامه (کاربر، سفارش، پرداخت، انبار) در **یک پروژه و یک فرایندِ واحد**. «دهکده» همان **Microservices**: هر قابلیت یک سرویسِ کوچکِ مستقل با دیتابیسِ خودش، که از طریق شبکه با بقیه حرف می‌زند.

این یک **انتخاب با هزینه و فایده** است، نه «کدام بهتر است»:

| | Monolith (خانه‌ی بزرگ) | Microservices (دهکده) |
|---|---|---|
| شروع و توسعه | ساده و سریع | پیچیده از روز اول |
| استقرار (Deploy) | یک‌جا، همه با هم | هر سرویس جداگانه |
| مقیاس‌پذیری | کل خانه را باید بزرگ کنی | فقط آشپزخانه‌ی شلوغ را بزرگ کن |
| خرابی | یک مشکل، کل خانه می‌خوابد | خرابیِ یک سرویس، بقیه سرِ پا |
| تراکنش/داده | راحت (یک دیتابیس) | سخت (داده پخش است) |
| ارتباط بین اجزا | فراخوانی تابع (ارزان) | شبکه (کند، ممکن است شکست بخورد) |

> نکته‌ی مصاحبه: تله‌ی کلاسیک، **«دیتابیس مشترک» بین چند microservice** است. آن‌وقت دیگر سرویس‌ها واقعاً مستقل نیستند؛ یک distributed monolith ساخته‌ای که بدترین‌های هر دو دنیا را دارد. اکثر تیم‌ها بهتر است با monolith شروع کنند و فقط وقتی واقعاً لازم شد، تکه‌تکه جدا کنند.

## صحنه‌ی ۶ — تابلوی اعلانات (Event-Driven Architecture)

نانوا که روی تابلو می‌نوشت «نان تازه آماده شد!» و سراغ کارش می‌رفت، **معماری رویدادمحور** (Event-Driven Architecture) را اجرا می‌کرد.

- نانوا = **Producer** (تولیدکننده‌ی رویداد)
- تابلوی میدان = **Message Broker** (مثل Kafka یا RabbitMQ)
- قنادی و چای‌خانه = **Consumer**‌ها (مصرف‌کننده)
- «نان تازه آماده شد» = یک **Event**

نکته‌ی کلیدی: نانوا اصلاً **نمی‌داند** چه کسی خبرش را می‌خواند. به این می‌گویند **decoupling** (جداسازی). فردا یک بستنی‌فروشیِ جدید هم می‌تواند خبرها را بخواند، بدون اینکه نانوا حتی متوجه شود.

```
[نانوایی] ──"نان آماده شد"──▶ [ تابلو / Broker ] ──▶ [قنادی]
                                                  └──▶ [چای‌خانه]
                                                  └──▶ [خانه‌ی سام]
```

این برعکسِ مدل قدیمی است که نانوا باید **مستقیم به همه زنگ می‌زد** (request/response و coupling شدید).

## صحنه‌ی ۷ — محله‌ها با زبان خودشان (Domain-Driven Design)

تقسیم دهکده به محله‌ها، با کلمه‌هایی که در هر محله معنی خودش را دارد، دقیقاً **Domain-Driven Design (DDD)** است:

| در قصه | اصطلاح DDD | یعنی |
|---|---|---|
| محله (با مرز مشخص) | **Bounded Context** | مرزی که داخلش یک مدل و یک زبان معنا دارد |
| «تور» معنیِ ثابت در هر محله | **Ubiquitous Language** | یک کلمه، یک معنی، برای همه‌ی تیم و کد |
| یک اهالیِ مشخص با شناسنامه | **Entity** | شیئی که با **شناسه‌اش** هویت دارد (نه فقط مقدارش) |
| یک خانه با همه‌ی ساکنانش به‌عنوان یک واحد | **Aggregate** | مجموعه‌ای از اشیا که با هم یک واحدِ یکپارچه‌اند و یک «درِ ورودی» (Aggregate Root) دارند |

مثال «تور»: در محله‌ی فروش، `Order` یعنی سفارش با وضعیت پرداخت؛ در محله‌ی انبار، همان سفارش فقط یعنی «این چیزها را بردار و بسته‌بندی کن». **یک کلمه، دو مدلِ متفاوت در دو Bounded Context.** اشتباهِ بزرگ این است که بخواهی یک مدلِ غول‌پیکر برای همه‌ی محله‌ها بسازی.

## صحنه‌ی ۸ — بازرسِ ساختمان (Testing)

بازرس همان **Testing** است، و کارهای سه‌گانه‌اش دقیقاً سه نوع تست:

| در قصه | نوع تست | چه چیزی را چک می‌کند |
|---|---|---|
| باز کردن تک‌تکِ شیرها | **Unit Test** | یک تابع/کلاس به‌تنهایی، سریع و جدا |
| آشپزخانه + حمام با هم | **Integration Test** | چند جزء که با هم کار می‌کنند (مثلاً کد + دیتابیس) |
| راه‌رفتن از در تا پشت‌بام | **End-to-End (E2E)** | کل سیستم از دید کاربر واقعی |

**هرم تست (Test Pyramid):** تستِ پایین‌تر، ارزان‌تر و سریع‌تر — پس بیشترش را بنویس:

```
        /\
       /E2E\        کم، کند، گران
      /------\
     /  Integ \     متوسط
    /----------\
   / Unit Tests \   زیاد، سریع، ارزان  ← پایه‌ی هرم
  /--------------\
```

**Mocking — ماکتِ آب.** بازرس برای امتحانِ سینک، لازم نیست واقعاً به شبکه‌ی آبِ شهر وصل شود؛ یک شیلنگِ ماکت کافی است. در کد هم وقتی یک unit را تست می‌کنی، وابستگی‌های کند یا بیرونی (دیتابیس، API، ساعت) را با یک **mock** (بدلِ کنترل‌شده) جایگزین می‌کنی تا تست سریع و قطعی بماند.

**TDD — اول معیار، بعد ساختن.** آن عادتِ بازرس که *قبل* از ساخت می‌گفت «اتاق باید این شکلی شود»، همان **Test-Driven Development** است؛ چرخه‌ی سه‌مرحله‌ایِ آن:

```
🔴 RED    — اول یک تستِ شکست‌خورده بنویس (هنوز کدی نیست)
🟢 GREEN  — کمترین کد را بنویس تا تست سبز شود
🔵 REFACTOR — حالا با خیال راحت کد را تمیز کن (تست تور ایمنی است)
        ↺ تکرار
```

```python
# 🔴 RED — تست اول
def test_add():
    assert add(2, 3) == 5      # add هنوز وجود ندارد → شکست

# 🟢 GREEN — کمترین کد
def add(a, b):
    return a + b               # تست سبز شد

# 🔵 REFACTOR — اگر لازم بود تمیز کن (اینجا لازم نیست)
```

## 🗺️ کل سفر در یک نگاه

```
نقشه‌ی تمیز (Clean Code)
   │  اتاق‌های کوچک، اسم‌های درست
   ▼
نوسازی اتاق‌به‌اتاق (Refactoring)  ←── بوی بد را حس کن، آرام مرتب کن
   │
   ▼
طرح‌های آماده (Design Patterns)  ←── Singleton/Factory · Decorator/Adapter · Strategy/Observer
   │
   ▼
قانون‌های بنّایی (SOLID)  ←── تغییرِ فردا را امروز ارزان کن
   │
   ▼
خانه‌ی بزرگ یا دهکده؟ (Monolith vs Microservices)  ←── انتخابِ هزینه/فایده
   │
   ▼
تابلوی اعلانات (Event-Driven)  ←── Producer → Broker → Consumers
   │
   ▼
محله‌ها با زبان خود (DDD)  ←── Bounded Context · Ubiquitous Language · Entity · Aggregate
   │
   ▼
بازرسِ ساختمان (Testing)  ←── Unit → Integration → E2E · Pyramid · Mock · TDD
   │
   ▼
خانه‌ای محکم، تمیز و قابل‌اعتماد 🏠✨
```
