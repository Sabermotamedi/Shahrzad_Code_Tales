# سطح ۳ — توضیح فنی: System Design ⚙️

> دیگر قصه‌ای در کار نیست. این سند، **playbookِ مصاحبه‌ی طراحی سیستم** است — همان روش، عددها، و trade-offهایی که یک مهندس باید آماده داشته باشد. این فصل **capstone** است: همه‌ی اجزای فصل‌های قبل را کنار هم می‌چیند.

---

## ۱. روشِ مصاحبه‌ی System Design (گام‌به‌گام)

مصاحبه‌ی طراحی سیستم، آزمونِ حفظیات نیست؛ آزمونِ **روش** است. مصاحبه‌گر می‌خواهد ببیند آیا ساختاریافته فکر می‌کنی. این ۶ گامِ استاندارد را همیشه به همین ترتیب طی کن:

```
① Clarify Requirements   → چه می‌سازیم؟ (functional + non-functional)
② Estimation             → چند کاربر؟ چند QPS؟ چقدر storage؟  (back-of-envelope)
③ API Design             → endpointها چیست؟ ورودی/خروجی؟
④ High-Level Design       → جعبه‌ها و فلش‌ها (LB, server, DB, cache, queue...)
⑤ Deep Dive               → یکی-دو جزء را عمیق بشکاف (مصاحبه‌گر هدایت می‌کند)
⑥ Bottlenecks & Scale     → کجا می‌شکند؟ چطور رفعش می‌کنی؟ trade-offها
```

### گام ① — شفاف‌سازیِ نیازمندی‌ها (Requirements)

هیچ‌وقت بدونِ سؤال شروع نکن. دو دسته نیازمندی هست:

- **Functional (کارکردی)** — سیستم *چه کاری* می‌کند؟ مثال برای سرویس عکس: «کاربر عکس آپلود می‌کند»، «فید دوستان را می‌بیند»، «لایک می‌کند». محدوده (scope) را با مصاحبه‌گر توافق کن — چه چیزهایی **بیرونِ** محدوده است؟
- **Non-functional (غیرکارکردی)** — سیستم *چطور* باید باشد؟ این‌ها معماری را تعیین می‌کنند:
  - **Scale**: چند کاربر؟ چند درخواست در ثانیه؟
  - **Availability**: چند ۹ (مثلاً 99.9%)؟
  - **Latency**: جوابِ سریع چقدر مهم است؟
  - **Consistency**: داده باید *همین لحظه* یکسان باشد یا «دیر یا زود» کافی است؟
  - **Durability**: گم‌شدنِ داده چقدر فاجعه است؟
  - **Read/Write ratio**: خواندن بیشتر است یا نوشتن؟

> نکته‌ی مصاحبه: نسبتِ read/write مهم‌ترین عددی است که جهتِ کلِ طراحی را عوض می‌کند. سیستمِ read-heavy (عکس، خبر) به replica و cache زیاد نیاز دارد؛ سیستمِ write-heavy (لاگ، متریک) به sharding و queue.

### گام ② تا ⑥

گام ② (estimation) را در بخش بعدی کامل می‌کنیم. گام ③ (API) و ④ (high-level) و ⑤ (deep dive) را در دو طراحیِ نمونه‌ی بخش ۵ می‌بینی. گام ⑥ (trade-offها) در بخش ۴.

> نکته‌ی مصاحبه: زمان را مدیریت کن. در یک مصاحبه‌ی ۴۵ دقیقه‌ای: ~۵ دقیقه requirements، ~۵ دقیقه estimation، ~۵ دقیقه API، ~۱۰ دقیقه high-level، ~۱۵ دقیقه deep dive و bottleneck. زیاد در estimation غرق نشو.

## ۲. تخمینِ Back-of-the-Envelope (با مثالِ کامل)

هدف این نیست که دقیق باشی؛ هدف این است که **مرتبه‌ی بزرگی** (order of magnitude) را درست حدس بزنی تا بفهمی به چند سرور، چقدر storage و چه پهنای باندی نیاز داری.

### عددهایی که باید از بر باشی (Powers of two & latency)

| واحد | تقریب |
|---|---|
| 1 KB | 10³ بایت |
| 1 MB | 10⁶ بایت |
| 1 GB | 10⁹ بایت |
| 1 TB | 10¹² بایت |
| یک روز | ~86,400 ثانیه ≈ **10⁵ ثانیه** |
| یک ماه | ~2.5 میلیون ثانیه ≈ 2.5 × 10⁶ |

**عددهای latency (ترتیبِ Jeff Dean) — برای deep dive حیاتی:**

| عمل | زمانِ تقریبی |
|---|---|
| L1 cache reference | ~1 ns |
| دسترسی به RAM (memory reference) | ~100 ns |
| فشرده‌سازیِ 1KB | ~10 µs |
| خواندنِ 1MB از RAM | ~10 µs |
| round-trip داخلِ یک دیتاسنتر | ~0.5 ms |
| خواندنِ 1MB از SSD | ~1 ms |
| disk seek (دیسکِ مکانیکی) | ~10 ms |
| خواندنِ 1MB از HDD | ~30 ms |
| round-trip بین‌قاره‌ای (مثلاً CA↔Netherlands) | ~150 ms |

> نکته‌ی مصاحبه: درسِ این جدول — RAM حدود ۱۰۰٬۰۰۰ برابر سریع‌تر از دیسک است، و یک سفرِ بین‌قاره‌ای از هزاران عملِ محلی گران‌تر است. به‌خاطر همین cache و CDN وجود دارند.

**ظرفیتِ تقریبیِ یک گره (برای حسِ مقیاس):**

| منبع | ظرفیتِ مرتبه‌ای |
|---|---|
| یک app server | ~چند هزار QPS (بسته به کار) |
| یک گره‌ی SQL (نوشتن) | ~صدها تا چند هزار write/s |
| یک گره‌ی SQL (خواندن، با index) | ~چند هزار تا ده‌ها هزار read/s |
| یک گره‌ی Redis | ~صدها هزار op/s (در RAM) |
| یک سرور | ده‌ها GB تا چند صد GB RAM، چند TB دیسک |

### مثالِ کامل: تخمین برای «شهر عکس گربه‌ها»

**فرض‌ها (با مصاحبه‌گر توافق کن):**
- DAU (Daily Active Users) = ۱۰ میلیون = 10⁷
- هر کاربر روزی ۲ عکس آپلود می‌کند (write) و ۲۰۰ عکس می‌بیند (read) → نسبت read:write ≈ **100:1**
- هر عکس به‌طور میانگین ۳۰۰ KB است (بعد از فشرده‌سازی)

**① محاسبه‌ی QPS (Queries Per Second):**

```
نوشتن در روز  = 10⁷ کاربر × 2 عکس       = 2 × 10⁷ آپلود/روز
write QPS     = 2 × 10⁷ ÷ 10⁵ ثانیه     ≈ 200 write/s
peak write    ≈ 200 × 2 (ضریبِ پیک)      ≈ 400 write/s

خواندن در روز = 10⁷ × 200                 = 2 × 10⁹ view/روز
read QPS      = 2 × 10⁹ ÷ 10⁵            ≈ 20,000 read/s
peak read     ≈ 20,000 × 2               ≈ 40,000 read/s
```

→ درس: ~۴۰هزار read/s یعنی **حتماً** به read replicaها و cache و CDN نیاز داریم؛ یک دیتابیس کافی نیست.

**② محاسبه‌ی Storage (سالانه):**

```
عکسِ تازه در روز = 2 × 10⁷ عکس
حجمِ روزانه      = 2 × 10⁷ × 300 KB     = 6 × 10⁹ KB = 6 TB/روز
حجمِ سالانه      = 6 TB × 365            ≈ 2.2 PB/سال  (بدون احتسابِ replicaها)
با replication (×3)                      ≈ 6.6 PB/سال
```

→ درس: petabyteها عکس یعنی **Object Storage** (مثل S3)، نه دیتابیس. metadataـشان سبک است: `2×10⁷ رکورد/روز × ~1KB ≈ 20 GB در روز` (حدود ۳۰۰ برابر کوچک‌تر از خودِ عکس‌ها) → ~۷TB در سال، که ابتدا در یک DB جا می‌گیرد و با رشد، شارد می‌شود.

**③ محاسبه‌ی Bandwidth (پهنای باند):**

```
دانلودِ پیک  = 40,000 read/s × 300 KB    = 12 × 10⁶ KB/s = 12 GB/s
```

→ درس: ۱۲ گیگابایت بر ثانیه را هیچ سرورِ تنهایی نمی‌دهد → بارِ media باید روی **CDN** بیفتد، نه origin.

> نکته‌ی مصاحبه: همیشه با عددهای گرد کار کن و واحدها را همراه ببر. اشتباهِ یک‌میلیون‌برابری (KB با GB) معمول است. «۱۰⁵ ثانیه در روز» را حفظ کن؛ کلِ محاسبه‌ی QPS رویش سوار است.

## ۴. Trade-offهای کلیدی

طراحیِ خوب یعنی انتخابِ آگاهانه بینِ گزینه‌ها. مصاحبه‌گر همیشه می‌پرسد «چرا این و نه آن؟».

### SQL در برابر NoSQL

| | SQL (رابطه‌ای) | NoSQL |
|---|---|---|
| ساختار | schema ثابت، جدول‌ها و JOIN | schema منعطف (document/key-value/wide-column/graph) |
| سازگاری | ACID قوی | معمولاً BASE / eventual |
| مقیاس | عمودی (scale-up) راحت‌تر؛ افقی سخت‌تر | برای scale-out افقی ساخته شده |
| کِی؟ | رابطه‌ها و تراکنشِ مهم (پرداخت، سفارش) | حجمِ عظیم، schema متغیر، نوشتنِ زیاد (لاگ، فید، تایم‌لاین) |

> نکته‌ی مصاحبه: جوابِ تنبل «NoSQL چون مقیاس‌پذیر است» را نده. SQL مدرن (Postgres) هم با replica و sharding خیلی دور می‌رود. انتخاب را به **شکلِ داده و نیازِ سازگاری** گره بزن.

### CAP — سازگاری در برابر دسترس‌پذیری (مرورِ فصل ۸)

در یک سیستمِ توزیع‌شده، وقتی شبکه پارتیشن می‌شود (P که اجتناب‌ناپذیر است)، باید بینِ این دو یکی را انتخاب کنی:

- **CP** (Consistency): جوابِ غلط نده؛ اگر مطمئن نیستی، خطا بده. → بانک، موجودیِ انبار.
- **AP** (Availability): همیشه جواب بده، حتی اگر کمی کهنه باشد. → فید، لایک، تعدادِ بازدید.

> نکته‌ی مصاحبه: CAP فقط هنگامِ **partition** یک انتخابِ دو-گزینه‌ای است؛ در حالتِ عادی هم C و هم A را داری. سؤالِ واقعی: «در لحظه‌ی قطعیِ شبکه، کدام را قربانی می‌کنی؟»

### Latency در برابر Throughput

- **Latency** = یک درخواست چقدر طول می‌کشد (تجربه‌ی تک‌کاربر).
- **Throughput** = چند درخواست در ثانیه از کل سیستم می‌گذرد.
- این دو همیشه هم‌جهت نیستند: batching، throughput را بالا می‌برد ولی latency را هم بالا می‌برد. در مصاحبه بگو کدام را بهینه می‌کنی و چرا.

### Sync در برابر Async

| | Synchronous | Asynchronous (با Queue) |
|---|---|---|
| کاربر منتظر می‌ماند؟ | بله، تا اتمامِ کار | نه، فوری جواب می‌گیرد |
| کِی؟ | کاربر به نتیجه نیاز دارد (لاگین، پرداخت) | کارِ کند که نتیجه‌اش فوری لازم نیست (thumbnail، email، fan-out) |
| هزینه | ساده | پیچیده‌تر (queue، worker، at-least-once، idempotency) |

## ۵. دو طراحیِ نمونه‌ی کامل

### نمونه‌ی ۱ — کوتاه‌کننده‌ی لینک (URL Shortener) — کلاسیکِ مصاحبه

**① Requirements:**
- Functional: `long URL → short URL`؛ رفتن روی short → redirect به long؛ alias دلخواه (اختیاری)؛ انقضا (اختیاری).
- Non-functional: **به‌شدت read-heavy** (redirect خیلی بیشتر از create)؛ latency پایین؛ short URL یکتا و کوتاه.

**② Estimation:**
```
فرض: 100M URL تازه در ماه → 100M ÷ 2.5×10⁶ s ≈ 40 write/s
read:write ≈ 100:1 → ~4,000 read/s (redirect)
storage: 100M/ماه × 60 ماه (۵ سال) = 6×10⁹ رکورد × ~500 بایت ≈ ~3 TB → یک DB کافی است
```

**③ API:**
```
POST /shorten      { "url": "...", "alias?": "...", "ttl?": ... } → 201 { "short": "abc123" }
GET  /{short}      → 301/302 Location: <long URL>
```

**④ کلیدِ طراحی — چطور short code بسازیم؟**

| روش | چطور | مزیت / عیب |
|---|---|---|
| **Base62 از شناسه‌ی auto-increment** | ID عددیِ ۱،۲،۳... دیتابیس را به `[a-zA-Z0-9]` (۶۲ نماد) تبدیل کن | کوتاه و یکتا و تضمینی؛ ولی قابل‌حدس و ترتیبی (می‌شود با range encoding پنهان کرد) |
| **Hash (مثل MD5) + برش** | hash از long URL، ۶-۷ کاراکترِ اولش را بردار | بدونِ شمارنده‌ی مرکزی؛ ولی خطرِ **collision** → باید چک کنی و در صورتِ تصادم دوباره بزنی |
| **کلیدِ تصادفی + چک یکتایی** | یک رشته‌ی random بزن، اگر در DB بود دوباره | ساده؛ با پر شدنِ فضا کند می‌شود |

> ۶ کاراکترِ Base62 = ۶۲⁶ ≈ ۵۶ میلیارد ترکیب — برای بیشترِ سیستم‌ها بیش از کافی.

**⑤ Redirect: کدِ `301` یا `302`؟**
- `301` (دائمی): مرورگر/CDN آن را **کش** می‌کند → دفعه‌ی بعد اصلاً به سرورِ تو نمی‌آید. سریع‌تر، ولی **آمارِ کلیک را از دست می‌دهی** و نمی‌توانی مقصد را عوض کنی.
- `302` (موقت): هر بار به سرور برمی‌گردد → آمار و کنترل داری، ولی بارِ سرور بیشتر.
- انتخابِ معمول برای سرویسِ تجاری: **`302`** (برای آنالیتیکس)، با cache.

**⑥ مقیاس و bottleneck:**
- redirectـها read-heavyـاند → **hot URLها را در Redis کش کن** (`short → long`). cache hit یعنی redirect در حدِ میکروثانیه.
- DB می‌تواند key-value باشد (NoSQL برازنده است: کلید = short، مقدار = long).
- جلوی سرویس یک CDN/edge بگذار تا redirectـهای داغ حتی به origin نرسند.

```
[کاربر] → [CDN] → [LB] → [App] → [Redis cache] → [(miss) DB/KV store]
                                       │
                              (hit → 302 فوری)
```

### نمونه‌ی ۲ — سرویسِ اشتراکِ عکس (همان «شهر عکس گربه‌ها»)

**① Requirements:**
- Functional: آپلودِ عکس، دیدنِ فیدِ دوستان، follow/unfollow، لایک.
- Non-functional: read-heavy (نسبت ~100:1 که در بخش ۲ حساب کردیم)، latency پایینِ فید، durability بالای عکس‌ها، availability بالا (فید می‌تواند eventual باشد).

**② Estimation:** همان بخش ۲ — ~40k read/s پیک، ~2.2 PB عکس در سال، ~12 GB/s پهنای باند.

**③ API:**
```
POST /photos        (multipart عکس + caption)      → 201 { photoId, url }
GET  /feed?cursor=  (فیدِ کاربرِ لاگین‌شده)          → 200 { items[], nextCursor }
POST /photos/{id}/like                              → 200
POST /users/{id}/follow                             → 200
```

**④ ذخیره‌ی عکس — جداسازیِ blob از metadata:**
- خودِ فایلِ عکس → **Object Storage (S3)**؛ پخش از **CDN**.
- در DB فقط metadata: `photoId, userId, s3_url, caption, createdAt` (~1KB).

**⑤ deep dive — تولیدِ فید: Push در برابر Pull (مهم‌ترین بخش!)**

این کلاسیک‌ترین trade-offِ سیستم‌های اجتماعی است؛ به آن **fan-out** هم می‌گویند:

| رویکرد | چطور کار می‌کند | خوب برای | مشکل |
|---|---|---|---|
| **Pull (fan-out on read)** | فیدِ کاربر **هنگامِ خواندن** ساخته می‌شود: عکس‌های آخرِ همه‌ی کسانی که follow کرده، جمع و مرتب می‌شود | کاربرانی که خیلی follow دارند؛ نوشتن ارزان | خواندن کند و گران (هر بار محاسبه) |
| **Push (fan-out on write)** | لحظه‌ای که کسی عکس می‌گذارد، در **فیدِ همه‌ی follwerهایش** نوشته می‌شود (هر کاربر یک فیدِ آماده در Redis دارد) | خواندنِ فوری و ارزان (فید آماده است) | نوشتن گران؛ **مشکلِ celebrity**: کاربری با ۵۰M follower → ۵۰M نوشتن در یک آپلود! |
| **Hybrid (راهِ واقعی)** | برای کاربرانِ عادی **push**؛ برای سلبریتی‌ها **pull** (هنگامِ خواندن، عکس‌های سلبریتی‌ها به فیدِ push‌شده تزریق می‌شود) | بهترینِ هر دو دنیا | پیچیده‌تر |

> نکته‌ی مصاحبه: اگر «push vs pull» پرسیده شد، حتماً به **مشکلِ سلبریتی** و راه‌حلِ **hybrid** اشاره کن — همین تمایزِ یک کاندیدای متوسط از خوب است.

**⑥ معماریِ کامل (با async برای fan-out و thumbnail):**

```
آپلود:
[کاربر] → [LB+Auth] → [App] ──① عکس──▶ [S3] ──▶ [CDN]
                          │
                          ②──▶ [Queue] ──▶ [Workers] → thumbnail + fan-out push به فیدِ followerها (Redis)
                          │
                          ③──▶ [DB Primary] metadata
                          └──▶ کاربر فوری «آپلود شد» می‌گیرد

خواندنِ فید:
[کاربر] → [LB] → [App] → [Redis: فیدِ آماده] ──(+ تزریقِ سلبریتی‌ها با pull)──▶ آدرسِ عکس‌ها
                                                              │
                                            [CDN] خودِ عکس‌ها را می‌دهد
خواندنِ metadata سنگین → [Read Replicas]
```

- **notificationها** (لایک/فالوِ بلادرنگ): با **WebSocket** به کاربرِ آنلاین.
- **sharding** دیتابیس: بر اساسِ `userId` تقسیم کن تا با رشدِ کاربرها، نوشتن روی یک گره نماند.

## 🔬 آزمایشگاه

اینجا کاغذ و قلم برمی‌داری و خودت طراحی می‌کنی. برای هر تمرین، **اول** خودت روش ۶-گامیِ بخش ۱ را اجرا کن، بعد چک‌لیستِ راهنما را نگاه کن.

### تمرینِ طراحی ۱ — Rate Limiter (محدودکننده‌ی نرخ)

> طراحی کن: سرویسی که اجازه ندهد یک کاربر بیش از ۱۰۰ درخواست در دقیقه بزند.

چک‌لیستِ چیزهایی که باید در نظر بگیری:
- [ ] الگوریتم: **Fixed Window**، **Sliding Window Log/Counter**، **Token Bucket**، یا **Leaky Bucket**؟ trade-offـشان؟
- [ ] state کجا نگه داشته شود تا در **چند سرور** هماهنگ بماند؟ (راهنما: Redis با عملیاتِ atomic مثل `INCR` + `EXPIRE`)
- [ ] کلیدِ محدودسازی: per-user؟ per-IP؟ per-API-key؟
- [ ] وقتی رد می‌شود چه کدی برگردانی؟ (راهنما: `429 Too Many Requests` + هدرِ `Retry-After`)
- [ ] مشکلِ مرزِ پنجره (burst در لبه‌ی دو پنجره‌ی fixed) چطور حل می‌شود؟

### تمرینِ طراحی ۲ — اپلیکیشنِ چت (Chat App)

> طراحی کن: چتِ دو‌نفره و گروهی، با وضعیتِ آنلاین و تحویلِ پیام.

چک‌لیست:
- [ ] حملِ پیامِ بلادرنگ با چه چیزی؟ (راهنما: **WebSocket**، نه polling)
- [ ] پیام‌های آفلاین کجا صف می‌شوند؟ (راهنما: persist در DB + push هنگامِ آنلاین‌شدن)
- [ ] چطور یک کاربر را به سروری که اتصالِ WebSocketـش آنجاست، route کنی؟ (راهنما: registry اتصال‌ها در Redis؛ pub/sub بین سرورها)
- [ ] ترتیبِ پیام‌ها چطور تضمین می‌شود؟ تیکِ «تحویل شد/خوانده شد»؟
- [ ] read-heavy تاریخچه‌ی چت → sharding بر اساسِ `chatId`.

### تمرینِ طراحی ۳ — فیدِ خبری (News Feed)

> طراحی کن: فیدِ شبکه‌ی اجتماعی (همان مسئله‌ی نمونه‌ی ۲، ولی خودت از صفر).

چک‌لیست:
- [ ] **Push vs Pull vs Hybrid** — کدام و چرا؟ مشکلِ celebrity را حل کردی؟
- [ ] فیدِ آماده کجا کش شود؟ (راهنما: Redis، لیستِ `photoId`ها)
- [ ] ranking: زمانی ساده یا امتیازی؟
- [ ] pagination با **cursor** (نه offset) چرا بهتر است؟
- [ ] fan-out روی **queue** ناهمگام چرا لازم است؟

### تمرینِ محاسبه — back-of-the-envelope

> **مسئله:** یک سرویسِ پیام‌رسان ۵۰۰ میلیون کاربرِ فعالِ روزانه دارد. هر کاربر روزی ۴۰ پیام می‌فرستد. هر پیام به‌طور میانگین ۲۰۰ بایت است (متنِ خام). تخمین بزن: (الف) write QPS، (ب) storage روزانه، (ج) storage در ۵ سال.

<details><summary>**راه‌حل**</summary>

```
کاربرها = 5 × 10⁸
پیام در روز = 5×10⁸ × 40 = 2 × 10¹⁰ پیام/روز

(الف) write QPS:
   2×10¹⁰ ÷ 10⁵ ثانیه = 2 × 10⁵ = ~200,000 write/s (میانگین)
   peak (×2)          ≈ 400,000 write/s
   → خیلی فراتر از یک DB → حتماً sharding + queue

(ب) storage روزانه:
   2×10¹⁰ × 200 بایت = 4 × 10¹² بایت = 4 TB/روز (متنِ خام)
   با metadata/overhead و replication (×3) ≈ ~12 TB/روز

(ج) storage در ۵ سال:
   4 TB × 365 × 5 ≈ 7,300 TB ≈ ~7.3 PB (خام، بدون replication)
   با ×3 ≈ ~22 PB
```

درس: ۴۰۰k write/s و چند petabyte → این سیستم **write-heavy** است؛ NoSQL با sharding بر `userId`/`chatId`، نوشتن از طریقِ queue، و طرحِ نگه‌داریِ پلکانی (پیام‌های قدیمی به storageِ ارزان‌تر).

</details>

## ✅ چک‌لیست تسلط

اگر بتوانی به این‌ها جواب بدهی، این فصل را بلدی:

- [ ] ۶ گامِ مصاحبه‌ی System Design را به ترتیب نام ببر و هرکدام را در یک جمله توضیح بده.
- [ ] فرقِ نیازمندیِ functional و non-functional؟ سه non-functional که معماری را عوض می‌کنند؟
- [ ] از روی DAU و نسبتِ read/write، write QPS و read QPS را حساب کن. «۱۰⁵ ثانیه در روز» را چرا حفظ می‌کنیم؟
- [ ] چرا برای ۴۰هزار read/s نمی‌توان به یک دیتابیس بسنده کرد؟ چه سه چیزی اضافه می‌کنی؟
- [ ] چرا عکس‌ها در Object Storage می‌روند نه دیتابیس؟ چه چیزی در دیتابیس می‌ماند؟
- [ ] تفاوتِ Cache (Redis) و CDN در یک جمله؟ هرکدام برای چه نوع داده‌ای؟
- [ ] داستانِ مقیاس را از single server تا city کامل بگو: هر گلوگاه و راه‌حلش.
- [ ] چرا سرورها باید stateless باشند و این چه ربطی به Load Balancer دارد؟
- [ ] push در برابر pull در فید: هرکدام کِی؟ مشکلِ celebrity چیست و hybrid چطور حلش می‌کند؟
- [ ] برای URL shortener: Base62 از auto-increment در برابر hash — مزیت و عیب؟ چرا `301` خطرناک است؟
- [ ] SQL در برابر NoSQL: انتخاب را به چه چیزی گره می‌زنی (نه «مقیاس‌پذیری»)؟
- [ ] CAP: در لحظه‌ی partition، فید را CP می‌سازی یا AP؟ حسابِ بانکی را؟

### روش و رویکرد

<details><summary>**۱. مصاحبه‌ی System Design را از کجا شروع می‌کنی؟ چرا مستقیم سراغِ کشیدنِ معماری نمی‌روی؟**</summary>

اول **شفاف‌سازیِ نیازمندی‌ها**: کارکردی (سیستم چه می‌کند) و غیرکارکردی (scale, latency, consistency, read/write ratio). بدونِ این، در خلأ طراحی می‌کنی و ممکن است چیزی بسازی که مصاحبه‌گر نخواسته. این مرحله نشان می‌دهد ساختاریافته فکر می‌کنی و scope را مدیریت می‌کنی.

</details>

<details><summary>**۲. شش گامِ استاندارد چیست؟**</summary>

① Clarify requirements (functional/non-functional) → ② Back-of-envelope estimation → ③ API design → ④ High-level architecture → ⑤ Deep dive روی یک‌دو جزء → ⑥ Bottlenecks و scaling و trade-offها.

</details>

<details><summary>**۳. مهم‌ترین عددِ غیرکارکردی که جهتِ طراحی را عوض می‌کند چیست؟**</summary>

نسبتِ **read/write**. سیستمِ read-heavy (عکس، خبر، URL shortener) به replica، cache و CDN نیاز دارد؛ سیستمِ write-heavy (چت، لاگ، متریک) به sharding، queue و NoSQL.

</details>

### تخمین (Estimation)

<details><summary>**۴. سرویسی با ۱۰ میلیون DAU داری، هر کاربر روزی ۲۰۰ بار می‌خواند. read QPS تقریبی؟**</summary>

`10⁷ × 200 = 2×10⁹ خواندن/روز`. تقسیم بر `~10⁵ ثانیه/روز` → `~20,000 read/s` میانگین، و با ضریبِ پیک (×2) حدود `40,000 read/s`. نتیجه: یک دیتابیس کافی نیست؛ replica + cache + CDN لازم است.

</details>

<details><summary>**۵. چرا «۱۰⁵ ثانیه در روز» را حفظ می‌کنیم؟**</summary>

چون ستونِ فقراتِ هر محاسبه‌ی QPS است: روز ≈ 86,400 ثانیه ≈ 10⁵. کافیست تعدادِ روزانه‌ی عملیات را بر 10⁵ تقسیم کنی تا QPS میانگین به‌دست بیاید — بدون ماشین‌حساب.

</details>

<details><summary>**۶. عددهای latency: خواندن از RAM در برابر دیسک در برابر round-trip بین‌قاره‌ای؟**</summary>

RAM ~100ns، SSD ~1ms (برای 1MB)، disk seek ~10ms، RTT داخلِ دیتاسنتر ~0.5ms، RTT بین‌قاره‌ای ~150ms. درس: RAM حدودِ ۱۰⁵ برابر سریع‌تر از دیسک است، و یک سفرِ بین‌قاره‌ای از هزاران عملیاتِ محلی گران‌تر — توجیهِ وجودِ cache و CDN.

</details>

<details><summary>**۷. چطور تخمین می‌زنی که سرویسِ عکست به چقدر storage نیاز دارد؟**</summary>

`(عکسِ تازه در روز) × (حجمِ هر عکس) × (تعدادِ روزها) × (ضریبِ replication)`. مثال: `2×10⁷ عکس/روز × 300KB × 365 × 3 ≈ ~6.6 PB/سال`. نتیجه‌ی عددی مهم نیست؛ مهم این است که بفهمی این مقیاس یعنی **Object Storage**، نه DB.

</details>

### مقیاس‌بندی (Scaling)

<details><summary>**۸. تفاوتِ scaling عمودی و افقی؟ چرا معمولاً افقی برنده است؟**</summary>

عمودی (vertical) = سرورِ قوی‌تر؛ ساده ولی سقف دارد و یک نقطه‌ی شکست می‌ماند. افقی (horizontal) = سرورهای بیشتر کنار هم؛ نیاز به load balancer و سرورهای stateless دارد، ولی تقریباً بی‌نهایت رشد می‌کند و تحمل‌پذیرِ خطاست. سیستم‌های بزرگ افقی مقیاس می‌خورند.

</details>

<details><summary>**۹. داستانِ مقیاس را از یک سرور تا یک سیستمِ بزرگ روایت کن.**</summary>

single server → (vertical scaling تا جایی که جواب می‌دهد) → چند سرور + **Load Balancer** → سرورها را **stateless** کن (state به Redis) → **read replica** برای بارِ خواندن → **cache (Redis)** برای داده‌ی پرتکرار + **CDN** برای فایل → **queue + workers** برای کارِ سنگینِ ناهمگام → **sharding** وقتی یک گره‌ی DB کم آورد. هر قدم پاسخِ یک bottleneckِ مشخص است.

</details>

<details><summary>**۱۰. چرا سرورها باید stateless باشند؟**</summary>

چون load balancer هر درخواست را ممکن است به سرورِ متفاوتی بفرستد. اگر state (مثل session) در حافظه‌ی یک سرور باشد، درخواستِ بعدیِ همان کاربر روی سرورِ دیگری آن را پیدا نمی‌کند. راه‌حل: state را در یک storeِ مشترک (Redis/DB) بگذار تا هر سرور بتواند سرویس بدهد و scale-in/out بی‌دردسر شود.

</details>

<details><summary>**۱۱. read replica چه مشکلی را حل می‌کند و چه هزینه‌ای دارد؟**</summary>

بارِ **خواندن** را از Primary برمی‌دارد و بین چند نسخه‌ی فقط-خواندنی پخش می‌کند (عالی برای read-heavy). هزینه: **replication lag** — نوشتنِ تازه با تأخیرِ میلی‌ثانیه‌ای به replicaها می‌رسد، پس خواندنِ بلافاصله ممکن است داده‌ی کهنه بدهد (eventual consistency). برای عکس قابل‌قبول، برای موجودی/مبلغ نه.

</details>

<details><summary>**۱۲. sharding چیست و بر چه کلیدی شارد می‌کنی؟**</summary>

تقسیمِ داده روی چند گره‌ی مستقل، وقتی داده یا نوشتن از ظرفیتِ یک گره فراتر رفت. کلیدِ shard را طوری انتخاب کن که بار یکنواخت پخش شود و کوئری‌های رایج در یک shard بمانند — مثلاً `userId` در شبکه‌ی اجتماعی، یا `chatId` در چت. خطر: **hot shard** (یک کلیدِ خیلی پرترافیک) و سختیِ JOINهای cross-shard.

</details>

### Cache و CDN

<details><summary>**۱۳. تفاوتِ Cache (Redis) و CDN؟**</summary>

هر دو کپیِ سریعِ موقت‌اند، ولی: **Redis/Cache** برای داده‌ی پرتکرارِ پویا و کوچک (پروفایل، session، نتیجه‌ی کوئری) و نزدیکِ سرور/دیتابیس قرار می‌گیرد. **CDN** برای فایل‌های ثابتِ سنگین (عکس، ویدیو، JS/CSS) و در نقاطِ جغرافیاییِ نزدیکِ کاربر در سراسر دنیا قرار می‌گیرد.

</details>

<details><summary>**۱۴. الگوی cache-aside را توضیح بده. cache hit و miss یعنی چه؟**</summary>

اپ اول از cache می‌پرسد؛ اگر بود (**hit**) همان را برمی‌گرداند. اگر نبود (**miss**) از DB می‌گیرد، کپی را در cache می‌گذارد، بعد برمی‌گرداند. مزیت: فقط داده‌ی واقعاً درخواست‌شده کش می‌شود. باید TTL و سیاستِ invalidation داشته باشی تا داده‌ی کهنه نماند.

</details>

<details><summary>**۱۵. مشکلِ cache stampede (یا thundering herd) چیست؟**</summary>

وقتی یک کلیدِ داغ منقضی می‌شود و **همزمان** هزاران درخواست miss می‌خورند و همه با هم به DB می‌روند و آن را غرق می‌کنند. راه‌حل‌ها: قفلِ بازسازی (فقط یکی DB را بزند)، TTL با jitter، و `stale-while-revalidate` (داده‌ی کهنه را بده تا نسخه‌ی تازه ساخته شود).

</details>

### دیتابیس و trade-offها

<details><summary>**۱۶. SQL یا NoSQL — انتخاب را به چه چیزی گره می‌زنی؟**</summary>

به **شکلِ داده و نیازِ سازگاری**، نه صرفِ «مقیاس‌پذیری». SQL برای داده‌ی رابطه‌ای با تراکنشِ مهم (پرداخت، سفارش) و JOIN. NoSQL برای حجمِ عظیم، schema منعطف، و نوشتنِ توزیع‌شده (فید، تایم‌لاین، لاگ). جوابِ خوب همیشه «بستگی دارد به ...» است.

</details>

<details><summary>**۱۷. CAP را در یک جمله توضیح بده و یک مثالِ CP و یک مثالِ AP بزن.**</summary>

هنگامِ partitionِ شبکه باید بینِ Consistency و Availability یکی را قربانی کنی. **CP**: حسابِ بانکی/موجودیِ انبار — بهتر است خطا بدهی تا جوابِ غلط. **AP**: تعدادِ لایک/فیدِ شبکه‌ی اجتماعی — بهتر است جوابِ کمی کهنه بدهی تا اصلاً جواب ندهی.

</details>

<details><summary>**۱۸. تفaوتِ latency و throughput؟ چطور یکی را به قیمتِ دیگری بهبود می‌دهی؟**</summary>

latency = زمانِ یک درخواست؛ throughput = تعدادِ درخواست در ثانیه. **batching** و صف‌بندی، throughput را بالا می‌برند ولی latency تک‌درخواست را هم بالا می‌برند. برعکس، پردازشِ فوریِ تک‌به‌تک latency را کم می‌کند ولی throughput را محدود. در طراحی باید بگویی کدام برایت اولویت دارد.

</details>

<details><summary>**۱۹. کِی کارِ async و queue به‌کار می‌بری به‌جای sync؟**</summary>

وقتی کار **کند** است و کاربر **به نتیجه‌ی فوری نیاز ندارد**: ساختِ thumbnail، فرستادنِ ایمیل/notification، fan-out فید، پردازشِ ویدیو. کاربر فوری جواب می‌گیرد و worker سرِ فرصت کار می‌کند. هزینه: پیچیدگیِ queue، تضمینِ at-least-once و نیاز به **idempotency** در worker.

</details>

### Object storage و media

<details><summary>**۲۰. چطور ۵۰۰ میلیون عکس را ذخیره می‌کنی؟**</summary>

خودِ فایل‌ها در **Object Storage** (مثل S3) که برای blob سنگین، ارزان و durable و بی‌نهایت‌مقیاس است؛ و در **دیتابیس** فقط metadata (`photoId, userId, url, caption, createdAt`). پخش از طریقِ **CDN** که از S3 می‌خواند. عکس را هرگز در DB یا روی دیسکِ app server نگه نده.

</details>

<details><summary>**۲۱. چرا عکس را در دیتابیس ذخیره نمی‌کنیم؟**</summary>

DB برای blobهای بزرگ ساخته نشده: backup و replication را سنگین می‌کند، cache و buffer pool را با داده‌ی غیرقابلِ کوئری پر می‌کند، و گران است. Object Storage برای دقیقاً همین کار بهینه شده و CDN هم می‌تواند مستقیم از آن سرو کند.

</details>

### طراحی‌های کلاسیک

<details><summary>**۲۲. یک URL shortener طراحی کن — کامل توضیح بده.**</summary>

Requirements: read-heavy، redirect سریع. API: `POST /shorten` و `GET /{short}→301/302`. تولیدِ کد: **Base62 از auto-increment ID** (کوتاه و یکتا) یا **hash + چکِ collision**. ذخیره: key-value store (`short→long`). مقیاس: hot URLها را در **Redis** کش کن و جلویش CDN بگذار. redirect را معمولاً **`302`** بده تا آنالیتیکس و کنترل داشته باشی.

</details>

<details><summary>**۲۳. در URL shortener چرا `301` خطرناک است؟ چرا Base62 بهتر از hash؟**</summary>

`301` دائمی است و مرورگر/CDN آن را سخت کش می‌کنند → آمارِ کلیک را از دست می‌دهی و دیگر نمی‌توانی مقصد را عوض کنی. Base62 از یک شمارنده، یکتایی را **تضمین** می‌کند (بدونِ collision)، در حالی که hash باید برای هر تولید، تصادم را چک و در صورتِ برخورد دوباره تولید کند.

</details>

<details><summary>**۲۴. فیدِ خبری: push یا pull؟ کامل توضیح بده.**</summary>

**Pull (fan-out on read)**: فید هنگامِ خواندن ساخته می‌شود — نوشتن ارزان، خواندن گران؛ خوب برای کسانی که زیاد follow دارند. **Push (fan-out on write)**: هنگامِ نوشتن در فیدِ همه‌ی followerها نوشته می‌شود — خواندن ارزان (فیدِ آماده در Redis)، ولی نوشتن گران و **مشکلِ celebrity** (یک سلبریتی → میلیون‌ها نوشتن). راه‌حلِ واقعی **hybrid**: push برای کاربرِ عادی، pull برای سلبریتی‌ها (هنگامِ خواندن تزریق می‌شوند).

</details>

<details><summary>**۲۵. مشکلِ celebrity/hot-key در fan-out چیست و چطور حلش می‌کنی؟**</summary>

وقتی یک کاربر میلیون‌ها follower دارد، رویکردِ push باید برای هر پستِ او میلیون‌ها نوشتنِ فید انجام دهد — کند و گران. حل: این کاربران را **pull** کن؛ یعنی پست‌هایشان را در فیدِ آماده‌ی followerها ننویس، بلکه هنگامِ خواندنِ فید به‌صورتِ زنده تزریق کن. این همان رویکردِ hybrid است.

</details>

<details><summary>**۲۶. یک rate limiter طراحی کن. کدام الگوریتم و state کجا؟**</summary>

الگوریتم‌ها: Fixed Window (ساده ولی مشکلِ burst لبه)، Sliding Window (دقیق‌تر)، **Token Bucket** (اجازه‌ی burstِ کنترل‌شده — محبوب)، Leaky Bucket. state باید **مشترک** باشد تا روی چند سرور هماهنگ بماند → **Redis** با `INCR`/`EXPIRE` یا script اتمی. هنگامِ رد: کدِ `429 Too Many Requests` + هدرِ `Retry-After`.

</details>

<details><summary>**۲۷. notificationِ بلادرنگ (مثل لایک/پیامِ تازه) را چطور می‌رسانی؟**</summary>

برای کاربرِ آنلاین، **WebSocket** (کانالِ دوطرفه‌ی پایدار) به‌جای polling. سرور باید بداند هر کاربر روی کدام نمونه اتصال دارد (registry در Redis) و با pub/sub بینِ نمونه‌ها پیام را به نمونه‌ی درست برساند. برای کاربرِ آفلاین، notification را persist کن و push/badge بفرست.

</details>

<details><summary>**۲۸. اپلیکیشنِ چت را در سطحِ بالا چطور طراحی می‌کنی؟**</summary>

اتصالِ بلادرنگ با **WebSocket**؛ پیام‌ها persist در DB (شارد بر `chatId`)؛ پیام‌های کاربرِ آفلاین صف و هنگامِ آنلاین‌شدن push می‌شوند؛ registry اتصال‌ها در Redis + pub/sub برای رساندن پیام به سروری که گیرنده آنجاست؛ ترتیب با sequence/timestamp و وضعیتِ تحویل/خوانده‌شدن با ackها.

</details>

### عمومی و دامی‌ها

<details><summary>**۲۹. اولین چیزی که در یک طراحی به سراغش می‌روی چیست — دیتابیس یا requirements؟**</summary>

requirements. پریدن مستقیم به انتخابِ تکنولوژی (مثلاً «از Kafka و Cassandra استفاده می‌کنم») بدونِ دانستنِ scale و الگوی دسترسی، دامِ کلاسیکِ کاندیدای ضعیف است. اول بفهم چه می‌سازی و با چه مقیاسی، بعد ابزار را به نیاز گره بزن.

</details>

<details><summary>**۳۰. تفاوتِ availability و durability؟**</summary>

**Availability** = سیستم همین الان در دسترس و پاسخ‌گو هست (uptime، با replica و failover و حذفِ single point of failure به‌دست می‌آید). **Durability** = داده‌ی نوشته‌شده گم نمی‌شود حتی اگر سخت‌افزار بسوزد (با replication، backup و write-ahead log). یک سیستم می‌تواند durable ولی موقتاً غیرقابل‌دسترس باشد.

</details>

<details><summary>**۳۱. single point of failure را چطور حذف می‌کنی؟**</summary>

هیچ جزئی نباید تک و حیاتی باشد: چند app server پشتِ LB، خودِ LB را افزونه (دو نمونه + failover)، DB با replica و failover خودکار، چند availability zone، و صف برای جذبِ spike. هر جزء را بپرس «اگر این بمیرد چه می‌شود؟» و برایش پلنِ B بگذار.

</details>

<details><summary>**۳۲. چطور تصمیم می‌گیری کِی یک جزءِ تازه (cache, queue, shard) اضافه کنی؟**</summary>

با داده، نه حدس. ابتدا ساده بساز و **اندازه‌گیری** کن (observability از فصل ۷)؛ وقتی متریک‌ها یک bottleneck را نشان دادند (DB CPU بالا → cache/replica؛ latency نوشتن → queue؛ یک گره پر شد → shard) آن گلوگاهِ مشخص را رفع کن. over-engineeringِ زودهنگام به‌اندازه‌ی under-engineering خطرناک است.

</details>
